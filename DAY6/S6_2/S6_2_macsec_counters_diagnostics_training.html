IEEE 802.1AE MAC Security observability

MACsec counters and diagnostics explained from the wire upward

MACsec counters are not random interface statistics. They are evidence left by a frame as it passes through the MAC Security Entity, the Secure Channel lookup, the Secure Association lookup, packet-number replay checks, and GCM-AES authentication/decryption.

SecY

Where the decision happens

The MAC Security Entity separates protected controlled-port traffic from uncontrolled traffic such as key agreement.

SecTAG

What the receiver reads first

EtherType, TCI/AN, Short Length, Packet Number, and optional Secure Channel Identifier drive parsing and lookup.

PN + replay

Freshness diagnosis

Packet Number movement explains late, delayed, replayed, duplicated, and out-of-order reception signatures.

ICV

Cryptographic failure evidence

A bad Integrity Check Value points to wrong key, wrong SA, tampering, bit corruption, or mismatched protected bytes.

Teaching rule. Always read MACsec counters together with the frame format. A rising counter tells you not only that a problem occurred, but also which parsing or validation stage accepted or rejected the frame.

1. The counter mental model

A good diagnostics model has three layers. The first layer is the SecY or interface layer, which sees whether the frame even looks like a MACsec frame. The second layer is the Secure Channel layer, which represents one transmitting peer or one local transmit channel. The third layer is the Secure Association layer, which is the actual key and packet-number context selected by the Association Number.

Interface counters answer

Did the frame have a MACsec EtherType? Was the SecTAG syntactically acceptable? Could the receiver identify the Secure Channel?

SC counters answer

For this peer/channel, were frames accepted, late, delayed, invalid, not valid, or using a wrong/unavailable SA?

SA counters answer

For this key generation and Association Number, how many frames used this SA, and what packet number should come next?

2. MACsec PDU and header format

On the wire, MACsec inserts the SecTAG after the source MAC address and appends the Integrity Check Value at the end of the protected frame. The MACsec EtherType is 0x88E5. The SecTAG is 8 octets without Secure Channel Identifier and 16 octets when the optional 8-octet Secure Channel Identifier is present.

TCI/AN byte as a diagnostic source

The exact bit numbering is normally hidden by device software, but the receiver uses the TCI/AN byte as follows: version/reserved validity checks, End Station or Single Copy Broadcast indications, SCI-present indication, encryption/protection indication, and the two-bit Association Number.

Counter implication. A malformed SecTAG increments a tag-level counter before the receiver can make a meaningful cryptographic decision. A valid SecTAG with wrong SCI or AN reaches lookup counters. A valid lookup with bad ICV reaches validation counters.

3. Receive diagnostic pipeline

The receive path is where most troubleshooting happens. Read the counters in pipeline order. A frame can only increment a later-stage counter if it survived earlier parsing and lookup.

Most useful receive question

How far did the packet get? NoTag and BadTag mean it failed before SCI/SA lookup. UnknownSCI means the header was parseable but peer/channel mapping failed. NotUsingSA or UnusedSA means the channel was found but the Association Number/key state was wrong. Invalid or NotValid means cryptographic validation failed.

Replay-window nuance

A packet can be out of order but still acceptable if it is within the configured replay window. It becomes late when it falls below the lowest acceptable packet number under replay protection. Some stacks also distinguish delayed from late when validation or replay policy accepts observation but marks old order.

4. Transmit diagnostic pipeline

Transmit counters prove that the local SecY is protecting frames and show whether it is protecting only integrity or both confidentiality and integrity. They also expose packet-number exhaustion, frame-size expansion, and whether traffic bypassed MACsec as untagged.

1. Controlled or uncontrolled?Data traffic should enter the controlled port. MKA and allowed exceptions use the uncontrolled path.OutPktsUntagged
2. TxSC selectChoose the transmit Secure Channel for the local SecY.TxSC in use
3. TxSA selectUse encoding SA, write AN, and use that SA key.TxSA[AN]
4. PN allocationUse NextPN and advance after protected frame construction.NextPN
5. SecTAG insertAdd EtherType 0x88E5, TCI/AN, SL, PN, optional SCI.OutPktsProtected
6. GCM-AESAuthenticate visible fields and encrypt Secure Data if E/C policy enables confidentiality.OutPktsEncrypted
7. Size checkMACsec overhead can make an original frame too long for the egress MAC/PHY path.OutPktsTooLong
Counter implication. If OutPktsProtected rises but OutPktsEncrypted stays flat, the link may be in integrity-only protection. That can be correct when configured, but it is not confidentiality.

7. Interesting diagnostic scenarios

These examples are intentionally written as investigations. Each scenario starts with symptoms, then interprets the counter fingerprint, then identifies the most likely root cause.

A. Healthy encrypted link baseline

1
Observation

Traffic flows. OutPktsEncrypted rises on transmitter. InPktsOK and InOctetsDecrypted rise on receiver. InPktsInvalid, InPktsLate, and InPktsUnknownSCI stay flat.

2
Packet interpretation

SecTAG parse succeeds, SCI maps to RxSC, AN maps to active RxSA, PN is fresh, GCM-AES ICV succeeds.

3
Teaching point

Capture this baseline before inducing faults. It gives a known-good fingerprint for every later scenario.

B. MKA is alive, but user data is clear or blocked

1
Symptom

MKA state looks connected, but protected user traffic does not pass. Receive side shows InPktsNoTag or InPktsUntagged.

2
Why this can happen

MKA Protocol Data Units use the uncontrolled path. They can still be exchanged while the controlled port is not enabled, the data traffic is on the wrong subinterface, or the peer is not applying SecY protection to that traffic class.

3
Action

Check controlled-port state, policy exceptions, VLAN/subinterface binding, and whether the data frame actually has EtherType 0x88E5.

C. Unknown SCI after a switch or topology change

1
Symptom

InPktsUnknownSCI increases. InPktsBadTag stays flat. ICV-related counters do not rise.

2
Interpretation

The SecTAG is syntactically valid. The receiver can read an SCI, but no matching receive Secure Channel exists.

3
Likely root causes

Peer moved to another port, old RxSC was not removed, MKA has a stale peer identity, or SCI omission/inference is invalid now that the link is no longer unambiguous point-to-point.

D. Replay attack versus benign reordering

1
Symptom

InPktsLate rises during a test where old captured frames are injected. In another test, InPktsDelayed rises but traffic still passes.

2
Interpretation

Late means the packet number is below the replay window threshold and replay protection rejects it. Delayed means the packet is older than the current expected sequence context but is still within what the policy/window can observe or tolerate.

3
Action

Compare received PN to RxSA NextPN and replay-window size. For TSN or preemption cases, verify whether reordering below the MACsec hop is possible.

E. Rekey AN race

1
Symptom

Short burst of drops during rekey. Counters show InPktsUnusedSA, InPktsNotUsingSA, or InPktsNotValid. After a few seconds, traffic recovers.

2
Interpretation

The transmitting peer selected a new Association Number before the receiver had the corresponding RxSA installed/enabled, or the old SA was disabled too early.

3
Action

Review MKA live peer state, TxSA encoding SA, RxSA in-use flags, and deletion timing. Rekey must maintain overlap so fresh keys can replace old keys without dropping live traffic.

F. MTU and frame-size expansion failure

1
Symptom

Large payloads fail, small payloads pass. Transmitter shows OutPktsTooLong.

2
Interpretation

MACsec adds SecTAG and ICV overhead. If SCI is included, overhead is higher. Stacked VLAN tags make the total wire length larger too.

3
Calculation

Approximate protected frame growth: base Ethernet frame + 8 B SecTAG without SCI or 16 B with SCI + 16 B ICV. Adjust MTU or maximum frame-size limits accordingly.

G. Wrong key, wrong cipher suite, or wrong XPN mode

1
Symptom

SCI is known and AN exists, but InPktsInvalid or InPktsNotValid rises. Replay counters are not the primary movers.

2
Interpretation

The receiver reached cryptographic verification. Failure here usually means the GCM nonce/key/protected bytes do not match what the transmitter used.

3
Action

Check SAK identity, cipher suite, XPN versus non-XPN mode, SSCI/salt in XPN mode, ICV length, confidentiality offset, and offload configuration.

H. TSN or switch counters disappear behind MACsec

1
Symptom

After enabling encryption, downstream stream filters, deep packet classifiers, or protocol-specific counters stop matching payload fields.

2
Interpretation

MACsec leaves outer DA, SA, and SecTAG visible, but Secure Data may hide the original EtherType and payload. If the switch needs those fields, it must classify before encryption, terminate MACsec at that hop, or rely on fields intentionally left visible.

3
Action

Decide whether MACsec is hop-by-hop at the bridge, end-to-end across a bridge, or implemented in a PHY/MAC offload before/after classification. Observability depends on this placement.

9. Automotive and AUTOSAR-style integration notes

In an automotive Ethernet design, MACsec counters are useful not only for live debugging but also for diagnostic trouble codes, end-of-line testing, security monitoring, and service-bay analysis. The key design question is where the MACsec function sits: inside the host MAC, inside the switch, inside the PHY, or in software.

Placement changes visibility

If MACsec is terminated before a bridge classifier, the bridge can see the original VLAN, priority, IP, or stream fields. If encrypted traffic crosses the bridge without termination, many inner fields become hidden and only outer MACsec-visible fields remain usable.

Counter ownership changes

A PHY offload may maintain counters in the PHY. A switch offload may expose per-port or per-SecY counters. A software implementation may expose Linux-style statistics. Do not compare raw names without confirming which SecY instance owns them.

10. Standards anchors and reading map

IEEE 802.1AE MACsec

Defines the MAC Security Entity, SecTAG, Secure Channels, Secure Associations, packet-number freshness, GCM-AES protection, validation behaviour, and MACsec frame format.

IEEE8021-SECY-MIB / YANG management

Provides the management view for SecY, controlled and uncontrolled ports, SC/SA objects, cipher suite selection, SCI inclusion, replay protection, and statistics.

IEEE 802.1X MKA

Defines MACsec Key Agreement, which establishes the Connectivity Association, distributes or installs SAKs, controls TxSA/RxSA state, and enables rekey without service interruption.

AUTOSAR Classic MACsec integration

In automotive software stacks, the MKA module and Ethernet Interface/Switch/Transceiver layers can expose SecY parameters and statistics to diagnostics and monitoring paths. Exact APIs are implementation/profile dependent.

Recommended reading order for engineers

  1. Start with SecTAG format and the receive validation pipeline.
  2. Then read the SecY statistics model so each counter has a home.
  3. Then study MKA rekey and SA overlap, because many real faults happen at AN transitions.
  4. Finally map the standard counters to your vendor CLI, SNMP, AUTOSAR diagnostic event, or hardware offload registers.
