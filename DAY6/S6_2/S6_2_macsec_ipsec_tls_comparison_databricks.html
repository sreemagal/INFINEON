MACsec vs IPsec vs TLS

A protocol-engineering comparison of where each security mechanism sits, what its PDU/header looks like, what it hides, what it exposes, and how it behaves in automotive Ethernet, enterprise, and cloud-connected designs.

MACsecIEEE 802.1AE link-layer frame protection using SecTAG, packet number, secure channel, secure association and ICV.
IPsecIETF IP-layer protection using AH/ESP, SPI, sequence number, security association database and IKEv2.
TLSIETF session protection above transport, with TLS 1.3 AEAD records and application/server identity.

1. The simplest correct mental model

MACsec, IPsec and TLS are not three names for the same security function. They are three different places where cryptographic protection can be inserted into the communication stack.

MACsec
Link lock

Protects Ethernet frames as they cross a LAN link or MAC service. It is excellent when the attack is a physical tap, rogue device, or link-local frame injection on an Ethernet segment.

Layer 2EtherType 0x88E5SecTAG + ICV
IPsec
IP tunnel / IP shield

Protects IP datagrams at the network layer. It can secure host-to-host, gateway-to-gateway, or gateway-to-host traffic without each application being modified.

Layer 3ESP protocol 50AH protocol 51
TLS
Application session lock

Protects an application stream above TCP. It is the usual answer for HTTPS, service APIs, diagnostic clients, OTA servers and application-level identity.

Above TCPTLS 1.3 RFC 8446AEAD records
Important: In a real vehicle, data center, or cloud design, these can be stacked. A diagnostic session may run DoIP over TLS, inside an IPsec tunnel, over MACsec-protected Ethernet links. Each layer solves a different exposure problem.

This reinforces the same architectural split: MACsec at Ethernet link/MAC service level, IPsec at IP level, TLS at application session level.

5. MACsec frame format and field semantics

MACsec adds a Security Tag (SecTAG) after the Ethernet source address in the canonical MACsec frame, protects the MAC service data as Secure Data, and appends an Integrity Check Value (ICV). It provides connectionless integrity, origin authenticity, replay protection and, when enabled, confidentiality.

Practical automotive interpretation: A MACsec-protected switch port can reject unauthenticated frames before they enter the trusted bridge pipeline. This is valuable for physical-link attacks, rogue ECUs, debug-port abuse and compromised wiring harness segments.

5.3 What MACsec protects and what it does not automatically protect

Strong points

Physical tap resistanceRogue L2 frame rejectionLow latencyTransparent to applications

MACsec is often implemented in hardware close to the Ethernet MAC. It can therefore run at line rate with deterministic latency, which matters for time-sensitive automotive Ethernet links.

Caveats

Usually link/local domainRouter/switch termination may be requiredL2 metadata may remain visible

Do not treat MACsec as a replacement for application identity. It proves that a frame came from an authorized MACsec peer on that secure connectivity association, not that a remote cloud service or application user is authorized.

6. IPsec AH and ESP packet formats

IPsec is a suite. The architecture defines security policies and associations, while AH and ESP define the actual packet protection. In modern deployments, ESP is the dominant data-plane protocol because it can provide confidentiality plus integrity.

Practical automotive interpretation: IPsec is attractive when you want an ECU, zone controller, gateway or telematics unit to secure IP flows without every application implementing its own cryptographic session. It is especially natural for gateway-to-gateway, gateway-to-cloud and host-to-gateway traffic.

6.3 AH format and why it is less common in many practical deployments

IP headerProtocol/Next Header = 51
Next Header1 B
Payload Len1 B
Reserved2 B
SPI4 B
Sequence4 B
Authentication Data / ICVvariable

AH provides integrity, data-origin authentication and replay protection, but no confidentiality. It authenticates as much of the IP header as possible, excluding fields that legitimately change in transit. This is useful conceptually, but NAT and address/field changes make AH less convenient in many routed deployments. ESP with authentication is therefore more common.

7. TLS 1.3 record format and field semantics

TLS secures an application byte stream. The TLS record layer fragments data, protects records with AEAD once keys are active, and delivers verified plaintext to the application.

Practical automotive interpretation: TLS is the right mental model for application-specific trust: DoIP diagnostics to a tester, OTA update client to backend, service API to cloud, or an ECU application that needs server certificates, client certificates, PSK-based authentication, or explicit application authorization.

7.3 What TLS does not hide

TLS does not hide Ethernet headers, IP addresses, TCP ports, TCP sequence behavior, routing metadata, or packet timing. In base TLS 1.3, much of the ClientHello is visible. Later handshake and application data are protected after the key schedule activates. Server Name Indication privacy requires additional mechanisms such as encrypted ClientHello and should not be assumed from “TLS 1.3” alone.

9. Key establishment and security-context vocabulary

MACsec vocabulary
  • SecY — MAC Security Entity that protects and validates frames.
  • CA — Secure Connectivity Association among MACsec participants.
  • SC — Secure Channel, usually unidirectional.
  • SA — Secure Association under a Secure Channel.
  • SAK — Secure Association Key used to protect frames.
  • MKA — MACsec Key Agreement protocol, responsible for peer discovery, election and SAK distribution.
IPsec vocabulary
  • SPD — Security Policy Database deciding bypass, discard, or protect.
  • SAD — Security Association Database containing active SAs.
  • SA — Directional set of algorithms, keys, SPI, sequence state and lifetime.
  • SPI — On-wire identifier for selecting an SA.
  • IKEv2 — Negotiates authentication, keys and Child SAs.
  • ESN — Extended Sequence Number support for high-rate SAs.
TLS vocabulary
  • Handshake — Negotiates version, cipher suite, keys and peer authentication.
  • Certificate/PSK — Authentication input for server and optionally client.
  • Traffic secrets — Derived keys for protecting records in each direction.
  • Record sequence number — Implicit per direction and used in nonce construction.
  • AEAD — Unified encryption and authentication for TLS 1.3 records.
Operational difference: MACsec and IPsec can be policy-driven by the platform/network stack. TLS is normally application-visible, because the application chooses the server identity, certificate policy, hostname, ALPN, client authentication and authorization model.

10. Interesting scenarios that expose the differences

Scenario A — Automotive TSN backbone with strict latency classes

An ADAS ECU sends sensor-fusion messages across two Ethernet switches to a zone controller. The network uses VLAN Priority Code Point (PCP), time-aware scheduling and policing.

MACsec option: Each physical link is protected. A malicious tap cannot read or inject accepted frames. The switch port decrypts/validates before trusted forwarding logic, or the design intentionally exposes selected tags for scheduling.
IPsec option: The IP payload can be protected host-to-host or gateway-to-gateway, but IPsec may hide transport ports and inner addresses in tunnel mode. This can complicate network-layer inspection and diagnostics.
TLS option: Useful if the application protocol is TCP-based and needs end-to-end application identity. It does not protect non-TCP L2 control traffic or stop link-local L2 injection outside the TLS session.
Best engineering answer: use MACsec for link integrity/confidentiality in the in-vehicle Ethernet fabric, then add TLS only for application sessions that need end-to-end application identity or cloud trust.

Scenario B — Telematics gateway builds a secure tunnel to cloud backend

The vehicle has many internal IP services, but only the telematics control unit has cellular connectivity to the backend.

MACsec: Protects in-vehicle Ethernet links, but stops at the Ethernet segment. It cannot secure the cellular/wide-area routed path by itself.
IPsec: Natural gateway-to-cloud VPN. ESP tunnel mode hides inner vehicle service addresses and gives one platform-level security policy for multiple applications.
TLS: Still valuable for each cloud API, especially when application identity, certificates, authorization scopes and service-level logging matter.
Best engineering answer: IPsec for network tunnel protection, plus TLS for critical backend APIs if application-layer identity and zero-trust style controls are required.

Scenario C — Diagnostics over IP from service tester to ECU

A workshop tester connects to a vehicle Ethernet port and runs diagnostic sessions to several ECUs.

MACsec only: Can authenticate the directly attached tester or switch peer at L2. Good for excluding random devices, but not enough to express diagnostic application permissions.
IPsec: Can protect IP communication to a gateway or ECU, but authorization of diagnostic services still needs higher-layer policy.
TLS: Maps naturally to a diagnostic client/server trust relationship. The ECU can validate tester certificates and the application can authorize diagnostic service levels.
Best engineering answer: MACsec for port/link admission where available, TLS for diagnostic endpoint identity and application authorization.

Scenario D — Rogue laptop plugged into an unused Ethernet branch

An attacker physically accesses a connector and tries ARP spoofing, DHCP abuse, SOME/IP service discovery flooding or frame injection.

MACsec: The strongest first gate. Without MKA/SecY authorization and valid ICV, injected frames are discarded at the link boundary.
IPsec: Protects selected IP flows, but unauthenticated L2/L3 noise may still hit parts of the network unless additional filtering exists.
TLS: Protects individual TCP application sessions, but does not stop the rogue node from sending unrelated L2/L3 traffic.
Best engineering answer: MACsec plus switch access control and rate limiting. TLS/IPsec protect valuable sessions but are not port-admission mechanisms by themselves.

Scenario E — Full layered defense for OTA update

An OTA client downloads software over cellular, passes through a vehicle gateway, and distributes update artifacts to ECUs.

TLS: Protects the OTA HTTPS/API session, authenticates backend endpoints and prevents tampering on the application channel.
IPsec: Optional VPN tunnel between vehicle gateway and backend to hide internal addressing and consolidate policy.
MACsec: Protects in-vehicle Ethernet links during distribution from gateway to target ECUs, reducing risk from local physical taps and injection.
Still required: Signed update packages and secure boot. Channel security does not replace artifact authenticity and rollback protection.
Best engineering answer: use channel security plus signed artifacts. TLS/IPsec/MACsec protect transport paths; software signing protects the payload after storage, forwarding and reboot.

Choose MACsec when…

You control the Ethernet links, need low-latency L2 protection, must reject rogue frame injection at the port, and want protection transparent to IP/application software.

Choose IPsec when…

You need to protect many IP flows between hosts or gateways, especially across routed networks, without modifying every application.

Choose TLS when…

The application needs authenticated server/client identity, certificates, API-level authorization, web compatibility, or end-to-end protection across untrusted intermediaries.

13. Standards and source trail

  • IEEE 802.1AE project page: MACsec provides connectionless user-data confidentiality, integrity, origin authenticity and privacy protection, and uses MKA from IEEE 802.1X to discover authenticated MACsec peers and distribute SAKs.
  • IEEE 802.1AE-2018 project page: 802.1AE-2018 incorporates the base 802.1AE standard and amendments such as GCM-AES-256, extended packet numbering and Ethernet Data Encryption devices.
  • RFC 4301: IPsec architecture, IP-layer services, AH/ESP architecture, anti-replay, confidentiality and limited traffic-flow confidentiality.
  • RFC 4302: Authentication Header format and AH services.
  • RFC 4303: ESP top-level packet format and ESP protocol value 50.
  • RFC 8446: TLS 1.3 record layer, TLSPlaintext and TLSCiphertext structures, AEAD record protection and opaque outer record type.
  • AUTOSAR R24-11 local project files consulted: CP SWS MACsec Key Agreement and FO RS IPsec Protocol, used for automotive framing and terminology.
