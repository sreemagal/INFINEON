MACsec Frame Format Deep Dive

A protocol-standard based explanation of the MACsec protected Ethernet frame, the SecTAG header, the protected and encrypted regions, the Integrity Check Value, and realistic automotive Ethernet examples.

Core mental model

MACsec does not create a new Layer 3 tunnel. It protects the MAC service data on an Ethernet link by inserting a security tag after the source MAC address and by appending an integrity check value before the Ethernet frame check sequence.

1. Standards basis and scope

IEEE 802.1AE MAC Security, normally called MACsec, specifies connectionless data confidentiality, data integrity, data origin authenticity, and privacy-related protection services for MAC clients. The security function is transparent to higher-layer clients because a protected frame is still an Ethernet frame on the wire.

IEEE 802.1X is closely related but not the same protocol. IEEE 802.1AE defines the protection of data frames. IEEE 802.1X defines the port access control and MACsec Key Agreement machinery that discovers authenticated peers and establishes the Secure Associations used by MACsec.

802.1AE

Defines MACsec entities, secured frame format, Secure Channels, Secure Associations, cipher suites, validation, replay protection, and the SecTAG/ICV frame transformation.

802.1X and MKA

Provides the MACsec Key Agreement protocol. MKA authenticates peers, elects a key server, distributes Secure Association Keys, and rotates keys.

AUTOSAR usage

Automotive stacks usually integrate MACsec with an MKA daemon or module, port-specific configuration, monitoring, and explicit handling of unprotected traffic classes.

Training boundary: this document explains the protected data frame format. It intentionally separates that from MKA control frames. MKA control traffic uses IEEE 802.1X/EAPOL control-plane PDUs. MACsec protected data frames use EtherType 0x88E5.

2. MACsec PDU placement inside Ethernet

A normal Ethernet frame has destination MAC address, source MAC address, an EtherType or length field, MAC service data, and a frame check sequence. MACsec transforms the MAC service data into a secured MACsec PDU by inserting a SecTAG after the source MAC address and by appending an Integrity Check Value before the FCS.

Unprotected frame

Destination MAC6 BVisible to every bridge on the link. MACsec does not hide the outer destination address in the basic frame format.
Source MAC6 BVisible to every bridge on the link. Some privacy extensions address traffic correlation, but the ordinary protected frame still carries outer MAC addresses.
EtherType / VLAN2 or more BFor a VLAN-tagged frame, this begins with 0x8100 or another tag protocol identifier. This becomes part of the User Data protected by MACsec.
PayloadUser DataThe MAC client data, including Layer 3 packet and possibly VLAN tags. MACsec can integrity-protect it only, or both encrypt and integrity-protect it.
FCS4 BThe physical MAC calculates FCS after MACsec has produced the protected frame.

MACsec protected frame

Destination MAC6 BAuthenticated by the ICV, but still transmitted in clear text so the link can deliver the frame.
Source MAC6 BAuthenticated by the ICV. May also help infer the SCI when the explicit SCI is omitted and the End Station bit is used.
SecTAG8 or 16 BContains MACsec EtherType 0x88E5, TCI/AN, Short Length, Packet Number, and optional Secure Channel Identifier.
Secure Dataoriginal User Data transformedIn integrity-only mode, this remains readable. In confidentiality mode, all or part of it is ciphertext depending on the configured confidentiality offset.
ICVtypically 16 BAuthentication tag produced by the cipher suite. It validates the outer MAC addresses, SecTAG, and protected data. It is not the Ethernet FCS.
FCS4 BThe FCS protects against random transmission errors. It is recomputed on the final transmitted Ethernet frame.
Important: Preamble, Start Frame Delimiter, Interpacket Gap, and physical coding are outside the MACsec protected PDU. The Ethernet FCS is also not a MACsec cryptographic authentication tag.

3. SecTAG header format

The SecTAG is the MACsec-specific header immediately after the source MAC address. Its base form is 8 octets. If the Secure Channel Identifier is explicitly present, the SecTAG grows to 16 octets.

MACsec EtherType2 B
0x88E5Identifies the frame as a MACsec protected frame. It replaces the original EtherType position. The original EtherType moves into Secure Data as part of the protected User Data.
TCI / AN1 BTag Control Information plus Association Number. It says whether an SCI is present, whether confidentiality is used, and which receive Secure Association should validate the frame.
SL1 B
6 useful bitsShort Length. Encodes the Secure Data length when it is less than 48 octets. Otherwise it is zero. Values 48 and above are invalid in normal validation.
PN4 BPacket Number. It is unique per transmit Secure Association and is used for replay protection and AES-GCM nonce construction. PN 0 is not a normal transmitted data PN.
SCIoptional 8 BSecure Channel Identifier. Normally composed from a 6-octet system identifier and a 2-octet port identifier. Present only when indicated by the SC bit.
FieldSizeWhy it existsCommon implementation interpretation
MACsec EtherType2 octetsMarks the frame as a MACsec frame using 0x88E5.The original inner EtherType, such as IPv4 0x0800 or VLAN TPID 0x8100, is no longer in the outer EtherType slot.
TCI/AN1 octetCombines tag-control flags and the 2-bit Association Number.Receiver uses the AN to choose one of up to four receive SAs for the selected Secure Channel.
Short Length1 octet, with valid short values below 48Lets the receiver recover the original short MAC service data length after padding.If Secure Data length is 48 octets or more, SL is normally 0.
Packet Number4 octets visible in the tagPrevents replay and contributes to the cryptographic nonce.Transmitter increments per SA. Receiver validates monotonicity according to replay-window policy.
SCI8 octets when presentExplicitly identifies the Secure Channel.Often included when the receiver cannot safely infer the transmitting Secure Channel from the port or source address.

Do not teach E/C as a four-mode selector. Valid data frames normally use E=0,C=0 for integrity-only or E=1,C=1 for confidentiality. The confidentiality offset is not encoded as a separate SecTAG byte. It is an SA/cipher-suite operating parameter known to both peers.

5. Integrity and encryption boundaries

MACsec uses authenticated encryption principles. Even when confidentiality is disabled, the receiver still verifies that the protected frame was generated by a peer possessing the active Secure Association Key.

Authenticated by the ICV

Outer DAOuter SASecTAGSecure Data

The ICV catches tampering with addresses, PN, AN, VLAN tag, IP header, transport header, or payload, depending on what is present in the frame.

Not authenticated by MACsec ICV

PreambleSFDInterpacket gapFCS as a separate CRC

The FCS is recalculated by the Ethernet MAC and catches accidental transmission errors. It is not a substitute for the MACsec ICV.

With a non-zero confidentiality offset, the first configured number of octets of User Data are carried in clear text but still included in the authenticated data. That allows limited classification on early headers while still preventing undetected modification.

6. Protection modes

For a frame-format discussion, the three practical operating modes are easiest to understand by looking at the Secure Data region.

ModeTCI E/CSecure Data on the wireWhat an observer can seeWhat an attacker cannot do silently
Integrity onlyE=0, C=0Original User Data remains visible.VLAN, EtherType, IP, transport headers and payload if not otherwise encrypted.Modify protected fields, replay old frames outside policy, or forge frames without SAK.
Confidentiality, offset 0E=1, C=1All User Data after SecTAG is ciphertext.Outer DA, outer SA, SecTAG, length/timing, and ICV.Read or modify the protected User Data.
Confidentiality with offsetE=1, C=1First offset octets are clear and authenticated; the rest is ciphertext.Early headers up to the configured offset, plus outer MAC addresses and SecTAG.Modify the clear headers silently or read encrypted portion.
Automotive interpretation: integrity-only may be chosen where confidentiality is unnecessary but tamper and replay protection matter. Full confidentiality is strongest for data privacy but hides inner VLAN/IP fields from devices that do not terminate MACsec. Confidentiality offset is a compromise for systems that need limited classification on initial header bytes.

8. Interesting scenarios

Scenario 1 — Attacker changes VLAN PCP to get higher priority

Start: ECU transmits a VLAN-tagged control frame with PCP 3. MACsec is in integrity-only mode because the payload is not confidential.
Attack: an inline device flips the VLAN TCI bits so the frame appears as PCP 7.
Receive: the receiver recomputes the authentication check using DA, SA, SecTAG, Secure Data, and the SAK.
Result: ICV verification fails. The frame is discarded before being delivered to the controlled port. The attack did not require encryption to be blocked.

Scenario 2 — ECU A talks to ECU B through a switch

MACsec is normally link protection, not end-to-end IPsec. If ECU A reaches ECU B through a switch that participates in MACsec, the switch terminates and regenerates protection per port.

ECU ASecY A
SC A→SWSAK #1
Switch Port 1decrypt + validate
Switch Port 2encrypt + protect
SC SW→BSAK #2
ECU BSecY B

The frame that leaves switch port 2 has a new PN, possibly a different AN, a different SCI, and a new ICV. A failure on the first link does not produce a valid frame on the second link.

Scenario 3 — Replay attack using an old valid frame

Capture: attacker records a valid MACsec frame with PN 1050 and correct ICV.
Progress: the receiver has already accepted frames up to PN 1102 on the same receive SA.
Replay: attacker retransmits the old PN 1050 frame without modification.
Result: ICV can still be mathematically valid, but replay validation rejects the stale PN according to the configured replay window.
Replay protection is not merely a cryptographic property of the ICV. It is the receiver's stateful policy over packet numbers for the selected Secure Association.

Scenario 4 — Key rollover with the Association Number

The 2-bit AN allows a Secure Channel to carry up to four distinguishable Secure Associations. During key rollover, the transmitter can start sending with a new AN while the receiver still accepts the previous AN for a transition period.

TimeActive transmit SATCI/AN fieldReceiver behaviour
T0SA using AN 0AN=0Receiver validates with SAK for AN 0.
T1MKA distributes new SAKNo data-frame change yetReceiver installs new receive SA but may not use it yet.
T2Transmitter switches to AN 1AN=1Receiver selects SAK for AN 1 and validates new frames.
T3Old SA retiredAN 0 rejectedOld frames fail because the receive SA is no longer valid or replay state rejects them.

Scenario 5 — SCI omitted to save 8 bytes

On a simple point-to-point link, the receiver may infer the Secure Channel from the controlled port and source-address context. This allows the explicit SCI to be omitted and reduces overhead by 8 octets.

SCI present

SC=1SecTAG = 16 B

Best when several possible secure channels can arrive at the receiver or when explicit management identity is useful.

SCI omitted

SC=0SecTAG = 8 B

Best when the link context is unambiguous. The receiver still needs a secure way to map the frame to the correct SC and SA.

An omitted SCI is not a missing identity check. It means identity is inferred from configured context rather than carried explicitly in every frame.

Scenario 6 — Frame preemption can reorder protected traffic

Automotive Ethernet may combine MACsec with Time-Sensitive Networking features such as frame preemption. If frames from different priorities can be reordered by the network behaviour, strict replay protection can become difficult if all priorities share one packet-number sequence.

Priority A frame: PN 200 is large and starts transmission.
Priority B frame: a higher-priority express frame with PN 201 preempts or overtakes the lower-priority frame in an allowed architecture.
Receiver risk: if strict replay is expecting monotonic arrival on a single SC, PN 201 may arrive before PN 200.
Standards-aware design: separate transmit Secure Channels can be used for traffic classes where reordering is expected, preserving strict replay behaviour per SC.

Scenario 7 — MKA control frame is not a MACsec data frame

A common lab confusion is to capture both 0x888E EAPOL/MKA frames and 0x88E5 MACsec protected data frames and treat them as one PDU family. They serve different roles.

Frame typeOuter EtherTypePurposeTypical parser action
MKA / EAPOL control0x888EAuthenticate peers, announce capabilities, distribute keys, manage live peers.Pass to controlled/uncontrolled port logic and MKA engine.
MACsec protected data0x88E5Carry protected user data using installed Secure Associations.Pass to SecY receive validation, decrypt if required, then deliver recovered User Data.

9. MACsec transmit and receive pipeline

Transmit path

Controlled Port accepts MAC service request: DA, SA, and User Data arrive from the MAC client.
Classify security context: SecY identifies the transmit Secure Channel and current transmit Secure Association.
Assign PN: next packet number is consumed. PN reuse with the same SAK must not occur.
Construct SecTAG: write EtherType 0x88E5, TCI/AN, SL, PN, and optional SCI.
Apply cipher suite: produce Secure Data and ICV using SAK, PN, SCI/context, and authenticated data.
Transmit final Ethernet frame: physical MAC adds or recalculates FCS.

Receive path

Parse outer Ethernet: DA and SA are read. EtherType 0x88E5 routes the frame to the MACsec receive path.
Validate SecTAG structure: check version, SCI rules, SL validity, PN validity, and minimum frame size.
Identify SC and SA: use explicit SCI or configured inference, then use AN to choose the receive SA.
Replay check: compare PN with replay-protection state before or during validation policy.
ICV verification and decryption: accept only if cryptographic validation succeeds. Decrypt if confidentiality was used.
Deliver recovered User Data: the original EtherType/VLAN/IP data is delivered to the MAC client.

State carried outside the frame

CA Connectivity Association membership.

SC Secure Channel identity and direction.

SA Secure Association and selected SAK.

Replay window Receiver PN acceptance state.

Cipher suite GCM-AES-128, GCM-AES-256, or XPN variants depending on implementation and agreement.

Confidentiality offset Not a separate SecTAG field. It is configured/agreed state.

11. Validation pitfalls and “bad tag” examples

Bad SecTAG layout

Wrong EtherType, version bit set, impossible SCI combination, invalid SL value, or too-short frame for the claimed fields.

Wrong SA selection

SCI or inferred SC maps to a channel that has no live receive SA for the indicated AN.

Replay failure

PN is repeated, stale, zero, or outside the configured replay window for the selected SA.

ICV failure

Frame was modified, wrong SAK is installed, PN/nonce context is wrong, or confidentiality-offset/cipher-suite state is mismatched.

Quick parser checklist

  1. Is the outer EtherType 0x88E5?
  2. Is the TCI version value expected?
  3. Are ES, SC, and SCB combinations legal for the configured link?
  4. If SC is set, are 8 SCI bytes present before Secure Data?
  5. Is SL either zero or a short length less than 48?
  6. Is PN non-zero and acceptable for the receive SA replay policy?
  7. Does AN select a live receive SA for that Secure Channel?
  8. Does ICV verify using the selected SAK and cipher-suite parameters?
