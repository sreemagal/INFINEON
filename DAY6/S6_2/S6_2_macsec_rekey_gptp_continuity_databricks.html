Rekeying impact on gPTP continuity

A standards-oriented, packet-level guide to what happens when MACsec rolls from one Secure Association Key to another while generalized Precision Time Protocol traffic is running on the same Ethernet link.

802.1AEMACsec data-plane protection using SecY, Secure Channels, Secure Associations, SecTAG, Packet Number, and ICV.
802.1XMKA control plane that authenticates peers and distributes Secure Association Keys for MACsec.
802.1ASgPTP timing profile for time-sensitive Ethernet networks.
0 loss goalRekey should not cause Sync, Follow_Up, Pdelay, or Announce discontinuity.

The short answer

Correctly implemented MACsec rekeying should be transparent to gPTP. During a normal MKA-driven Secure Association Key rollover, both peers install a new receive Secure Association before one peer starts transmitting on the new Association Number. The receiver uses the Association Number in the SecTAG to select the correct receive Secure Association. If both old and new receive SAs overlap for a controlled time, gPTP packets keep flowing.

The problem is not the mathematical act of changing keys. The problem is the transition window. A gPTP Sync or peer-delay message can be delayed or dropped if the new Secure Association is not installed in hardware, if the replay window rejects a packet, if the controlled port is briefly blocked, if EAPOL/MKA is filtered, or if MACsec processing moves timestamps away from the real wire time.

Engineering rule: Treat MACsec rekey as a Layer 2 maintenance event that must be hitless for time traffic. Validate it with packet captures and counters, not only with “MACsec session up”.

What happens during a normal SAK rekey?

T0 Existing secure traffic
Peer A transmits protected frames on SA0 with AN=0. Peer B validates them using its receive SA0. gPTP messages, if protected, are just secure data frames.
T1 MKA decides to rekey
The Key Server prepares a fresh SAK. Rekey may be time-based, policy-based, administrative, triggered by PN exhaustion risk, or triggered by membership changes.
T2 New receive SA is installed
Peers install SA1 for receive before using it for transmit. This is the key condition for hitless rollover.
T3 Transmit switches to new AN
Peer A begins transmitting frames with AN=1 and fresh PN space. The SecTAG tells Peer B to validate with SA1.
T4 Overlap window
Peer B may still accept old AN=0 packets that were already in flight while accepting new AN=1 packets.
T5 Old SA retires
After the overlap, old SA0 is no longer used. gPTP should have seen no discontinuity.

gPTP continuity model

gPTP continuity means more than “packets are not dropped”. It means the timing servo continues receiving consistent timestamped events so that offset, frequency ratio, and path-delay estimates remain bounded.

What gPTP needs

  • Event messages delivered with predictable intervals.
  • Accurate egress and ingress timestamps close to the actual MAC/PHY transmit and receive instants.
  • Stable peer-delay measurement across each link.
  • No false loss of asCapable state.
  • No unintended grandmaster or time-aware system role change.

What MACsec rekey can disturb

  • Frame validation if receive SA is not ready.
  • Replay checking if PN/window state is wrong.
  • Latency if encryption hardware stalls during table update.
  • Port state if MKA liveness fails.
  • Timestamp correctness if timestamps are taken at the wrong layer boundary.

Deployment choice: protected gPTP or bypassed gPTP?

Option A

gPTP protected by MACsec

gPTP packets pass through the controlled port and are protected like other data. After validation, the receiver hands the PTP PDU to the gPTP stack.

Advantages

  • Link-layer integrity and origin authenticity for timing messages.
  • Consistent security policy for control traffic.
  • Protection against simple L2 injection on the secured link.

Rekey risk

  • SA rollover failure can drop gPTP packets.
  • Replay-window settings and PN transitions must be correct.
  • Timestamp placement must be carefully validated.
Option B

gPTP bypassed or left clear

gPTP packets are not protected by MACsec, commonly by explicit bypass policy based on EtherType, destination address, VLAN, or local implementation rules.

Advantages

  • SAK rekey does not directly block gPTP packet validation.
  • Timestamp path can be simpler on some hardware.
  • Useful for early bring-up and diagnosing security/timing separation.

Security risk

  • Timing messages are not protected by MACsec.
  • Misclassification can accidentally protect some PTP messages and bypass others.
  • Attack or fault injection on timing traffic remains a design concern.
Do not mix policies accidentally. A common failure is protecting Sync and Follow_Up but bypassing peer-delay messages, or protecting one VLAN/domain but bypassing another. That can keep the session “mostly alive” while corrupting the delay model.

Timestamp placement is the hidden issue

When MACsec is inserted into a TSN Ethernet port, you must ask: what exact event is timestamped? gPTP needs the timestamp of the frame as it actually leaves or enters the link. If the timestamp is taken before a variable MACsec queue or key-table stall, the timestamp can become optimistic or inconsistent.

The timestamp observes the protected frame as it is actually emitted or received. Fixed MACsec latency can be calibrated or naturally included. Variable rekey stalls become visible as delayed packet emission rather than hidden timestamp error.

What about MACsec overhead?

MACsec adds SecTAG and ICV bytes. That changes serialized frame length and can add a fixed processing delay. Fixed delay is usually manageable. Variable delay during rekey is dangerous because it can appear as Packet Delay Variation or timestamp error.

Diagnosis patterns

Pattern A: gPTP sequence gap + MACsec NotValid

Root cause is likely SA mismatch or validation failure. Look at AN and receive-SA installation timing.

Pattern B: no sequence gap + offset spike

Root cause is likely timestamp placement or variable MACsec pipeline latency rather than frame loss.

Pattern C: all protected traffic stops

Root cause is likely MKA liveness, controlled-port blocking, or EAPOL filtering rather than gPTP itself.

Pattern D: only Pdelay fails

Root cause may be multicast address handling, peer-delay classification, bypass inconsistency, or link-local filtering.

Design rules for robust gPTP continuity under MACsec rekey

  1. Use make-before-break SA rollover. New receive SA must be active before transmit switches to the new Association Number.
  2. Classify all gPTP messages consistently. Sync, Follow_Up, Announce, Pdelay_Req, Pdelay_Resp, and Pdelay_Resp_Follow_Up should not accidentally follow different security paths.
  3. Validate timestamp placement. Hardware timestamps should represent actual protected-frame transmit and receive timing, or any fixed security latency must be correctly compensated.
  4. Keep MKA visible and reliable. EAPOL/MKPDUs must not be filtered, consumed, or VLAN-misclassified by intermediate devices.
  5. Size replay windows for the real datapath. Consider traffic class ordering, multiple queues, preemption, and any path that can reorder frames.
  6. Rekey well before PN exhaustion. At high link rates, do not wait until PN is nearly exhausted before initiating rollover.
  7. Correlate counters, not symptoms. A gPTP offset spike is a symptom. The cause might be MACsec validation, MKA liveness, timestamp location, or queueing.
  8. Test rekey under load. Validate with normal traffic, full-rate traffic, gate schedules, and priority mixes, not only with an idle link.
Practical acceptance criterion: Force repeated MACsec rekeys while measuring gPTP offset, mean path delay, sequenceId continuity, SecY counters, and MKA logs. A production-grade TSN security design should show no packet validation gap and no unexplained timing jump at the rollover instant.
