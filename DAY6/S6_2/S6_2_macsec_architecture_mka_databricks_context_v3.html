1. MACsec architecture from the port outward

The clean mental model is Connectivity Association (CA) as the secure membership context, Secure Channel (SC) as the directional traffic relation, Secure Association (SA) as the keyed instance, and MAC Security Entity (SecY) as the enforcing port entity.

Key insight: the Secure Channel is long-lived and directional. The Secure Association is the rotating keyed state inside that Secure Channel. Rekeying switches the active Secure Association, not the identity of the Secure Channel.

Controlled Port

Carries protected user traffic after authorization and key establishment. This is where the MAC Security Entity enforces validation, replay protection, confidentiality policy, and drop or forward behavior.

Uncontrolled Port

Carries authentication and key-management traffic needed before the Controlled Port can safely carry protected traffic. MACsec Key Agreement traffic logically belongs here.

7. Counters and diagnostics as a failure-language

MACsec counters are best understood as symptoms pointing to classification, key-state, replay, validation, or mode-mismatch failures.

Framing and classification

in_pkts_no_tag, in_pkts_bad_tag. Think missing Secure Tag, wrong parsing, wrong bypass or protect rule, or peer not using MACsec.

Key-state mismatch

in_pkts_no_sa, in_pkts_no_sa_error. Think wrong Association Number, receiver missing the Secure Association, failed or incomplete MACsec Key Agreement rollover.

Replay and sequencing

in_pkts_late. Think old Packet Number, replay-window issue, duplicate frame after timeout, or packet reordering beyond tolerance.

Integrity and authentication

in_pkts_invalid, in_pkts_not_valid. Think wrong Secure Association Key, modified frame, wrong Integrity Check Value, or corruption in protected data.

Protection mode

out_octets_protected versus out_octets_encrypted. Use these to confirm whether integrity-only or confidentiality mode is really active.

Transmit-side issues

out_pkts_untagged, out_pkts_too_long. Think bypass rules, Maximum Transmission Unit expansion, Secure Tag / Integrity Check Value overhead, or frame-size planning.

Operational pattern: do not debug MACsec from packet captures alone. Combine captures with SecY counters, Secure Channel counters, MACsec Key Agreement state, Association Number state, Packet Number state, and Controlled Port or Uncontrolled Port policy.

8. Elucidative scenarios

These examples connect the PDU and header model to real failures and design decisions.

Scenario A: Passive tap on a camera Ethernet link

confidentiality useful

A camera Electronic Control Unit sends video or metadata to a zonal switch. An attacker taps the physical pair.

Without MACsec: attacker observes raw Ethernet payloads and can study traffic content.
Integrity-only MACsec: attacker still sees payload content but cannot modify frames without detection.
Confidentiality plus integrity: attacker sees MAC addresses, optional VLAN information depending on placement, MACsec EtherType, Secure Tag, and Integrity Check Value, but payload is encrypted.
Diagnostic counter view: modification attempts produce invalid or not-valid type symptoms at the receiver.

Scenario B: Rekey while generalized Precision Time Protocol is running

timing-sensitive

The link is carrying generalized Precision Time Protocol event messages and protected data. MACsec Key Agreement prepares a fresh Secure Association Key.

Correct behavior: receiver installs the new Secure Association before the sender switches Association Number.
Risk window: if the sender starts using Association Number = 1 before receiver has it, protected frames can be counted as no-SA or invalid.
Timing consequence: if generalized Precision Time Protocol frames are lost or delayed during rollover, the clock servo may see jitter or offset disturbance.
Mitigation: overlap Secure Associations, minimize rekey frequency, use Extended Packet Number where needed, and prefer a stable two-step timestamping architecture.

Scenario C: Wrong bypass rule for MACsec Key Agreement or timing traffic

policy bug

A switch or Electronic Control Unit accidentally sends a control class through the wrong path.

MACsec Key Agreement trapped incorrectly: key agreement may fail because it must be available on the Uncontrolled Port path.
Protected traffic bypassed incorrectly: user traffic may leak outside the Controlled Port protection boundary.
generalized Precision Time Protocol bypassed intentionally: timing may be easier to preserve, but the timing plane remains exposed unless separately protected or physically trusted.
Diagnostic view: mismatches show up as no-tag, bad-tag, untagged, or mode counters inconsistent with the expected security policy.

1. Data-communications context first

1.1 Start with the real security problem

On an Ethernet link, an attacker does not need to “break the whole network” to cause trouble. It is enough to listen passively, inject forged frames, replay previously captured frames, or modify traffic in transit. The IEEE 802.1 Security Task Group says IEEE 802.1AE Media Access Control Security (Media Access Control Security (MACsec)) provides authenticity, integrity, and optionally confidentiality for the MAC service, while IEEE 802.1X Port-Based Network Access Control provides the authentication and key-agreement framework used to bring MACsec up. The practical Ethernet threat is set in the same terms: eavesdropping, arbitrary message injection, replay, and in-transit modification.

That gives the first deep idea to get right: MACsec is a Layer 2 link-security mechanism. It protects traffic as it traverses an Ethernet link between directly connected peers or ports. It is not, by itself, an end-to-end application-security protocol. It is a point-to-point or hop-by-hop protection mechanism, and the IEEE 802.1AE page frames it as protection for communication through the MAC service between peers.

1.2 The core MACsec architecture: Connectivity Association, Secure Channel, Secure Association, and MAC Security Entity (SecY)

The architecture is easiest to understand in four layers.

A Connectivity Association (Connectivity Association (CA)) is the overall secure relationship among the devices that are supposed to communicate securely on a given local-area-network context.

A Secure Channel (Secure Channel (SC)) is one directional secure traffic relationship inside that CA.

A Secure Association (Secure Association (SA)) is one keyed cryptographic operating context used within an SC.

The port-side entity that enforces all this is the MAC Security Entity (SecY).

IEEE 802.1AE defines an SC as a security relationship that provides guarantees for frames transmitted from one member of a CA to the others, and it says an SC is supported by a sequence of SAs, which allows periodic use of fresh keys without tearing the whole relationship down.

It also defines the Secure Channel Identifier (Secure Channel Identifier (SCI)) as a unique identifier composed of a MAC address and a Port Identifier.

That “sequence of Secure Associations” point is extremely important.

It means an SC is the long-lived secure traffic relationship, while the individual SAs inside it can be rotated over time.

In other words, MACsec was designed for rekeying without link teardown from the start.

IEEE 802.1X then adds the port model around this. Its standard page says port-based network access control uses an unsecured MAC service to support both a Controlled Port, which carries secure access-controlled communication, and an Uncontrolled Port, which carries authentication and key-management traffic needed to bring the secure Controlled Port up. That distinction is central to understanding how MACsec Key Agreement (MACsec Key Agreement (MKA)) works on a live link.

1.3 MACsec frame format from first principles

A MACsec-protected Ethernet frame is still an Ethernet frame, but the protected part is wrapped in a specific MACsec protocol data unit structure. IEEE 802.1AE shows the MACsec EtherType as 88-E5 and describes the protected data unit as three major components: SecTAG, Secure Data, and Integrity Check Value (Integrity Check Value (ICV)). The same figure shows the SecTAG fields as Tag Control Information (Tag Control Information (TCI)), Association Number (Association Number (AN)), Short Length (Short Length (SL)), Packet Number (Packet Number (PN)), and optionally the SCI.

The SecTAG is therefore the control-and-sequencing header for MACsec. The TCI carries protocol-control bits. The AN tells the receiver which Secure Association inside the Secure Channel is being used. The PN provides the per-frame sequence number used for replay protection and nonce construction. The SCI, when present, identifies the transmitting Secure Channel. IEEE 802.1AE also says the 32 least significant bits of the PN are encoded in the SecTAG and that they support replay protection.

The ICV is the authentication result attached to the frame. IEEE 802.1AE defines the ICV as the value derived by algorithmic transformation on the protected data unit and says it is recalculated and compared by the receiver to detect modification. The standard also states that the ICV length depends on the cipher suite and is between 8 and 16 octets, and it specifically notes that the ICV protects the destination and source MAC address parameters as well as all fields of the MACsec protocol data unit.

So the clean frame-format picture is:

  • Ethernet header
  • MACsec EtherType 88-E5
  • SecTAG with TCI, AN, SL, PN, and optional SCI
  • Secure Data
  • ICV.

1.4 Cipher suites: Galois/Counter Mode–Advanced Encryption Standard 128 and 256

IEEE 802.1AE supports both integrity without confidentiality and integrity with confidentiality. The standard says a conformant implementation must include at least one cipher suite that provides integrity without confidentiality, and that this requirement is met by the mandatory default cipher suite. Its cipher-suite table then shows Galois/Counter Mode–Advanced Encryption Standard 128 (GCM-AES-128) as mandatory and GCM-AES-256 as optional in the base standard. The IEEE 802.1AE web page also states that GCM-AES-256 was added as an option in the later amendment history.

That means the two main deployment choices are:

  • GCM-AES-128 as the mandatory baseline
  • GCM-AES-256 where the implementation and system policy want the larger key size.

1.5 Integrity-only versus encryption modes

MACsec is often described as “Ethernet encryption,” but that is only half-right. IEEE 802.1AE explicitly supports an integrity-only mode. The standard says that when the default cipher suite is used without confidentiality, the Secure Data is the unmodified User Data and a 16-octet ICV is used. In that mode, tools that understand MACsec parsing can still read the payload, but they cannot modify it without detection by the receiving SecY. The OPEN Alliance automotive MACsec profile also explicitly requires support for integrity without confidentiality and for confidentiality and integrity with confidentiality offset 0.

So the correct distinction is:

  • integrity-only mode means the payload remains visible but tamper-evident
  • confidentiality plus integrity mode means the payload is encrypted and authenticated.

That distinction matters operationally because integrity-only mode is often easier for diagnostics, testing, and protocol observability, while full confidentiality protects payload secrecy as well. The OPEN Alliance profile’s requirement for confidentiality offset 0 also tells you that automotive deployments are steering away from partial-cleartext modes and toward either plain integrity-only or full encrypt-and-authenticate behavior.

1.6 MKA and the 802.1X key hierarchy: Connectivity Association Key (CAK), Key Encryption Key (KEK), Integrity Check Value Key (ICK), and Secure Association Key (SAK)

MKA is the control protocol that makes MACsec usable in practice. The IEEE 802.1 Security Task Group says IEEE 802.1X specifies the MACsec Key Agreement (MKA) protocol and that MKA discovers mutually authenticated MACsec peers, elects one as a Key Server, and distributes the Secure Association Keys (SAKs) used by MACsec to protect frames. IEEE 802.1AE says the SAK is the secret key used by a Secure Association.

The AUTOSAR MACsec requirements and the Technica automotive MACsec presentation make the hierarchy especially clear. They describe the Connectivity Association Key (CAK) as the long-term secret shared by the members of a CA, and then define two keys derived from it: the Key Encryption Key (KEK), used to encrypt or decrypt a SAK inside MKA messages, and the Integrity Check Value Key (ICK), used to protect MKA messages themselves. AUTOSAR further says both KEK and ICK are derived from the CAK using the Key Derivation Function (Key Derivation Function (KDF)) defined in IEEE 802.1X. Technica’s slides summarize the same relationship and say MKA generates fresh SAKs from either a pre-shared CAK or an Extensible Authentication Protocol (Extensible Authentication Protocol (EAP))-generated CAK.

So the clean key hierarchy is:

  • CAK = long-term shared secret for the Connectivity Association
  • KEK = derived from CAK, used to protect key transport in MKA
  • ICK = derived from CAK, used to protect MKA message integrity
  • SAK = distributed session key used by the SecY for actual MACsec frame protection.

That is the key point most shallow explanations miss: the CAK is not the traffic-encryption key. The SAK is. The CAK is the root from which MKA derives the other keys it needs to authenticate the peers and safely distribute a fresh SAK.

1.7 MACsec versus Internet Protocol Security (IPsec) versus Transport Layer Security (TLS)

Now to the comparison.

MACsec secures traffic at the Layer 2 MAC boundary. It protects Ethernet frames hop by hop on a link or between directly connected MACsec peers and can therefore cover non-IP traffic and ordinary Ethernet control/data traffic on that link. IEEE 802.1AE describes it as protection for the MAC service itself.

IPsec, defined by RFC 4301, is the security architecture for the Internet Protocol layer. RFC 4301 says it provides security services for traffic at the IP layer, in both IPv4 and IPv6, and lists services including access control, connectionless integrity, data-origin authentication, replay detection, confidentiality, and limited traffic-flow confidentiality. It can protect host-to-host, gateway-to-gateway, or host-to-gateway IP paths.

Transport Layer Security (TLS), defined in its current form by RFC 8446, is a client/server application security protocol. RFC 8446 says TLS allows client/server applications to communicate over the Internet in a way designed to prevent eavesdropping, tampering, and message forgery. So TLS is normally used to protect specific application connections rather than arbitrary link traffic.

So the clean layering comparison is:

  • MACsec protects Ethernet link traffic
  • IPsec protects IP traffic
  • TLS protects application sessions.

That layering difference also explains practical behavior. The Technica automotive MACsec presentation says MACsec can cover more communication because it is below IP and therefore can protect more protocols, including multicast and broadcast on the protected link. It also says MACsec, IPsec, and TLS can all exchange fresh keys, but MACsec startup can be faster because it is one security relationship per link, whereas IPsec is typically per Electronic Control Unit relationship and TLS is typically per application connection. That is an industry presentation rather than a normative standard, but it is a good practical summary of the deployment difference.

The most important caution is that none of these alone is “the answer to all security.” MACsec is hop-by-hop, so traffic is unprotected inside trusted intermediate switches after decryption. IPsec is constrained to IP-layer traffic. TLS protects specific application channels and not arbitrary Layer 2 or Layer 3 control traffic. The Technica presentation explicitly calls MACsec and end-to-end mechanisms complementary rather than mutually exclusive.

1.8 Rekeying and what it means for continuity

Because an SC is supported by a sequence of SAs, MACsec is expected to rekey. Technica’s automotive MACsec slides explicitly say MKA allows fresh SAKs to be generated for MACsec and that the goal in automotive is to minimize rekey frequency, ideally to one key exchange per power cycle, with Extended Packet Number (Extended Packet Number (XPN)) support helping make that feasible. IEEE 802.1AE’s amendment history also notes that extended packet numbering was added so that more than (2^{32}) frames can be protected with a single SAK.

That already gives the key continuity insight: rekeying is not free. It changes the active SA, packet-number space, and keying state. In ordinary data communications that is mainly a security/availability concern. In time synchronization, it can also be a timing-continuity concern if the protected timing traffic cannot tolerate perturbation.

1.9 Rekeying impact on generalized Precision Time Protocol continuity

The cleanest standards-based statement here is that generalized Precision Time Protocol (generalized Precision Time Protocol (gPTP)) can run with MACsec inline, but timing accuracy becomes sensitive to how timestamping and protection are combined. The IEEE 802.1AS draft model explicitly shows MACsec inline with the 802.3 MAC and PHY in the time-aware system model. A later analysis of security controls on gPTP then says MACsec presents obstacles for one-step timing because encryption and decryption after timestamping can create variable delay, and concludes that for MACsec to protect Precision Time Protocol without degrading accuracy, latency variation must be constrained and Precision Time Protocol (PTP) should be used in two-step mode rather than one-step.

So the practical interpretation is:

  • MACsec itself does not forbid gPTP continuity
  • but rekeying and protection changes must not create unacceptable timing disturbance
  • one-step timing is the more fragile case
  • two-step timing is the safer fit when timing traffic is MACsec-protected.

The automotive implication of that will come later, but even in pure data-communications terms the message is already clear: timing continuity depends on both cryptographic continuity and timestamp-path continuity.

  • out_pkts_encrypted.

Those counters already suggest a useful diagnostic taxonomy.

in_pkts_no_tag or in_pkts_bad_tag usually point to framing or classification problems. in_pkts_no_sa or in_pkts_no_sa_error point to key-state or peer-state mismatch. in_pkts_late points to replay-window or sequencing trouble. in_pkts_invalid and in_pkts_not_valid point toward failed validation or frame-authentication problems. out_octets_protected versus out_octets_encrypted tells you whether the system is mainly operating in integrity-only or confidentiality mode. This last interpretation is an operational inference from the counter names and the MACsec mode model.

The OPEN Alliance profile also says that if MACsec cannot drop a frame that failed verification, such as an authentication failure, it shall forward the Ethernet frame with a corrupted Frame Check Sequence (Frame Check Sequence (FCS)). That is an especially important diagnostic detail because it tells you how some implementations may surface verification failure downstream when direct discard is not possible in hardware.

2. Automotive perspective and context

2.1 The threat landscape on automotive Ethernet

In vehicles, Ethernet security failures are not just privacy failures. NHTSA’s 2022 cybersecurity guidance says the attack surface includes all interfaces where an unauthorized user can inject or extract data or modify system behavior. It explicitly warns about open IP ports, exposed services, weak segmentation, and wireless entry points, and gives concrete examples of spoofed brake-related or vehicle-state messages leading to dangerous consequences.

Marvell’s automotive MACsec tutorial describes the Ethernet-specific side of this in more direct link terms: passive tapping, replay, arbitrary frame injection, message modification, and man-in-the-middle style attacks are all plausible on in-vehicle Ethernet links. That is why link security is especially attractive on camera links, zonal-controller uplinks, gateway links, and other critical backbones.

So the automotive threat landscape is best understood as a combination of:

  • external entry
  • lateral movement
  • spoofing of high-value in-vehicle traffic
  • direct link-level attacks on Ethernet segments.

2.2 Why MACsec is a strong automotive fit

MACsec is appealing in automotive because it secures the Ethernet link itself, not only one application running on top of it. That matters in cars because many important traffic types on the link are not ordinary client/server application sessions. They may include generalized Precision Time Protocol, diagnostics, Audio Video Transport Protocol, control traffic, or tunneled payloads that are not good fits for TLS. Marvell’s automotive presentation explicitly says MACsec can protect more communication because it is below IP and can cover more protocols, including multicast and broadcast on the protected link.

That does not make MACsec “better than IPsec or TLS in all cases.” It makes it a very natural baseline link-protection layer inside the vehicle, especially for switched Ethernet segments carrying mixed traffic. IPsec and TLS remain complementary where network-layer or end-application protection is also needed. The Technica slides explicitly describe them as complementary solutions rather than substitutes.

2.3 Automotive-specific MACsec profile choices

The OPEN Alliance automotive MACsec profile makes several interesting choices that are highly relevant to a training course.

It requires support for:

  • at least one transmit Secure Channel and one receive Secure Channel per Ethernet port
  • operation without requiring IEEE 802.3br frame pre-emption support
  • the SecTAG without encoding the SCI, to save 8 bytes per frame
  • integrity without confidentiality
  • confidentiality plus integrity with confidentiality offset 0.

It also requires the ability to forward selected packets to the Uncontrolled Port based on at least:

  • destination MAC address
  • EtherType
  • matching IEEE 802.1Q VLAN identifier,

in both transmit and receive directions, with at least 8 such rules.

Those choices say a lot about automotive priorities. The profile is trying to reduce hardware complexity, control overhead, preserve useful observability, and still keep enough flexibility to exempt selected traffic from normal MACsec protection when system design requires it.

2.4 MKA and startup time in vehicles

Automotive systems care a great deal about startup time and deterministic bring-up. The AUTOSAR MACsec explanatory document says MKA discovers MACsec peers and negotiates the keys required to secure the link, and it notes that two key-generation mechanisms exist: Pre-Shared Keys (PSK) and keys based on successful EAP authentication. The same AUTOSAR guidance and the Technica presentation both emphasize pre-shared CAKs for fast startup, and Technica explicitly recommends pre-shared CAKs for the fastest bring-up.

That is a very automotive-specific pressure. In enterprise networks, slower certificate-driven onboarding may be acceptable. In a vehicle, a several-second startup penalty on a critical Ethernet link may be completely unacceptable. So the automotive key-hierarchy story is not only about security hygiene. It is also about meeting boot-time and availability constraints.

2.5 Rekeying and gPTP continuity in automotive systems

This is one of the most important applied design questions.

If a vehicle uses MACsec to protect timing traffic, rekeying must not create visible timing discontinuity. The general timing-security analysis says MACsec protection of PTP-like traffic is feasible, but one-step mode is problematic because encryption and decryption after timestamping can create delay variation. It concludes that protecting time synchronization without degrading accuracy requires constrained latency variation and the use of two-step mode. The automotive-minded Technica slides then add another operational recommendation: avoid frequent rekeying by aiming for one key exchange per power cycle, with Extended Packet Number (XPN) support making that practical.

So the automotive continuity lesson is:

  • use MACsec in a way that does not disturb the timestamp path
  • strongly prefer two-step time synchronization if timing traffic is MACsec-protected
  • minimize rekey frequency where possible, especially on timing-critical links
  • use XPN or equivalent packet-number-space expansion so normal operation does not force frequent SA rollover.

That is not merely a “security setting.” It is a combined security-and-timing architecture decision.

2.6 MACsec counters and diagnostics in automotive programs

In automotive programs, counters are not just for debugging. They are part of bring-up, production validation, field diagnosis, and safety/security monitoring.

The OPEN Alliance automotive MACsec profile explicitly requires a baseline set of counters, including transmit SecY counters, receive SecY counters, transmit Secure Channel counters, and receive Secure Channel counters. It also requires frame verification according to IEEE 802.1AE and defines what to do if verification failures cannot be dropped directly.

That makes the most useful automotive diagnostic interpretation something like this:

  • rising in_pkts_bad_tag or in_pkts_no_tag suggests framing, classification, or peer mismatch
  • rising in_pkts_no_sa suggests key-state or Secure Association synchronization trouble
  • rising in_pkts_late suggests replay-window or sequencing issues
  • rising in_pkts_invalid or in_pkts_not_valid suggests integrity/authentication failures
  • out_octets_protected and out_octets_encrypted help verify which protection mode is actually in use
  • if verification failures are being forwarded with bad FCS instead of being dropped, downstream diagnostics must be able to interpret that correctly.

That is why MACsec counters should be taught as a diagnostic model, not just a statistics list.

3. Summary

In pure data-communications terms, MACsec is a Layer 2 hop-by-hop security architecture built around Connectivity Associations (CA), Secure Channels (SC), Secure Associations (SA), and the MAC Security Entity (SecY). A MACsec frame carries the 88-E5 MACsec EtherType, a SecTAG with Tag Control Information (TCI), Association Number (AN), Short Length (SL), Packet Number (PN), and optional Secure Channel Identifier (SCI), followed by Secure Data and an Integrity Check Value (ICV). MACsec supports both integrity-only and confidentiality-plus-integrity modes, with GCM-AES-128 mandatory and GCM-AES-256 optional in the base standard. MKA, from IEEE 802.1X, discovers peers, elects a key server, and uses the Connectivity Association Key (CAK) to derive the Key Encryption Key (KEK) and Integrity Check Value Key (ICK) and distribute a fresh Secure Association Key (SAK).

Compared with other security protocols, MACsec protects Ethernet link traffic, IPsec protects IP traffic at Layer 3, and TLS protects client/server application sessions. In automotive systems, MACsec is attractive because it covers mixed link traffic directly, but it is still complementary to higher-layer security. Rekeying must be designed carefully around time synchronization, especially gPTP, and the safest approach is usually to minimize rekey frequency and use two-step timing where timing traffic is MACsec-protected. Finally, the MACsec counters and diagnostics matter operationally because they reveal whether the system is unprotected, mis-keyed, replaying, failing validation, or silently using a different protection mode than intended.
