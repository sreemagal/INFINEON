Standards-based training note

MKA / IEEE 802.1X key hierarchy for Automotive Ethernet

A practical, protocol-level explanation of how MACsec Key Agreement (MKA) uses CAK, ICK, KEK, and SAK to bring up a secure Ethernet link, how the EAPOL-MKA packets look on the wire, and how to debug common automotive scenarios.

0x888E
EAPOL EtherType carrying EAPOL-MKA control traffic.
Type 5
EAPOL packet type used for MKA. The packet body is the MKPDU.
0x88E5
MACsec data-frame EtherType after SecTAG insertion.
AN 0..3
Association Number selects which Secure Association Key protects the frame.
One-sentence mental model: MKA is the authenticated control-plane conversation that proves both ports belong to the same Connectivity Association and safely installs short-lived MACsec data-plane keys into the MAC Security Entity.

1. Standards map: where each rule comes from

MKA is often discussed together with MACsec, but the two standards have different jobs. IEEE 802.1X defines Port-Based Network Access Control and the MKA protocol used to discover and establish MACsec security associations. IEEE 802.1AE defines the MACsec data-plane service, SecTAG, Secure Channel, Secure Association, packet number, and integrity/encryption behavior.

3. Key hierarchy: CAK, ICK, KEK, SAK

The MKA hierarchy separates membership, control-plane authentication, key transport, and data protection. This separation is the main reason MKA is easier to reason about than a single “one key protects everything” design.

Important security rule: never precompute or reuse SAKs as static configuration. SAKs are session/data-plane keys and must be fresh enough to avoid packet-number reuse and cryptographic nonce reuse.

4. PDU format: what the wire actually carries

MKA packets are not IP packets. They are EAPOL frames carried directly over Ethernet. In a normal direct link, the destination is the EAPOL group address and the EtherType is 0x888E. The EAPOL packet type identifies the body as EAPOL-MKA.

Bridge behavior insight: the default EAPOL group address is link-local. A normal bridge is not expected to forward it as ordinary traffic. That is why MKA is naturally a direct-peer / hop-by-hop protocol, and why “MKA across a random switched cloud” often fails unless the network has an explicit tunneling or provider-mode design.

The EAPOL body is the MACsec Key Agreement Protocol Data Unit (MKPDU). It is a sequence of parameter sets, beginning with the Basic Parameter Set and normally ending with an MKA integrity value so the receiver can validate the control message.

Why the KEK exists: the peer should be able to receive the SAK over the link without the SAK appearing in cleartext inside the control frame. The KEK is the key-encryption key used for this wrapping operation.

Physical link becomes active. The PHY or switch port reports link-up, but the upper-layer communication stack should not assume that secure communication is ready.
2
PAE starts the KaY. The MKA participant checks whether the CAK is available for the configured CKN. Without CAK, there is no valid ICK or KEK.
3
MKPDUs are exchanged. Each side advertises CKN, SCI, MI, MN, MACsec capability, MACsec desired status, and key-server information.
4
Peer membership is validated. The receiver uses the derived ICK to validate the MKA ICV. Matching CKN alone is not enough.
5
Key server chooses cipher and distributes SAK. The key server generates a fresh SAK, wraps it with KEK, and advertises SAK use information.
6
SecY is programmed. Receive and transmit Secure Associations are installed, including SAK, AN, SCI/SSCI, packet-number handling, replay settings, and confidentiality offset.
7
Controlled port becomes operational. Now normal Ethernet payloads flow as MACsec frames, and the ECU may begin upper-layer protocols.

8. Automotive design patterns

Automotive Ethernet usually wants fast deterministic startup, controlled key storage, and simple operational ownership. That drives several practical patterns.

ECU ↔ Switch port
Common direct-peer pattern. The switch port and ECU have a shared CKN/CAK profile. MKA must finish before upper-layer services are considered reachable.
Switch ↔ Switch uplink
Used between central/zonal switches. MACsec can protect the trunk, but VLAN/stream/application authorization is still required after decryption.
HSM-mediated keys
CAK storage, ICK/KEK derivation, SAK unwrap, and SAK installation may be mediated by HSM/CSM to avoid exposing raw secrets to application software.

9. Observability and debug guide

The fastest way to debug MKA is to separate the problem into transport, membership, agreement, SAK distribution, and data-plane protection.

1. Do I see EAPOL-MKA?
Filter for EtherType 0x888E and EAPOL type 5. If absent, check link state, MKA enablement, EAPOL DA handling, VLAN/bypass, and port-control state.
2. Do both sides advertise the same CKN?
Different CKN means they are not trying the same CAK profile. Same CKN does not prove same CAK.
3. Does ICV validation succeed?
Matching CKN with ICV failure usually indicates wrong CAK or wrong derived ICK. Do not move to SAK debugging yet.
4. Is a key server active?
No key server means no Distributed SAK. In fixed-role automotive designs, inspect configuration first.
5. Is Distributed SAK present?
If absent, look for cipher mismatch, live-peer failure, or key-server role issue. If present but no data traffic works, look at unwrap/SecY install.
6. Are MACsec data frames valid?
Filter EtherType 0x88E5. Check AN, PN/XPN, SCI, replay window, and SecY counters.

11. Common pitfalls and design checks

PitfallCorrect interpretationDesign check
“CKN is a secret.”CKN is a name. The CAK is the secret.Do not derive security from hiding the CKN. Protect CAK provisioning and storage.
“CAK encrypts Ethernet data.”CAK authenticates CA membership and derives ICK/KEK. SAK protects data frames.Audit whether SAK rotation and PN exhaustion handling work.
“KEK is the MACsec data key.”KEK wraps SAKs. The SecY uses SAKs for frame protection.Separate key-wrap failure from SecY data-plane ICV failure.
“EAPOL-MKA is already inside MACsec.”MKA bootstraps MACsec and uses ICK for its own integrity. It must be allowed before the controlled port is operational.Ensure control-plane admission allows EAPOL-MKA but does not accidentally bypass unrelated traffic.
“MACsec solves application authorization.”MACsec proves link/port membership and protects link traffic. It does not decide whether a diagnostic tester is authorized to run a UDS routine.Keep DoIP/UDS authentication, SecOC, firewalling, VLAN policy, and service authorization.
“A switch port can transparently forward MKA everywhere.”Default EAPOL-MKA is link-local and direct-peer oriented.For non-direct designs, explicitly validate provider/tunneling behavior and threat model.
“Fail-open is harmless because it is temporary.”Temporary plain traffic can still leak or permit unsafe control flows.Limit permissive mode, log it, and ensure safety/security-relevant PDUs do not bypass protection.
“PN wrap is a theoretical problem.”At high line rates, 32-bit packet-number space can be consumed quickly. Rekey must be tested under load.Use rekey thresholds and consider XPN cipher suites where appropriate.

11.1 Pre-production validation checklist

Provisioning
Each protected port has correct CKN/CAK binding, HSM slot, role, cipher suite, and bypass policy.
Startup
Upper layers see link-up only after the MACsec protected port is operational, unless a conscious fail-open policy exists.
PDU decode
Captured EAPOL-MKA shows expected CKN, SCI, MI/MN progression, MACsec Desired, key-server role, Announcement, Distributed SAK, and SAK Use.
Negative tests
Wrong CAK, wrong CKN, wrong role, cipher mismatch, missing HSM key, blocked EAPOL, and SAK rekey failure are all observable and logged.
Data-plane tests
MACsec frames have expected SecTAG, AN, PN/XPN behavior, SCI inclusion, confidentiality offset, and valid ICV under load.
Rekey tests
Rekey is tested by time threshold and packet-number threshold without packet loss beyond acceptable limits.
