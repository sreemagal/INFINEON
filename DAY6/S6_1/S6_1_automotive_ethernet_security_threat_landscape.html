Security threat landscape on Automotive Ethernet

A standards-oriented, packet-oriented explanation of how attacks become possible in a modern in-vehicle Ethernet network, how they show up in headers and counters, and how to design observability and mitigations without reducing the subject to generic “firewall and encryption” advice.

Threat modelingSOME/IPDoIPgPTP / TSNMACsecSecOCAvailability
1
Compromised endpoint can become a bridgehead into service discovery, diagnostics, and safety-domain traffic if segmentation is weak.
L2
Link-layer identity is not equivalent to ECU trust. MAC address, VLAN, and priority fields are metadata to validate, not proof of legitimacy.
TSN
Timing and availability are security properties. Bad time, bad priority, or bad policing can create safety-relevant failures without changing application payload.
Defense
Layered controls are required: secure boot, port trust, MACsec/MKA, SecOC, IP filtering, IDS, diagnostic authorization, and lifecycle governance.
Core teaching idea: Automotive Ethernet security is not a single cryptographic feature. It is the disciplined control of who may attach, what frames may enter, which services may be discovered, which diagnostic routes may activate, what timing source is trusted, and whether the switch can enforce those decisions at line rate.

1. Standards map: which document answers which security question?

There is no single “Automotive Ethernet security standard” that covers the whole threat landscape. The landscape is a stack of complementary standards and practices.

AreaRelevant standards / practicesWhat they help you answerWhat they do not solve alone
Cybersecurity lifecycleISO/SAE 21434, UN R155, Auto-ISAC, NHTSA guidanceHow to identify assets, threats, risks, cybersecurity goals, validation evidence, production and post-production processes.They do not define the exact wire format of each Ethernet security mechanism.
Link-layer protectionIEEE 802.1AE MAC Security (MACsec), IEEE 802.1X MACsec Key Agreement (MKA), OPEN Alliance Automotive MKAHow to authenticate and integrity-protect Ethernet frames between directly connected MACsec peers. Optionally encrypt user data.Does not replace application authorization. A compromised trusted ECU can still send malicious but cryptographically valid traffic.
PDU authenticityAUTOSAR Secure Onboard Communication (SecOC)How to authenticate and freshness-protect selected critical AUTOSAR I-PDUs with an authenticator and freshness value.Does not protect every byte on the link unless configured for the relevant PDU. Does not hide metadata.
Service communicationAUTOSAR SOME/IP and SOME/IP Service DiscoveryHow service IDs, method IDs, event notifications, service offers, subscriptions, endpoint options, and session identifiers appear on the wire.The protocol format itself is not an authorization policy. Training rule: Treat Ethernet security as a chain of evidence. For each critical message, ask: which ECU identity is trusted, which link is protected, which VLAN and priority are allowed, which service or diagnostic endpoint is allowed, which PDU has authenticity/freshness, and which logs prove it?

2. Automotive Ethernet attack surface stack

The threat landscape becomes clear when you draw the path from attacker position to vehicle effect. The attacker rarely begins inside a brake ECU. More commonly, the path starts from a connected surface and then moves toward in-vehicle services.

External entry
TCU, Wi-Fi, BT, USB, app, OTA, service tool
→
Compromised ECU
infotainment, telematics, camera, gateway app
→
Ethernet access
port, VLAN, multicast, IP route, switch ACL
→
Protocol leverage
SOME/IP-SD, DoIP, gPTP, ARP, DHCP, AVTP
→
Safety or privacy effect
wrong data, missing data, delayed data, leaked data

Remote-to-internal

A telematics or infotainment compromise gives the attacker an IP-speaking node. The first security question is whether that node can discover, route to, or spoof services in the safety network.

Physical / service access

Debug connectors, workshop tools, OBD-connected equipment, replaced ECUs, and test harnesses can introduce a valid electrical Ethernet node. The question becomes attachment trust and diagnostic authorization.

Supply-chain / update path

A vulnerable software component, mis-signed update, leaked key, or weak certificate provisioning can make later network controls ineffective because the ECU is now a legitimate but malicious speaker.

Why Ethernet changes the threat model compared with classic CAN

  • Service discovery exists. Attackers can learn service topology from offers, finds, subscriptions, multicast groups, and port usage.
  • IP routing exists. Once traffic is IP, firewalling, routing tables, NAT-like gateways, TCP state, and UDP multicast behavior all matter.
  • Priorities exist. PCP and queue selection can become an availability attack path if ingress classification is trusted too much.
  • Time is a network service. gPTP is not just a clock convenience. It affects schedules, timestamps, logging, sensor fusion, and deterministic forwarding.
  • Switches are policy enforcement devices. VLAN membership, port state, stream filters, policing, FDB learning, multicast replication, and mirror ports become security controls.

Service discovery is useful because clients can find providers dynamically. The same property creates a reconnaissance and spoofing surface. A fake service offer or fake event subscription can redirect clients, trigger unwanted data flow, or expose internal service topology.

4. Switch-pipeline view of Automotive Ethernet security

A secure automotive switch is not a passive cable concentrator. It is a policy engine that should decide whether a frame is admitted before the frame can consume high-priority resources.

5. Threat families on Automotive Ethernet

Spoofing
MAC, IP, service identity, diagnostic source address, time master identity.
Tampering
Payload modification, false sensor values, malformed SOME/IP, altered diagnostics.
Replay
Old control/status PDUs or old diagnostic responses reused at a later time.
Discovery abuse
Fake OfferService, excessive FindService, unauthorized event subscriptions.
Availability
Floods, multicast storms, high PCP misuse, queue starvation, buffer exhaustion.
Timing abuse
Grandmaster spoof, time jump, wrong domain, correction-field anomalies.

Confidentiality threats

Ethernet carries richer data than traditional control buses: camera streams, audio, diagnostics, vehicle identity, service metadata, and sometimes occupant-related information. Passive observation matters, especially at service ports, debug ports, and unprotected links.

Integrity threats

Integrity threats include payload tampering, false service providers, spoofed diagnostic source addresses, and message replay. SecOC and MACsec address different parts of the integrity problem; neither replaces authorization.

Availability threats

Availability threats are central in TSN networks because latency and jitter are part of correctness. A flood in the wrong priority class can become a safety problem even when every application payload is syntactically valid.

Authenticity and freshness threats

A receiver needs confidence that a message came from the expected sender and belongs to the current moment. This is why freshness counters, packet numbers, diagnostic state, and time synchronization logs are security artifacts.

7. Observability: how failures and attacks become visible

A security architecture that cannot be observed is difficult to validate and almost impossible to operate after deployment. Observability should be designed at every layer.

Best practice: Normalize evidence into a shared incident vocabulary: who spoke, on which port, with which cryptographic state, in which VLAN, to which service or diagnostic target, at what rate, using which priority, and with what verification result.

8. Mitigation strategy: map controls to attack chains

Good security design is not a list of controls. It is a set of controls placed at the exact point where an attack chain can be broken.

Attack chain stepPrimary controlsWhat to validate
Unauthorized node attaches to EthernetMACsec with MKA, port binding, secure device identity, production key provisioning, disabled unused ports.Link does not pass protected traffic until authentication and correct policy state are established.
Compromised node tries lateral movementGateway firewall, VLAN segmentation, service allowlists, routing control, least-privilege service topology.Infotainment/TCU cannot discover or call safety-domain services unless explicitly permitted.
Fake service offer or subscriptionSOME/IP-SD policy, endpoint allowlists, multicast scoping, IDS signatures, service authorization at provider.Client binds only to an expected provider identity and endpoint in the expected VLAN or zone.
Diagnostic misuseDoIP routing activation controls, UDS security access, secure diagnostics certificates, workshop-state checks, backend authorization.Diagnostic sessions and high-impact services are impossible from normal driving networks and unauthorized tools.
Payload replay or tamperSecOC for critical I-PDUs, freshness windows, key management, fault handling strategy, per-PDU monitoring.Old or modified PDUs are rejected and produce an observable, rate-limited diagnostic event.
Timing manipulationTrusted grandmaster policy, gPTP port-role restrictions, time jump detection, domain filtering, independent plausibility checks.Unexpected grandmaster changes and time steps are alarmed and do not silently destabilize TSN schedules.
Priority or flood DoSIngress priority regeneration, Qci stream filters and policers, storm control, multicast filtering, queue occupancy alarms.Invalid high-priority traffic is dropped before consuming protected queues.
Compromised software stackSecure boot, measured boot, code signing, memory safety, OTA rollback protection, SBOM/VEX-based vulnerability management.Network controls assume endpoint compromise is possible and still limit blast radius.
Why MACsec and SecOC are both useful

MACsec protects Ethernet frames on a link and can prevent untrusted physical peers from injecting, observing, or modifying traffic. SecOC protects selected AUTOSAR I-PDUs end-to-end at the PDU level with authenticity and freshness. In a routed or switched network, both are useful because they answer different questions. MACsec asks whether the link peer is allowed to participate. SecOC asks whether this specific PDU is authentic and fresh for the receiver.

Why a firewall alone is insufficient

A firewall can restrict IP addresses, ports, and sometimes application patterns. It usually does not prove freshness of a control PDU, enforce TSN stream rate at line rate, validate gPTP time authority, or protect confidentiality on a raw link. Automotive Ethernet security needs network filtering plus PDU authentication, link security, timing policy, and endpoint hardening.

9. Design review and validation checklists

Network architecture review

  • List all Ethernet ports, VLANs, multicast groups, and IP routes.
  • Mark each port as trusted ECU, service/debug, external, or manufacturing-only.
  • Define which ECUs may be service providers, service consumers, gPTP masters, diagnostic testers, and stream talkers.
  • Verify that priority and VLAN are regenerated at ingress where endpoint trust is not absolute.
  • Confirm that every unused port and unused service path has a deny-by-default state.

Protocol validation

  • For SOME/IP, validate allowed service IDs, method IDs, eventgroups, client IDs, interface versions, and endpoint options.
  • For DoIP, validate routing activation state, source logical address binding, target address authorization, and UDS security access.
  • For SecOC, validate freshness counter persistence across sleep/wake and reset cases.
  • For MACsec, validate peer authorization, MKA startup time, rekey behavior, and bypass rules.
  • For gPTP, validate allowed grandmaster identities and alarm behavior on time jumps.

Switch validation

  • Verify source MAC move alarms and static FDB violations.
  • Test VLAN admission on each port with allowed and disallowed tags.
  • Test multicast containment for service discovery and streaming groups.
  • Test stream filters, max service data unit, and policers for high-priority streams.
  • Check that drops are counted at the stage where they occur, not only at end-to-end application timeout.

Incident triage

  • Start with time: was there a grandmaster change, time jump, or clock-domain anomaly?
  • Check whether the sender identity moved to a different port.
  • Correlate service discovery changes with application symptoms.
  • Separate payload authentication failures from transport drops and queue drops.
  • Preserve key lifecycle and software version evidence for post-production analysis.
