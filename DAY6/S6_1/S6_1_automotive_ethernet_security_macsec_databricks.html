Automotive Ethernet security and MACsec

MACsec is best understood as a link-layer security engine for Ethernet. It is not merely encryption. It provides data-origin authenticity, integrity, replay protection, and optional confidentiality for frames on a protected Layer 2 link.

This note converts the source training material into a protocol-oriented explanation with PDU layouts, header field breakdowns, automotive deployment choices, and realistic examples involving zonal controllers, gateways, cameras, gPTP, and diagnostic bypass traffic.

1Data-communications context first

The correct starting point is not “should we encrypt?” The correct starting point is “what can an attacker do on or near this Ethernet link?”

Passive attack

An attacker taps a camera link or gateway uplink and reads unprotected traffic. Confidentiality mode prevents useful disclosure of the protected payload.

Active attack

An attacker injects a spoofed control or state frame. MACsec validation detects frames that are not protected with the active Secure Association.

Replay attack

An attacker records a valid old protected frame and resends it later. Packet Number based replay checks reject stale frames outside the replay window.

Important framing: automotive Ethernet security is about confidentiality, integrity, authenticity, replay resistance, segmentation, and survivability after partial compromise. Secrecy alone is not enough.

2The MACsec architecture

MACsec separates the long-lived security relationship from the short-lived cryptographic keying context. That separation is the reason it can rekey without tearing down the protected link.

CA

Connectivity Association. The overall security relationship among participants that are allowed to communicate securely.

SC

Secure Channel. A directional secure traffic relationship inside a Connectivity Association.

SA

Secure Association. A concrete cryptographic context using one Secure Association Key.

SecY

MAC Security Entity. The port-side entity that protects transmitted frames and validates received frames.

3Complete MACsec PDU format structure

A MACsec-protected frame remains an Ethernet frame, but the MACsec SecTAG is inserted after the source MAC address and the frame ends with an Integrity Check Value.

VLAN placement note: the standards-pure teaching model places SecTAG immediately after source MAC, so the original user-data region after the SecTAG is protected. Some deployments expose an outer service tag or operate a “VLAN in clear” mode for provider or WAN switching. Automotive designs must state the chosen SecY placement explicitly because it determines whether VLAN/PCP is visible to intermediate forwarding logic.

Integrity-only coverage

Destination MAC, Source MAC, SecTAG, and Secure Data are authenticated. Secure Data stays readable, so protocol analyzers can still decode it, but any modification breaks ICV validation.

Confidentiality coverage

The protected data region is encrypted and authenticated. Destination MAC, Source MAC, and SecTAG remain in clear so the frame can still be delivered and validated by the peer MACsec entity.

4MKA / 802.1X key agreement and PDU structure

MKA is the control plane. It discovers authenticated peers, elects a key server, distributes the SAK, and keeps the MACsec Secure Associations fresh.

CAK

Connectivity Association Key. A long-term secret or EAP-derived key used as the root for MKA operations.

ICK / KEK

Integrity Check Key authenticates MKPDUs. Key Encryption Key protects distributed SAK material.

SAK

Secure Association Key. The actual MACsec data-plane key installed in the SecY for one SA.

Automotive startup interpretation: PSK-based CAK/CKN provisioning is attractive because the link can establish MACsec quickly and deterministically after link-up. SAKs should still be fresh data-plane keys and must not be treated as static long-term keys.

5Cipher suites and protection modes

The most common misunderstanding is to equate MACsec with encryption. MACsec always authenticates. Encryption is an additional mode choice.

ModeTCI behaviorPayload visibilityProtection resultGood automotive use
Integrity-onlyE=0, C=0 in normal teaching modelPayload remains readableTampering and forgery are detected; no secrecyDiagnostics or monitoring links where observability matters but modification must be blocked.
Integrity + confidentiality, offset 0E/C indicate encrypted/changed textProtected data is encryptedSecrecy plus integrity and replay protectionCamera, sensor-fusion, gateway, zonal uplinks carrying sensitive or safety-relevant data.
Confidentiality offsetPartial-cleartext mode in suites that support itInitial protected-data bytes may remain clearSome header inspection remains possible, with remainder encryptedMostly legacy/deployment accommodation; automotive profiles commonly prefer offset 0.

GCM-AES-128

Mandatory baseline cipher-suite family for ordinary MACsec support. It uses AES with a 128-bit key in Galois/Counter Mode to combine confidentiality and authentication efficiently.

GCM-AES-256 and XPN variants

GCM-AES-256 increases key size. XPN variants extend packet numbering to reduce rekey pressure on very high-speed or long-lived links.

6Automotive deployment choices

A vehicle network is not a lab Ethernet cable. Startup, timing, diagnostics, TSN scheduling, VLAN policy, gateway boundaries, and production service workflows all affect the MACsec policy.

Protect by default

Use MACsec on exposed or mission-critical Ethernet links, especially camera-to-switch, zonal-controller uplinks, gateway-to-central-compute links, and high-value sensor paths.

Mode per traffic class

Choose integrity-only when visibility is needed but tamper detection is critical. Choose confidentiality plus integrity when information disclosure is also unacceptable.

Bypass as exception

Keep MKA on the Uncontrolled Port. Treat diagnostics, update, and timing bypasses as explicit exceptions with compensating controls.

7gPTP and MACsec: inline protection versus bypass

The standards model can include MACsec in the time-aware system port model. The real engineering question is whether the implementation can preserve timing accuracy while protecting the frames.

Inline gPTP with MACsec

Works best with two-step timing and bounded MACsec latency variation. Sync and Follow_Up semantics avoid the need to rewrite a protected one-step timestamp inside the protected frame.

gPTP bypass

May be used when one-step assumptions, startup constraints, or hardware limitations make inline protection unacceptable. It must be treated as a real security exception.

Practical rule: bypass is a design trade-off, not a free optimization. It may preserve timing or startup behavior, but it creates an exposed path that must be justified, filtered, monitored, and documented.

8Elucidative automotive scenarios

Scenario A: frame injection on a zonal uplink

Attack

A compromised diagnostic dongle or service device injects a spoofed Ethernet frame toward a zonal controller. The frame has a believable source MAC and VLAN ID, but no valid SecTAG/ICV for the active SA.

MACsec behavior

The receiving SecY cannot map the frame to a valid SA or cannot validate the ICV. The frame is discarded before it reaches the Controlled Port. Counters show invalid, not-valid, or unknown-SCI behavior depending on implementation.

Scenario B: replay of a valid old camera frame

Attack

An attacker records a valid encrypted camera frame and sends it again later to confuse perception or diagnostics.

MACsec behavior

The ICV may still be mathematically valid for the recorded frame, but the PN is stale. Replay protection rejects it if it is lower than the accepted PN window.

9Short recap

Protocol essentials

  • MACsec is Layer 2, hop-by-hop link protection.
  • SecY protects transmit frames and validates received frames.
  • CA defines the secure relationship, SC defines a directional channel, SA defines the active keyed cryptographic context.
  • SecTAG carries TCI/AN, Short Length, PN, and optional SCI.
  • ICV proves authenticity and integrity; confidentiality is optional.

Automotive essentials

  • Use MACsec on critical or exposed Ethernet links.
  • Use PSK-style MKA when deterministic startup matters.
  • Keep MKA on the Uncontrolled Port because it establishes MACsec.
  • Make gPTP bypass a documented exception, not a default assumption.
  • Track counters and logs because validation failures are security diagnostics.
