IEEE 802.1AE MAC Security

SecTAG and GCM-AES-128/256 explained from the wire upward

MACsec protects Ethernet at Layer 2 by inserting a MAC Security Tag, called the SecTAG, after the source MAC address, transforming the original user data into Secure Data, and appending an Integrity Check Value. The cryptographic work is performed by Galois Counter Mode with Advanced Encryption Standard, normally GCM-AES-128 or GCM-AES-256.

SecTAG

Protocol identity and SA selection

Identifies MACsec, tells the receiver which secure association to use, carries the packet number, and may carry the secure channel identifier.

GCM-AES

One pass protection

Encrypts selected user data and authenticates the clear fields that must remain visible on the link.

Replay defense

PN based freshness

The packet number is part of the cryptographic nonce and also drives replay protection.

MKA

Keys are separate

MACsec data frames use SecTAG. Key agreement uses MACsec Key Agreement Protocol Data Units on the uncontrolled port.

Read this first. MACsec is normally link protection between adjacent MACsec entities. A switch that needs to inspect VLAN priority, stream identifiers, IP ports, or time-sensitive networking fields usually needs MACsec termination at that hop, or a design that intentionally leaves the needed bytes visible.

1. Architecture: what MACsec adds to ordinary Ethernet

The MAC Security Entity, usually called SecY, sits between the MAC service client and the insecure LAN. It exposes a controlled port for protected data and an uncontrolled port for key agreement and other control traffic that must work before the secure channel is running.

Controlled port

Used for protected traffic after the secure association is usable. Data is admitted only after validation and replay checks.

Uncontrolled port

Used for MACsec Key Agreement traffic, commonly carried as EAPOL. It must be available before data protection is established.

Secure association

A secure association is the concrete transmit or receive context: Secure Association Key, association number, packet number state, cipher suite and replay parameters.

2. MACsec PDU format on the wire

A protected frame is still an Ethernet frame. The destination and source MAC addresses remain at the front so the link can deliver the frame. Immediately after the source address, the SecTAG identifies this as MACsec and carries the fields needed to validate and decrypt the secure data.

Overhead when SCI is omitted

SecTAG 8 B + ICV 16 B = 24 B before the Ethernet FCS. This is common on point-to-point links when the peer context already identifies the secure channel.

Overhead when SCI is included

SecTAG 16 B + ICV 16 B = 32 B before the Ethernet FCS. This is useful when the receiver needs the transmitting secure channel explicitly encoded in each frame.

SL, PN and SCI

Short Length

SL is set to the Secure Data length only when that length is less than 48 octets. Otherwise SL is zero. It solves a real Ethernet ambiguity: short frames may be padded, so the receiver needs to know where Secure Data ends and where the ICV begins.

Packet Number

PN is a per-SA monotonically increasing number. It is used to construct the GCM initialization vector and to support replay protection. Reusing a PN with the same SAK and SCI breaks the nonce uniqueness rule.

Secure Channel Identifier

SCI is 64 bits. It identifies the secure channel, typically using a MAC address plus port identifier. It may be omitted from the frame on simple point-to-point links, but it is still part of the cryptographic computation for the non-XPN suites.

Trainer’s shortcut. SecTAG is not “the encryption header.” It is the data-plane steering and freshness header. Encryption and authentication are performed by the cipher suite using the SecTAG and visible MAC addresses as inputs.

4. GCM-AES-128 and GCM-AES-256 in MACsec

Galois Counter Mode with Advanced Encryption Standard is an authenticated encryption mode. MACsec uses it in two important ways. It can provide integrity without confidentiality, or it can provide confidentiality and integrity together.

GCM-AES-128

128-bit SAK This is the default MACsec cipher suite. The Secure Association Key is 128 bits. The IV is 96 bits in non-XPN operation: SCI | PN.

K = 128-bit SAK IV = 64-bit SCI || 32-bit PN T = 128-bit ICV

GCM-AES-256

256-bit SAK This uses the same MACsec framing idea and the same 96-bit non-XPN IV structure, but the AES key is 256 bits.

K = 256-bit SAK IV = 64-bit SCI || 32-bit PN T = 128-bit ICV

Integrity-only mode

Integrity-only mode leaves the user data readable but binds it cryptographically to the destination address, source address and SecTAG. This is useful when intermediate devices or diagnostic tools must inspect fields, but unauthorized modification must be detected.

Additional authenticated data A = DA || SA || SecTAG || UserData Plaintext P = null SecureData = UserData ICV = GCM tag T

Confidentiality mode

Confidentiality mode encrypts the user data while still authenticating the clear link fields. A receiver that does not have the right SAK cannot recover the payload and cannot forge a valid ICV.

Additional authenticated data A = DA || SA || SecTAG Plaintext P = UserData Ciphertext C = AES-GCM encryption of P SecureData = C ICV = GCM tag T
Do not confuse AES key size with tag size. GCM-AES-128 and GCM-AES-256 differ in the SAK size. In normal MACsec use, the ICV/tag for these suites is still 128 bits.

Why GCM fits MACsec so well

Visible fields are authenticated

DA, SA and SecTAG must stay visible for Ethernet delivery and parsing. GCM authenticates them as additional authenticated data.

Payload can be encrypted

The user data can be encrypted without changing its length, so the MAC service sees a predictable overhead.

Hardware friendly

AES counter generation and GHASH authentication are well suited to high-throughput hardware pipelines in switches, PHYs and Ethernet controllers.

5. Transmit and receive processing

1. Classify to SCSecY assigns the outgoing frame to a transmit Secure Channel.
2. Select SAAssociation Number selects the current Secure Association and SAK.
3. Allocate PNNext packet number is inserted into SecTAG and used in IV.
4. Run GCMDA, SA, SecTAG and User Data become AAD/plaintext inputs.
5. Emit MPDUSecTAG, Secure Data and ICV are transmitted after DA and SA.
6. Peer validatesReceiver identifies SA, recomputes GCM tag, checks replay window and delivers data.

Successful receive path

  1. Parse EtherType 0x88E5.
  2. Read TCI, AN, SL, PN and optional SCI.
  3. Use SCI and AN to find the receive SA and SAK.
  4. Validate ICV with DA, SA, SecTAG and Secure Data.
  5. Decrypt if confidentiality was used.
  6. Apply replay protection using PN and the receive window.
  7. Deliver original user data to the controlled port.

Failure path

  1. Unknown SCI or no receive SA means the frame cannot be validated.
  2. Wrong SAK, modified DA/SA/SecTAG, modified ciphertext or modified ICV causes validation failure.
  3. Old PN outside the replay window is treated as a replay.
  4. Bad SL can make the receiver locate the ICV incorrectly, causing validation failure.

7. Interesting scenarios that expose the real behaviour

Scenario 1: attacker changes VLAN priority in a protected frame

A lab capture shows a MACsec frame where the VLAN Priority Code Point appears to have changed between two bridge ports. Does MACsec catch it?

1
Case A: VLAN tag is inside Secure Data.

If confidentiality is enabled, the VLAN tag is encrypted. The attacker cannot meaningfully change the PCP because the bytes are ciphertext. Any random bit change breaks the ICV.

2
Case B: VLAN tag is visible because of design choices.

If the VLAN tag is in authenticated clear text, changing the PCP changes authenticated data. The receiver recomputes the GCM tag and detects the modification.

3
Conclusion.

MACsec does not need to hide a field to protect its integrity. Clear fields can be authenticated as additional authenticated data.

Scenario 2: replaying a valid captured MACsec frame

An attacker records a valid encrypted command and retransmits it later.

1
The ICV is valid.

The frame was not modified, so the GCM validation can succeed cryptographically.

2
The PN is old.

Replay protection checks the received PN against the lowest acceptable PN for that SA. If it is outside the window, the frame is discarded.

3
Why the order matters.

The system can distinguish “authentic but stale” from “modified or unauthentic.” Both are rejected, but the operational diagnosis differs.

Scenario 3: key rollover without interrupting traffic

A sender is changing from an old SAK to a new SAK while traffic is flowing.

1
Old SA is still installed.

Frames with AN 0 are still accepted for a short transition interval.

2
New SA becomes active.

The sender starts transmitting with AN 1 and a fresh PN space under the new SAK.

3
Receiver uses AN to pick the right key.

The 2-bit AN lets the receiver validate both old and new frames without guessing the key.

4
Old SA is retired.

After the transition, old AN traffic is rejected. This prevents indefinite acceptance of stale keys.

Scenario 4: packet number exhaustion at high frame rates

With non-XPN GCM-AES-128/256, only 32 PN bits are transmitted. Reusing the same IV with the same key is not allowed.

What goes wrong

If PN wraps back to a value already used with the same SAK and SCI, the GCM nonce is reused. That is a cryptographic failure, not just a counter rollover issue.

Correct design response

Rekey well before PN exhaustion, or use the extended packet number suites where appropriate. Automotive designs should budget this from worst-case minimum-frame-rate traffic, not average traffic.

At minimum Ethernet frame spacing, 2^32 packets can be consumed quickly on high-speed links. Design rule: rekey before PN approaches exhaustion, not after a warning appears.

Scenario 5: a switch cannot classify a stream after encryption

A team enables confidentiality on a link and then discovers that a downstream bridge no longer sees the fields needed for stream filtering or scheduling.

1
Before MACsec.

The bridge classified using VLAN ID, PCP, EtherType, IP addresses, UDP ports or application headers.

2
After MACsec confidentiality.

Fields inside Secure Data are ciphertext. A bridge that does not terminate MACsec cannot classify them.

3
Design choices.

Use hop-by-hop MACsec termination at switches, classify before encryption, leave only carefully selected bytes visible where supported, or use outer visible metadata that does not leak unacceptable information.

Scenario 6: SCI omitted where the receiver needs it

A point-to-point lab test works with SCI omitted. Later the same configuration is reused in a multi-peer or ambiguous receive context.

Failure symptom

Frames parse as MACsec but cannot be assigned to the correct receive Secure Channel. Counters show unknown SCI, no SA, or not-valid packets depending on implementation.

Fix

Include the explicit SCI when the receiver cannot infer the transmitting Secure Channel unambiguously. Do not treat SCI omission as a universal overhead optimization.

9. Automotive Ethernet design notes

MACsec with TSN

Time-Sensitive Networking features such as stream filtering, traffic shaping, policing and scheduling often need visible classification fields. If MACsec encrypts those fields, the bridge must terminate MACsec before applying those functions, or the design must provide visible classification metadata.

MACsec with diagnostics

Diagnostic tools may see the outer DA, SA, SecTAG, PN, AN and ICV. They may not see the user data if confidentiality is enabled. This changes what can be validated from passive captures.

AUTOSAR angle

AUTOSAR MACsec Key Agreement configuration and APIs sit above Ethernet driver, switch driver and transceiver integration. For training, separate the cryptographic data-plane SecY from the MKA control-plane module that manages the secure channel.

Security architecture choice

Use MACsec when the threat is on the local Ethernet segment or link. It complements, rather than automatically replaces, end-to-end application security such as TLS, SOME/IP security, or higher-layer authentication.

10. Standards anchors and source map

This page intentionally avoids treating MACsec as a vendor feature. The wire format and core cryptographic interpretation are anchored in IEEE 802.1AE. The GCM construction is anchored in NIST SP 800-38D. AUTOSAR adds automotive implementation and configuration requirements for MACsec and MKA.

  • IEEE 802.1AE-2018: MACsec MPDU components, SecTAG format, TCI/AN encoding, SL/PN/SCI rules, ICV and cipher suites.
  • NIST SP 800-38D: Galois Counter Mode as authenticated encryption with associated data.
  • AUTOSAR R24-11 Requirements on MACsec: automotive requirement framing, acronym set, support for GCM-AES-128 and GCM-AES-256.
  • AUTOSAR R24-11 Specification of MACsec Key Agreement: MKA module responsibilities, secure channel establishment and cipher suite configuration.
