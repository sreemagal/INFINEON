Integrity-only vs Encryption Modes
and gPTP / MACsec Bypass

A practical automotive Ethernet guide to MACsec protection choices, SecTAG and ICV placement, gPTP frame behavior, MKA bootstrapping, bypass policy, and failure modes. The emphasis is on what actually happens to the Ethernet frame and what the switch or Ethernet interface must decide.

Core idea

MACsec is a Layer 2 security shim. It can authenticate frames without encrypting the payload, or it can authenticate and encrypt the payload. Either way, the Ethernet destination and source MAC addresses remain visible so the link can still be forwarded. The engineering issue is not only cryptography. It is also pipeline placement, timestamping, bypass policy, replay protection, and what must remain visible to the bridge.

1. Mental model: security is not one switch

In MACsec, “secure” can mean different things. A received frame may be authenticated but still readable, encrypted and authenticated, replay-protected or not, strictly discarded on failure or merely counted and passed for permissive operation. Treat these as separate engineering decisions.

Integrity-only

Readable but tamper-evident

The payload remains in clear text, but the receiver validates the Integrity Check Value (ICV). If a bit in the protected part of the frame is changed, validation fails. This helps detect modification and spoofing, but not passive listening.

Encryption mode

Confidential and authenticated

The payload, or the payload after a configured confidentiality offset, is encrypted. The receiver still validates the ICV. This protects against both passive observation and active modification, but requires deeper hardware participation.

Replay protection

Freshness, not secrecy

The Packet Number (PN) in the SecTAG advances with transmitted frames. The receiver can reject old packet numbers, or packet numbers outside a replay window, even if the cryptographic tag is otherwise valid.

Training shortcut: Integrity-only says “you may look, but you must not alter.” Encryption says “you may neither look nor alter.” Replay protection says “you may not reuse an old valid frame.”

2. Standards map

The key mistake in design reviews is to discuss “MACsec” as if it includes time synchronization, key agreement, VLAN forwarding, and AUTOSAR integration. It does not. Several standards and software specifications meet in the Ethernet switch pipeline.

AUTOSAR MACsec knobs used in this guide

  • MkaMacSecCapability: integrity with confidentiality, or integrity without confidentiality.
  • MkaMacSecConfidentialityOffset: confidentiality offset 0, 30, or 50 for supported non-XPN choices.
  • ValidateFrames: disabled, checked, or strict.
  • ReplayProtect and ReplayWindow: decide how old packet numbers are treated.
  • BypassedVlanPtrs and BypassedEtherTypesPtr: configure traffic that does not enter normal SecY protection.

AUTOSAR time-sync details used in this guide

  • gPTP-derived Ethernet time synchronization uses IEEE 802.1AS message format unless AUTOSAR extensions are configured.
  • AUTOSAR allows VLAN-tagged time sync if the switch hardware can forward reserved multicast addresses in the required range.
  • CRC-secured AUTOSAR time Sub-TLVs are not cybersecurity protection.
  • AUTOSAR notes that Pdelay messages do not contain the TLVs used for CRC or ICV protection.

4. Integrity-only vs encryption mode

Both modes authenticate the frame. The difference is whether the secure data is readable on the wire. In AUTOSAR terms, this aligns with configuring MACsec capability as integrity without confidentiality or integrity and confidentiality.

What the confidentiality offset is really doing

A confidentiality offset leaves the first configured number of octets of user data unencrypted while still integrity-protecting them. This is a compromise for deployments that need limited header visibility for classification, instrumentation, or legacy parsing but still want the rest of the payload encrypted.

Why integrity-only is not “weak encryption”

Integrity-only is not a partial cipher. It is a deliberate authentication mode. It is appropriate when the payload is not secret, but undetected modification, spoofing, or replay would be unacceptable.

6. Bypass policy: what should skip MACsec and why?

Bypass means a frame is intentionally not processed as normal MACsec secure data. It can be necessary for bootstrapping or timestamping, but it must be precise. A good bypass policy is narrow, documented, observable, and tested with negative cases.

Mandatory style bypass

MKA / EAPOL

MKA must run before the controlled secure data path is established. Therefore EAPOL/MKA frames are treated as control traffic through the uncontrolled side of the port. Without this exception, keys could not be negotiated.

match EtherType == 0x888E
action uncontrolled_port / MKA
Common automotive bypass

gPTP / EthTSyn

gPTP often bypasses MACsec so event timestamping, peer delay measurement, and time-aware bridge updates remain deterministic and inspectable. The bypass should match only the PTP EtherType and expected destination address/VLAN.

match EtherType == 0x88F7
and DA == 01:80:C2:00:00:0E
action EthTSyn timestamp path
Dangerous bypass

Broad VLAN bypass

Bypassing an entire diagnostic or service VLAN may accidentally expose DoIP, SOME/IP service calls, calibration data, or firmware chunks. Prefer EtherType plus reserved multicast plus port role checks over broad VLAN-only bypass.

bad idea:
match VLAN == 20
action bypass_all

Bypass decision tree

1. Is this EAPOL/MKA?
If EtherType is 0x888E and the port is configured for MKA, send to the uncontrolled MKA path. Do not expect a valid SAK before this path works.
2. Is this gPTP/EthTSyn?
If EtherType is 0x88F7, destination address and VLAN match the configured timing profile, steer to timestamp-aware time-sync path. Apply protocol-level validation if configured.
3. Is this explicitly bypassed by VLAN or EtherType?
Use the configured AUTOSAR bypass list only for narrow exceptions. Count and expose every bypassed class to diagnostics.
4. Otherwise, process through SecY.
Transmit side inserts SecTAG and ICV. Receive side validates, checks replay policy, decrypts if needed, strips SecTAG/ICV, and passes accepted frames upward.
5. If validation fails, apply ValidateFrames policy.
Strict mode discards invalid frames. Checked/permissive modes may pass but count invalid frames, useful during bring-up but risky in production.

7. Pipeline view inside an automotive Ethernet switch

Most confusion disappears when the decision is drawn as a pipeline. MACsec is not a magical cloud around Ethernet. It is a processing function in a specific place relative to parser, classifier, timestamping, forwarding, policing, shaping, and scheduling.

1. Ingress PHY/MACReceives bits, validates FCS, captures hardware timestamp for event frames if configured.
2. ParserDA, SA, optional VLAN, EtherType. Detects 0x88E5, 0x88F7, 0x888E.
3. Bypass classifierMKA path, gPTP timestamp path, configured VLAN/EtherType exceptions.
4. SecY receiveFor MACsec frames: identify SC/SA, validate ICV, replay check, decrypt if configured.
5. Bridge logicFDB lookup, VLAN membership, PCP/traffic class, policing, stream filters.
6. Egress SecYFor protected data: choose SA, increment PN, insert SecTAG, encrypt or integrity-only tag.
7. Egress MAC/PHYTAS/CBS/ATS scheduling, final FCS, timestamp for event frames.

Correct mental model for hop-by-hop MACsec

MACsec protects one link or one directly secured association at Layer 2. A MACsec-aware bridge commonly decrypts on ingress, forwards internally using clear metadata and policy, then re-encrypts on egress with a different secure association. This is different from end-to-end TLS or IPsec.

Why timestamp placement matters

If timestamping occurs before SecY on one side and after SecY on another, residence-time and peer-delay measurements can include inconsistent cryptographic processing latency. A deterministic design fixes the timestamp reference point and tests it.

10. Configuration and validation checklist

Production checklist

  • Define whether each traffic class requires confidentiality, integrity-only, or bypass.
  • Use strict validation for production data paths unless a safety case explicitly justifies another mode.
  • Enable replay protection for data classes where repeated old frames are hazardous.
  • Keep bypass rules narrow: EtherType + destination MAC + VLAN + ingress port role where possible.
  • Log counters for bypassed, untagged, invalid, delayed, late, unknown-SCI, and replay-discarded frames.
  • Verify timestamp reference points with MACsec enabled and disabled.
  • Test cipher suite mismatch, wrong CAK/CKN, missing key, PN rollover, and link flap recovery.

Anti-patterns

  • “Bypass the timing VLAN.” Too broad if diagnostics or service discovery share that VLAN.
  • “Encrypt everything including MKA.” Impossible as a bootstrapping model.
  • “gPTP bypass means timing is secure.” Bypass is an availability and timestamping choice, not a security mechanism.
  • “Integrity-only protects privacy.” It does not. The payload is still visible.
  • “MACsec is end-to-end.” In switched networks, it is usually hop-by-hop unless the topology is specifically designed otherwise.
  • “CRC-secured time is cyber-secure.” AUTOSAR explicitly distinguishes CRC integrity checking from cybersecurity authentication.
Best practical policy: Use MACsec encryption for sensitive service/data payloads, integrity-only where visibility is required and secrecy is not, and bypass only for bootstrapping and timing/control protocols that genuinely require it. Every bypass rule should have a named owner and a test case.

11. Knowledge checks

Question 1

A frame has a valid SecTAG and ICV. The secure data is readable in Wireshark. Is MACsec necessarily disabled?

No. It may be MACsec integrity-only mode. The presence of SecTAG and valid ICV means MACsec authentication may still be active.

Question 2

Why should MKA not be placed behind the controlled MACsec port?

Because MKA is used to establish the keys and secure associations needed by that controlled path. It must be able to run before protected data transfer is available.

Question 3

Why can gPTP bypass be useful even though it weakens link-layer security for timing frames?

Because gPTP requires precise timestamping and may require bridge processing of timing fields. Bypass keeps timing processing deterministic, but must be complemented by validation or time-authentication strategy.

Question 4

A receiver is configured with strict validation. What happens to a MACsec frame with an invalid ICV?

It is discarded. Counters and security events should record the validation failure for diagnostics and intrusion detection.
