Integrity-only vs encryption modes

A standards-oriented explanation of how MACsec uses the same SecTAG, packet number, secure association key, and integrity check value for two very different operational goals: visible but tamper-evident traffic and confidential authenticated traffic.

1. Mental model

Both modes are MACsec protection. The difference is whether the user data remains readable on the wire.

Assumption used in this artifact: “integrity-only vs encryption modes” is explained in the MACsec sense, because the distinction is explicit in IEEE 802.1AE SecY operation and in AUTOSAR MACsec/MKA configuration. The same ideas also help when discussing link-layer security offload in automotive Ethernet switches and PHYs.

Integrity-only mode

The transmitted Secure Data is the original User Data. A sniffer can read the Ethernet payload after the SecTAG, but a receiver can detect modification because the Integrity Check Value (ICV) no longer verifies.

Original user data
→
Plain Secure Data
+
ICV

Use when: observability, troubleshooting, or early deployment needs cleartext, while tamper detection, origin validation, and replay protection are still required.

vs

Encryption / confidentiality mode

The transmitted Secure Data contains ciphertext, except when a configured confidentiality offset deliberately leaves an initial prefix in clear. Modification and replay are still detected.

Original user data
→
Ciphertext Secure Data
+
ICV

Use when: payload privacy matters, for example camera/radar data, diagnostics containing sensitive identifiers, software update metadata, or vehicle state signals.

The key insight

Integrity is always the first gate

Even when confidentiality is enabled, the receiver must validate authenticity and integrity before releasing plaintext to the controlled port.

Encryption is not “MACsec plus something else”

Encryption is a MACsec cipher-suite usage mode. The SecTAG and ICV are still there; the user data is simply transformed into ciphertext.

Visible headers remain visible

Destination MAC, source MAC, SecTAG, packet number, and ICV remain observable. MACsec is not traffic-flow privacy.

2. PDU and header format

3. TCI bits and mode signalling

The TCI/AN octet is the visible hint that tells a receiver whether confidentiality is in use and whether the Secure Data is unchanged.

TCI/AN bit layout used for training

Bits are shown from bit 8 to bit 1 of the TCI/AN octet. The two least significant bits form the Association Number (AN). The common corrected interpretation is bit 4 as E and bit 3 as C.

Common mode encodings

Integrity-only using standard GCM-AES behavior

E=0 C=0

Secure Data is exactly the User Data. A 16-byte ICV authenticates the frame.

Encryption without offset

E=1 C=1

All user data supplied to the cipher suite is transformed into ciphertext.

Encryption with offset

E=1 C=1 offset=30/50

An initial prefix is clear but authenticated; the remainder is encrypted. Standard XPN cipher suites do not support offsets.

4. Cryptographic processing

The same AES-GCM family is used, but the split between authenticated data and plaintext changes.

GCM-AES inputs in MACsec terms

K

Secure Association Key (SAK), 128 or 256 bits depending on cipher suite.

IV

Constructed from SCI and PN for non-XPN, or SSCI and extended PN plus salt for XPN.

A

Additional authenticated data. It is checked by the ICV but not encrypted.

P → C

Plaintext P is encrypted into ciphertext C. If P is null, no payload encryption occurs.

Validation policyBehaviorWhy it matters
disabledValidation is not enforced and SecTAG/ICV can be stripped if present.Useful only for special transition/testing paths. Not a secure receive policy.
checkedValidation is performed, but invalid frames are not necessarily discarded.Can be useful for diagnostics in integrity-only mode because original data may still be recoverable.

Replay protection is independent of confidentiality

A replayed frame can be readable in integrity-only mode and unreadable in encryption mode, but both are stale. The Packet Number (PN) and replay window stop old valid frames from being accepted again.

PN increments per SA

Each protected frame uses a fresh non-zero PN. Reuse of the same PN under the same key is forbidden for GCM security.

Window tolerates reordering

A finite replay window can allow small reordering while still rejecting very old frames.

XPN extends lifetime

Extended Packet Number cipher suites use a 64-bit PN to avoid rapid SA exhaustion on high-rate links.

Important: encryption never makes replay acceptable. A replayed ciphertext may still decrypt perfectly if the key is valid, so replay protection must be evaluated separately from confidentiality.

6. AUTOSAR configuration view

In an automotive ECU or switch environment, these modes appear as SecY configuration and MKA-managed secure channel setup.

Bypassed VLAN/EtherTypeConfigured traffic classes that pass without MACsec protection.Bypass is different from integrity-only. Bypassed traffic has no SecTAG and no ICV.

MKA/SecY division of responsibility

MACsec Key Agreement Entity (KaY / MKA)

Discovers and authenticates peers, selects/announces cipher-suite capability, distributes or installs SAKs, handles key rollover, and updates the SecY with operational parameters.

MAC Security Entity (SecY)

Actually protects transmitted frames, validates received frames, applies PN/replay checks, encrypts/decrypts as configured, and exposes counters for protected, encrypted, invalid, late, and not-valid packets.

Training distinction: “bypassed” traffic is not integrity-only traffic. Integrity-only traffic still carries SecTAG, PN, and ICV. Bypassed traffic is deliberately outside MACsec protection and must be justified in the security concept.

8. VLAN, TSN, and observability issues

Integrity-only and encryption modes influence what a switch can classify, shape, mirror, or diagnose.

Case 1: TSN classification after encryption boundary

If the classifier needs inner VLAN Priority Code Point (PCP), stream ID, Internet Protocol (IP) header, User Datagram Protocol (UDP) port, or SOME/IP field, full confidentiality can hide those fields. Classification must happen before the SecY, use an outer clear tag, or use a carefully justified confidentiality offset.

Case 2: SPAN/RSPAN mirroring for diagnostics

Integrity-only gives excellent observability: mirrored traffic can be decoded while still being tamper-evident at the receiver. Encryption gives stronger privacy but packet captures show only metadata unless the analyzer is trusted with keys.

Case 3: Bypass rules and startup traffic

MKA, EAPOL, and selected management traffic may need to pass before the protected controlled port is enabled. Treat this as a narrow exception. Do not call it integrity-only.

Case 4: Hop-by-hop protection through switches

MACsec commonly protects each link. A MACsec-capable switch may validate/decrypt on ingress and protect/encrypt again on egress using a different secure association key. That means the switch becomes part of the trusted boundary.

9. Mistakes and engineering checks

Use this section as a validation checklist during training or design reviews.

MistakeCorrectionCheck in packet capture or counters
Calling bypassed VLAN traffic “integrity-only”Integrity-only still has SecTAG, PN, and ICV. Bypassed traffic has no MACsec protection.Look for EtherType 0x88E5 and ICV growth.
Assuming encryption hides all metadataDA, SA, SecTAG, PN, ICV, frame size, timing, and maybe outer tags remain visible.Check what remains before SecTAG and whether outer tags are present.
Ignoring replay in integrity-only modeReadable stale data can be just as dangerous as readable modified data.Check InPkts_Late / delayed counters and replay window configuration.
Using confidentiality offset without a data-minimization reviewClear prefix bytes are authenticated but not secret.Decode exactly which fields remain clear at offset 30 or 50.
Assuming XPN supports confidentiality offsetsStandard XPN cipher suites do not support offset confidentiality.Verify selected cipher suite and offset combination before deployment.
Forgetting key rollover under high frame ratesPN values must not repeat under the same SAK. XPN or timely rollover is needed.Monitor PN headroom, AN changes, and MKA rekey events.
Allowing upper-layer services before MACsec is readyProtected traffic should wait until MKA/SecY setup is complete and controlled port is enabled.Confirm EthSM/TcpIp startup waits for MACsec protected link readiness in the integration stack.

Minimal packet-capture checklist

Header

Do you see 0x88E5? Is the SecTAG 8 or 16 bytes? Is SCI present?

Mode bits

Are E and C consistent with the intended mode?

Data region

Can you decode the Secure Data? If yes, is that intended integrity-only or an unintended bypass?

PN

Does PN increase monotonically for the SA? Is replay protection enabled?

ICV

Is the 16-byte GCM tag present at the end of protected data?

Counters

Are invalid, late, not-valid, no-SA, or bad-tag counters incrementing unexpectedly?
