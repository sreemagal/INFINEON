Training material on IEEE 802.1CB FRER implementation with mechanisms, protocol headers, and field meanings

This page replaces a sketch-level explanation with implementation-oriented material. It explains how Frame Replication and Elimination for Reliability (FRER) identifies a stream, generates sequence numbers, inserts the Redundancy Tag (R-TAG), replicates frames over member paths, eliminates duplicates, and exposes the mechanism for automotive Ethernet validation.

1. Start with the real reliability problem

A normal switched Ethernet flow usually follows one active path. If a link, bridge, PHY, or egress queue on that path fails at the wrong instant, frames are lost until a separate recovery mechanism detects the fault and changes forwarding.

IEEE 802.1CB Frame Replication and Elimination for Reliability (FRER) avoids the detection-and-switchover delay by sending equivalent copies of the same frame over multiple member paths and then eliminating duplicates at a merge point or listener.

Core distinction: FRER does not compute the redundant paths. FRER assumes suitable paths exist and defines how a protected stream is identified, sequenced, replicated, decoded, recovered, and counted.
Classic recovery model Talker ───── active path ───── Listener X failure Loss occurs until failure detection + switchover completes. FRER model path A ───────────────► Talker ── Replication Elimination ──► one clean stream path B ───────────────► Both paths are used immediately. The first valid copy can be accepted.

That is why FRER is described as seamless redundancy. It is not waiting for the network to decide that a backup path must be activated. The redundant copies are already in flight.

2. The FRER processing pipeline

FRER is not one magic block. It is a chain of cooperating functions. A realistic bridge or end station implementation has to keep the following processing order straight.

Transmit-side direction Incoming frame │ ▼ [Stream identification] │ produces internal stream_handle ▼ [Sequence generation] │ assigns sequence_number for this stream ▼ [Sequence encode] │ inserts R-TAG or another sequence encoding ▼ [Stream splitting / replication] │ creates member-stream copies ├────────► member path A └────────► member path B Receive-side direction member path A copy ─┐ ├──► [Sequence decode] ─► [Sequence recovery / elimination] ─► one accepted frame member path B copy ─┘ extracts seq discards duplicates/rogues

What belongs to FRER?

Stream identification, sequence generation, sequence encode/decode, replication, individual recovery, compound recovery, duplicate elimination, and management counters.

What is outside FRER?

Physical topology design, explicit path computation, bridge forwarding path setup, resource reservation, and VLAN engineering. These may be handled by IEEE 802.1Q mechanisms, IEEE 802.1Qca path control, or a centralized controller.

3. Ethernet frame format and the R-TAG

The most important packet-level header in IEEE 802.1CB is the Redundancy Tag (R-TAG). It is an example sequence encode/decode format used to carry the sequence number needed by downstream recovery functions.

3.1 Basic Ethernet frame before FRER sequence encoding

Without FRER sequence encoding +------------------+------------------+------------------+--------------------+ | Destination MAC | Source MAC | EtherType/Length | Payload | | 6 octets | 6 octets | 2 octets | ... | +------------------+------------------+------------------+--------------------+ With VLAN tagging, one or more VLAN tags may appear between Source MAC and the carried EtherType.

3.2 Conceptual frame after R-TAG insertion

Typical conceptual view with R-TAG +------------------+------------------+----------------------+------------------+--------------------+ | Destination MAC | Source MAC | R-TAG | EtherType/Length | Payload | | 6 octets | 6 octets | 6 octets | 2 octets | ... | +------------------+------------------+----------------------+------------------+--------------------+ R-TAG = TPID/EtherType 0xF1C1 + Reserved + Sequence Number Important: the standard allows the position relative to other tags to depend on where the sequence encode/decode function is placed in the stack. Do not hard-code the assumption that every deployment has exactly the same tag order.

3.4 R-TAG decode behavior

if first_two_octets == 0xF1C1 and MSDU_length >= 6:
    remove first 6 octets from the MAC service data unit
    sequence_number = R_TAG.sequence_number
    ignore R_TAG.reserved
else:
    sequence_number = InvalidSequenceValue
    increment errored/tagless counters according to configuration
Implementation warning: R-TAG is a sequence encoding. It is not a routing header. It does not tell a bridge which path to take. It only carries information needed by the FRER recovery logic.

4. Stream identification: the step that must happen before sequencing

A sequence number is meaningful only inside a stream context. Sequence number 42 from camera stream A and sequence number 42 from radar stream B are unrelated. Therefore, FRER first needs to derive an internal stream_handle.

Frame fields on ingress Dst MAC + Src MAC + VLAN + PCP + EtherType + IP/UDP fields + ingress port │ ▼ Configured stream identification function │ ▼ stream_handle = "FrontCameraCriticalStream" │ ▼ Sequence generation / replication / recovery instance selected by stream_handle

The stream handle is not normally an Ethernet header field. It is an internal service parameter used to select the correct FRER sequence generation, replication, and recovery state.

5. Sequence generation: numbering the stream before replication

The sequence generation function assigns a monotonically increasing modulo-65,536 value to frames of a particular stream. The same original frame and all of its FRER replicas must carry the same sequence number, otherwise duplicate elimination cannot work.

Key state

GenSeqSpace = 65,536

GenSeqNum is the next sequence number to assign for this stream.

One instance per context

Each sequence generation instance is tied to a stream handle, port, and direction. Different streams do not share the same sequence state.

5.1 Simplified generation algorithm

on reset:
    GenSeqNum = 0

on DATA_REQUEST(frame, stream_handle):
    frame.sequence_number = GenSeqNum
    GenSeqNum = (GenSeqNum + 1) mod 65536
    send frame to sequence encode function

5.2 Wrap-around example

Original frame numberAssigned sequence numberNext GenSeqNumMeaning
Frame A6553465535Near the end of the 16-bit sequence space.
Frame B655350Maximum 16-bit value used.
Frame C01Modulo wrap has occurred.
Frame D12Receiver must compare sequence values modulo 65,536.
Why not just use a timestamp? Because duplicate elimination requires exact identity of replicas. Timing alone cannot reliably distinguish a delayed duplicate from a new frame under variable path delay.

6. Sequence encode/decode: putting the sequence number into the frame

Sequence generation creates an internal sequence_number. Sequence encode/decode is the mechanism that carries that number across the network. The R-TAG is the common IEEE 802.1CB sequence encoding example.

6.1 Transmit-side encode

input: frame with internal sequence_number = 0x1234

R_TAG.EtherType       = 0xF1C1
R_TAG.Reserved        = 0x0000
R_TAG.SequenceNumber  = 0x1234

insert R-TAG in configured protocol-stack position
send to replication / forwarding function

6.2 Receive-side decode

input: received MAC service data unit

if R-TAG is present at the configured sequence-decode position:
    sequence_number = R_TAG.SequenceNumber
    remove R-TAG before passing packet upward, if configured to do so
else:
    sequence_number = InvalidSequenceValue
    apply frerSeqRcvyTakeNoSequence policy
Important: Some systems may use other sequence encode/decode formats for interworking, such as High-availability Seamless Redundancy (HSR) sequence tags or Parallel Redundancy Protocol (PRP) trailers. For an IEEE 802.1CB R-TAG discussion, focus on the 6-octet R-TAG described above.

7. Replication: creating member streams without losing identity

Replication takes one sequenced compound stream and creates multiple member-stream copies. Each copy must retain the same sequence identity so that recovery functions can later recognize duplicates.

Original stream frame stream_handle = X sequence_number = 101 payload = P ┌──────────────────────────────► Member stream X-A, sequence 101, payload P Replication ────┤ └──────────────────────────────► Member stream X-B, sequence 101, payload P The copies are not new application frames. They are redundant representations of the same original frame.

7.1 Where replication can happen

At the talker

The source endpoint is FRER-aware and inserts sequence information before sending copies into two member paths.

At the first bridge

A bridge acts as a proxy for a non-FRER-aware talker. It identifies the stream, adds sequence information, and splits the stream.

At an intermediate bridge

The network may split a stream at a bridge where two configured member paths become available.

7.2 Replication is not path computation

Replication chooses configured member outputs for a stream. The existence of the disjoint or partially disjoint paths is a topology and path-control problem, not a consequence of the R-TAG itself.

Clearer interpretation: a configured replication table is a per-stream instruction table inside the talker, bridge, or FRER proxy. It says: when a frame belongs to this protected compound stream, make these redundant member-stream copies and send each copy into its configured member path.

7.3 Clear configured replication table example

Assume a bridge has already identified an incoming frame as protected stream X, assigned or preserved FRER sequence number 101, and inserted an R-TAG carrying that sequence number. The frame is still one original application frame at this point.

Incoming protected compound-stream frame stream_handle = X sequence_number = 101 payload = P │ ▼ Configured replication table for stream X │ ├──► Member stream X-A │ egress port = 5 │ VLAN ID = 42 │ destination MAC = 91:E0:F0:00:01:01 │ R-TAG sequence = 101 │ payload = P │ └──► Member stream X-B egress port = 6 VLAN ID = 43 destination MAC = 91:E0:F0:00:01:02 R-TAG sequence = 101 payload = P The forwarding information may differ. The FRER sequence identity must remain the same.

The two transmitted Ethernet frames may therefore differ in their forwarding envelope, because they are being forced into different engineered paths. But they must still represent the same original frame to FRER recovery.

    stream_handle: X
    sequence encoding: R-TAG

        egress port: 6
        VLAN ID: 43
        destination MAC: 91:E0:F0:00:01:02

Packet walk-through

Step 1: stream identification. The bridge classifies the incoming packet and maps it to internal stream_handle X. This stream handle is the key used to look up the replication rule.
Step 2: sequence identity. The FRER sequence generation or sequence encode function gives the original frame sequence number 101. This number identifies this original frame within stream X.
Step 3: replication. The bridge creates two member-stream copies. Copy X-A goes to port 5 and copy X-B goes to port 6.
Step 4: path steering. VLAN ID and destination MAC can differ between the copies. These fields help normal bridge forwarding send the copies along different configured paths.
Step 5: recovery invariant. Both copies keep R-TAG sequence number 101. This is the value the downstream recovery function uses to realize that the two arrivals are duplicates of the same original frame.

Merge-point behavior for the same example

ArrivalPathSequence numberRecovery decisionReason
First copyX-A101PassThis is the first valid copy of original frame 101 for protected stream X.
Later copyX-B101DiscardThe recovery function has already passed sequence 101, so this later arrival is a duplicate.
Do not confuse these two roles: VLAN, destination MAC, and egress port help steer the replicated copies through the network. The R-TAG does not steer the frame. The R-TAG carries the sequence number that lets recovery accept the first valid copy and discard later duplicates.

8. Elimination and sequence recovery

Elimination is not a byte-by-byte duplicate comparison. It is a sequence-aware decision based on stream identity, sequence number, configured recovery algorithm, history length, and reset timeout.

At the merge point or listener Path A copy: stream X, seq 101 ──┐ ├──► sequence recovery instance for stream X Path B copy: stream X, seq 101 ──┘ Decision: first valid seq 101 -> pass upward later seq 101 -> discard as duplicate Output to application or next bridge: one copy of sequence 101 only

In Vector recovery, it is the newest accepted sequence number at the forward edge of the history window.New arrivals are compared against this value to decide whether they are duplicates, old-but-still-valid out-of-order frames, newer frames, or rogue/out-of-window frames. Comparisons must be modulo the sequence space, so after 65535, sequence 0 can be newer.

frerSeqRcvyHistoryLengthSize of the recovery history window.Must cover expected path-delay differences and reordering, but not be so large that stale frames are accepted too easily.
frerSeqRcvyResetMSecTime allowed to pass without accepting a valid packet for this recovery instance. 9. Match recovery versus vector recovery

IEEE 802.1CB defines recovery algorithms with different complexity. The two names trainees must understand are Match and Vector.

9.1 Match recovery algorithm

The match algorithm is the simpler mental model: after reset, accept the first packet. After that, discard packets whose sequence number equals the last accepted sequence number. Accept packets whose sequence number is different.

Match recovery, simplified

on reset:
    TakeAny = true

on packet(seq):
    if no valid sequence:
        apply TakeNoSequence policy
    else if TakeAny:
        accept(seq)
        RecovSeqNum = seq
        TakeAny = false
    else if seq == RecovSeqNum:
        discard as duplicate
    else:
        accept(seq)
        RecovSeqNum = seq
Arrival orderSequenceDecisionWhy
110PassFirst after reset.
210DiscardSame as last accepted.
311PassDifferent from last accepted.
410Pass in simple match logicDifferent from last accepted 11, even though it may be an old delayed copy. This is why match is simple but weaker under reordering.

9.2 Vector recovery algorithm

The vector algorithm keeps a history window. It can therefore remember more than just the last accepted sequence number. This allows it to handle a controlled amount of out-of-order arrival and still discard duplicates.

Vector recovery, simplified teaching version

state:
    RecovSeqNum       = most advanced accepted sequence number
    SequenceHistory[0] = whether RecovSeqNum has been seen
    SequenceHistory[1] = whether RecovSeqNum - 1 has been seen
    SequenceHistory[2] = whether RecovSeqNum - 2 has been seen
    ... up to history length H

on packet(seq):
    delta = signed_modulo_difference(seq - RecovSeqNum, 65536)

    if TakeAny:
        accept seq, set RecovSeqNum = seq, mark history[0] = seen
    else if delta >= H or delta <= -H:
        discard as rogue / out of recovery window
    else if delta <= 0:
        index = -delta
        if SequenceHistory[index] == seen:
            discard as duplicate
        else:
            accept as old-but-not-yet-seen out-of-order packet
            mark SequenceHistory[index] = seen
    else:
        shift history forward by delta positions
        accept seq as newer packet
        set RecovSeqNum = seq
        mark history[0] = seen

9.3 Vector recovery worked example

Assume H = 4, RecovSeqNum = 105, and the recovery function has seen sequences 105, 104, and 102, but not 103.

Teaching rule: Match recovery remembers only the last accepted sequence. Vector recovery remembers a window of recent sequence values. That is why vector recovery is the mechanism to explain when path delay differences and reordering matter.

11. Disjoint path engineering: what FRER relies on but does not create

FRER gives the data-plane redundancy behavior. It does not automatically produce independent paths. A controller, path-control protocol, engineering design, or static configuration must arrange the member paths.

Good path diversity T ── SW1 ── SW2 ── L \ / └── SW3 ── SW4 ┘ A single link failure is unlikely to kill both copies. Weak path diversity T ── SW1 ── shared link ── SW2 ── L \ same cable / └───────────────────────┘ The diagram looks redundant, but a common shared segment can still be a single point of failure.
Automotive warning: In vehicles, “disjoint” should not only mean graph-disjoint. Check shared switch silicon, shared power supply, shared connector, shared harness branch, shared zone controller, and shared thermal or mechanical exposure.

12. IEEE 802.1CBdb and IEEE 802.1CBcv

IEEE 802.1CBdb

Extended Stream Identification Functions. This amendment expands the stream identification toolbox. Understand it as the answer to the practical deployment problem: “How do I classify real vehicle or industrial streams precisely enough for FRER?”

IEEE 802.1CBcv

Information Model, YANG Data Model, and Management Information Base Module. This is not a runtime duplicate-check packet protocol. Understand it as the management and observability layer for configuring and reporting FRER and stream-identification capabilities.

12.1 Why CBcv is important

FRER is only useful in production if it can be configured, audited, tested, and monitored. A validation engineer needs to know which stream identification functions are supported, which recovery algorithm is configured, what history length is used, what counters are exposed, and whether the device’s declared capabilities match the intended vehicle profile.

14. Counters, diagnostics, and validation

A production FRER implementation must expose evidence that the mechanism is actually doing what the architecture claims.

14.1 Validation test cases

Normal duplicate test: inject two copies with same stream and sequence number on two ports. Expected result: one passed, one discarded.
Path failure test: stop one member path. Expected result: stream continues using the remaining path with no switchover delay.
Delay skew test: delay one path enough to reorder copies. Expected result: vector recovery accepts legitimate in-window arrivals and discards duplicates.
Rogue sequence test: inject a sequence far outside the history window. Expected result: discard and rogue counter increment.
Talker reset test: reset sequence generation to zero while receiver still has old state. Expected result: temporary discard until configured recovery timeout, then recovery re-anchors.

15. Implementation checklist

Transmit side

  • Define protected streams precisely.
  • Map each stream to a stream_handle.
  • Instantiate sequence generation per stream and direction.
  • Insert R-TAG or configured sequence encoding.
  • Replicate to configured member streams.
  • Ensure member paths are actually diverse enough.

Receive / merge side

  • Identify member streams and map them to recovery context.
  • Decode R-TAG and extract sequence number.
  • Select match or vector recovery intentionally.
  • Choose history length based on delay difference and burst behavior.
  • Set reset timeout based on expected inter-arrival and fault behavior.
  • Monitor discard, rogue, lost, out-of-order, and tagless counters.

16. Common mistakes

MistakeCorrection
“FRER creates the redundant paths.”No. FRER uses configured paths. Path creation belongs to topology design, bridge forwarding configuration, IEEE 802.1Qca, or a controller.
“R-TAG tells the bridge where to forward the frame.”No. R-TAG carries sequence information for recovery. Forwarding still depends on normal bridge forwarding and configured stream/member-path state.
“Duplicate elimination is just byte comparison.”No. It is stream-aware and sequence-aware recovery.
“The first path is active and the second path is backup.”Not in the normal FRER protection model. Both copies can be sent simultaneously, and the receiver accepts the first valid one.
“Any two paths are enough.”No. They must be independent enough for the fault model. Shared harness, power, switch, or connector can defeat redundancy.
“History length can be arbitrary.”No. It must reflect path delay difference, burst behavior, and expected reordering. It is a safety-relevant engineering parameter.

17. Summary

IEEE 802.1CB FRER gives Ethernet a seamless redundancy mechanism by classifying frames into streams, assigning sequence numbers, encoding those numbers into frames, replicating frames over member paths, and eliminating duplicates through sequence recovery. The R-TAG is the key packet-level header: it is 6 octets long, uses EtherType 0xF1C1, has a 16-bit reserved field, and carries a 16-bit sequence number.

The core implementation insight is that FRER is not one feature but a coordinated chain: stream identification creates the context; sequence generation creates identity across replicas; R-TAG carries the sequence number; replication creates member streams; vector or match recovery decides what to pass and what to discard; counters prove whether the system is behaving correctly.

In automotive Ethernet, FRER is valuable because it can maintain delivery of critical streams without waiting for failure detection and switchover. But it only delivers real reliability when combined with precise stream identification, carefully engineered path diversity, appropriate recovery parameters, and management support from IEEE 802.1CBdb and IEEE 802.1CBcv.
