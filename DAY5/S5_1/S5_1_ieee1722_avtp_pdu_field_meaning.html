1. Where IEEE 1722 sits in the Ethernet frame

IEEE 1722 Audio Video Transport Protocol (AVTP) is normally carried directly over Ethernet as a Layer 2 payload. The Ethernet type is 0x22F0. This means that a normal AVB/TSN endpoint does not first parse Internet Protocol, User Datagram Protocol, or Transmission Control Protocol for the ordinary AVTP stream case.

Ethernet frame carrying IEEE 1722 AVTP

+----------------------+  6 bytes

+----------------------+
| Optional 802.1Q tag  |  4 bytes if VLAN-tagged
|  TPID = 0x8100       |
|  PCP / DEI / VLAN ID |
+----------------------+
| EtherType = 0x22F0   |  2 bytes
+----------------------+

+----------------------+

2. Common AVTP stream-data fields

Many timed AVTP formats share a common stream-data header structure. The exact subtype-specific words differ, but the core parser logic starts with the same idea: identify the AVTP subtype, bind the frame to a stream, check sequence continuity, and interpret the timestamp flags.

Corrected full timed AVTP stream-data PDU skeleton
The payload does not stop at avtp_timestamp. The timestamp is still header.
The real media/control data begins after the subtype-specific header fields.

FieldSizeMeaningImplementation / receiver interpretation
subtype8 bitsIdentifies the AVTPDU format.Dispatch field. For example 0x02 means AAF, 0x03 means CVF, 0x04 means CRF, 0x05 means TSCF, and 0x82 means NTSCF in common implementation headers.
SV1 bitStream ID Valid.If set, stream_id is meaningful and the receiver can bind the packet to a configured stream context. If clear, the packet cannot be matched using the normal stream identifier path.
version3 bitsAVTP version field.Used for compatibility checks. An implementation should reject or separately handle unsupported versions.
mr1 bitMedia clock restart.Signals a media clock discontinuity or restart. A listener may reset clock-recovery state, flush media timing history, or mark a discontinuity.
gv1 bitGateway information valid, for formats that use gateway information.When present and valid, gateway-specific information is meaningful. Not every subtype uses this bit in the same way.
tv1 bitAVTP timestamp valid.If tv = 1, avtp_timestamp carries a valid presentation timestamp. If tv = 0, the receiver must not use the timestamp field as a valid presentation time.
sequence_num8 bitsPer-stream AVTP sequence number.Incremented by the talker per AVTPDU. The listener uses it to detect loss, duplication, or reordering. It wraps modulo 256.
tu1 bitTimestamp uncertain.If set, timing is uncertain even if a timestamp field is present. A listener should treat the timing as degraded and may report diagnostics, conceal media, or avoid precise presentation.
stream_id64 bitsUnique stream identifier.Used to bind the AVTPDU to a configured stream. A common construction is source MAC address plus a 16-bit unique identifier.
avtp_timestamp32 bitsPresentation time, not transmit time.The listener consumes, renders, or applies the payload at this synchronized time. It is not simply the time at which the packet left the talker.
Implementation rule: timestamp handling is conditional. A parser must inspect tv and tu before treating avtp_timestamp as a reliable presentation time.

3. AAF PCM header fields and meanings

AAF means AVTP Audio Format. In the PCM case, the header tells the listener what audio sample representation is present, what nominal sampling rate is used, how many channels are packed per audio frame, how many valid bits exist per sample, how much payload is present, and whether sparse timestamp behavior is used.

mr1 bitMedia clock restart.Signals that the media clock relationship has restarted or changed. A listener should reset/interrogate media clock recovery state rather than blindly continuing interpolation.

sequence_num8 bitsPacket sequence counter for this stream.Used to detect dropped or duplicated audio packets. Loss handling may insert silence, conceal, or report diagnostics.

stream_id64 bitsAudio stream identifier.Maps the packet to a configured audio sink, channel map, presentation buffer, and reservation context.
avtp_timestamp32 bitsAudio presentation timestamp.Listener buffers the packet until this time before presenting the designated sample. It is not the send time.

sp1 bitSparse timestamp indication.0 means normal timestamp mode. 1 indicates sparse timestamp mode, where valid timestamps may be carried less frequently and the receiver must interpolate based on stream timing rules.
evt4 bitsEvent field.Used to indicate stream events when defined by the format/profile. For ordinary PCM streams, implementations commonly set it to zero unless a profile defines specific event semantics. A receiver should not invent behavior for undefined event values.

The value controls sample decoding. A mismatch between format, bit_depth, and actual payload layout leads to wrong audio reconstruction even if the Ethernet transport is perfect.

The receiver should treat this as the nominal media rate used for timing and buffer scheduling, not as a replacement for actual clock recovery.

3.3 AAF talker and listener behavior

AAF talker packetization

1. Take samples from the audio engine or ADC side.
2. Choose the number of audio frames per AVTPDU.
3. Fill stream identity and sequencing fields.
4. Fill format, nominal sample rate, channel count, bit depth, and payload length.
5. Compute avtp_timestamp as presentation time, not transmit time.
6. Append PCM payload.
7. Send with EtherType 0x22F0 and AVB/TSN traffic class.

AAF listener behavior

1. Match stream_id against configured stream.
2. Check sequence_num continuity.
3. Check tv and tu before trusting avtp_timestamp.
4. Use format/nsr/channels_per_frame/bit_depth to parse samples.
5. Buffer until presentation time.
6. Deliver samples to the audio renderer at the recovered media clock rate.

6. ACF carrier headers: TSCF and NTSCF

ACF means AVTP Control Format. ACF messages such as CAN and LIN are carried inside either TSCF or NTSCF. TSCF gives the ACF message list a presentation-time contract. NTSCF does not.

An ACF CAN message is not a textual encoding of a CAN frame. It is a binary ACF message with a typed ACF common header, CAN-specific control bits, a bus identifier, an optional message timestamp, a CAN identifier, and a padded payload.

7.2 ACF_CAN_BRIEF full PDU diagram

ACF_CAN_BRIEF is the abbreviated CAN ACF message. It is not a separate AVTP subtype. It is an ACF message type carried inside the ACF payload of either TSCF or NTSCF. Compared with ACF_CAN, it uses a shorter 8-byte ACF CAN Brief header and omits the 64-bit message_timestamp word pair from the message body. The mtv bit still exists as a field, but there is no per-message timestamp field in this brief message layout.

The ACF LIN message is shorter than the ACF CAN message. It identifies the ACF message type, length, padding, timestamp validity, logical LIN bus, LIN identifier, and optional message timestamp.

10. Implementation checklist and mistakes to avoid

10.1 Receiver checklist

  1. Accept the Ethernet frame only if MAC filtering, VLAN membership, and EtherType conditions match.
  2. Check EtherType == 0x22F0 before dispatching to the AVTP parser.
  3. Read subtype first and dispatch to the correct parser.
  4. Check version against what the implementation supports.
  5. Check SV before trusting stream_id.
  6. Use sequence_num per stream, not globally across all AVTP packets.
  7. Use tv and tu before trusting avtp_timestamp.
  8. For AAF, validate format, nsr, channels_per_frame, bit_depth, and stream_data_length before handing samples to the audio engine.
  9. For CVF, validate format, format_subtype, codec payload fields, marker behavior, and payload length before decode.
  10. For CRF, validate type, pull, base_frequency, timestamp_interval, and timestamp payload length before clock recovery.
  11. For TSCF/NTSCF, bound the ACF parsing loop using stream_data_length or ntscf_data_length.
  12. For CAN/LIN ACF messages, use acf_msg_length and pad to move safely through packed message lists.

10.2 Common mistakes

MistakeWhy it is wrongCorrect handling
Treating AVTP timestamp as send timeThe AVTP timestamp is a presentation/application time.Use it for receiver-side scheduling, not for measuring talker transmit time.
Ignoring tuA timestamp can be present but uncertain.Use both tv and tu.
Confusing CVF ptv with AVTP tvtv validates avtp_timestamp; ptv validates codec payload timestamp.Check both in their proper domains.
Parsing ACF without length boundsMalformed length can corrupt the message-list walk.Use carrier length plus each acf_msg_length and validate alignment/padding.
Using NTSCF for time-synchronous controlNTSCF has no AVTP presentation timestamp.Use TSCF when synchronized application time matters.
Treating message_timestamp as TSCF presentation timeACF message timestamp and TSCF AVTP timestamp have different meanings.Use message_timestamp for original bus-event timing, and avtp_timestamp for synchronized application/forwarding time.
Drawing only the AVTP header and forgetting the payloadThis makes it look as if avtp_timestamp, stream_id, or the carrier header is the whole PDU.Always draw the full PDU: Ethernet envelope, AVTP carrier header, subtype-specific header, payload, optional padding, and FCS.

11. Source basis and conformance note

This training page is based on field names, header lengths, and value mappings exposed by public AVTP reference implementations and public automotive/AVTP material. It is suitable for training and implementation orientation.

Conformance note: for product conformance, final bit-level behavior must be checked against the purchased IEEE 1722 standard version and the exact AVnu, AUTOSAR, or OEM profile used by the program.   • AAF format and NSR mappings are reflected in the public libavtp AAF header.
