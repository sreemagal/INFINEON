Training material on combined TSN stream protection with implementation details, header fields, examples, and diagrams

This page expands the earlier sketch into implementation-oriented material. It explains how stream classification, Per-Stream Filtering and Policing (PSFP), Frame Replication and Elimination for Reliability (FRER), Time-Aware Shaping (TAS), and frame preemption cooperate in a deterministic automotive Ethernet switch pipeline.

1. Start with the real systems problem

A deterministic Ethernet stream can fail in several different ways. It can be misclassified, sent on the wrong path, overrun its traffic contract, miss a scheduled transmission window, be blocked by a large lower-priority frame, or lose one redundant path while the application still appears healthy.

The important point is that no single Time-Sensitive Networking (TSN) mechanism solves all of these. IEEE 802.1CB FRER protects against path or component failure. IEEE 802.1Qci PSFP protects the network from bad or over-rate streams. IEEE 802.1Qbv TAS gives scheduled transmission windows. IEEE 802.1Qbu / IEEE 802.3br preemption reduces blocking from already-started lower-priority traffic.

Correct framing: FRER is only one layer of stream protection. The complete implementation story needs classification, policing, scheduling, preemption, redundancy, recovery, and observability.
Failure mode Mechanism that mainly handles it ──────────────────────────────────── ───────────────────────────────────────── Wrong stream recognized Stream identification / classification Over-size or over-rate sender IEEE 802.1Qci PSFP filter + gate + meter Late due to queueing interference IEEE 802.1Qbv time-aware gates Blocked by large low-priority frame IEEE 802.1Qbu / IEEE 802.3br preemption Link or bridge path failure IEEE 802.1CB FRER Hidden degradation after one path loss FRER counters and management visibility

2. Combined protection pipeline inside a TSN bridge

A useful implementation model is to walk a frame through the switch pipeline. The exact ordering can vary by silicon, but the logical responsibilities are stable.

Ingress port │ ▼ +----------------------+ Uses Ethernet/VLAN/IP/L4 fields | Parser | ────────────────────────────────────────┐ +----------------------+ │ │ │ ▼ │ +----------------------+ Produces stream_handle / stream ID │ | Stream identification| ◄────────────────────────────────────────┘ +----------------------+ │ ▼ +----------------------+ Stream filter, stream gate, meter, max SDU | Qci PSFP | pass / drop / mark / count +----------------------+ │ accepted frames only ▼ +----------------------+ Sequence generation, R-TAG insertion, | 802.1CB FRER | replication into member streams +----------------------+ │ ├──────── copy A ─────────► egress queue/path A │ └──────── copy B ─────────► egress queue/path B │ ▼ +----------------------+ | Qbv gate + scheduler | +----------------------+ │ ▼ +----------------------+ | MAC Merge preemption | +----------------------+ │ ▼ Physical link

Ingress-side discipline

Classification and Qci decide whether the frame is a valid member of a protected stream. This avoids replicating traffic that should have been blocked.

Egress-side discipline

Qbv and preemption determine whether each accepted FRER copy can leave at the intended time without being blocked by other traffic.

3. Protocol header fields used for stream classification

Qci and FRER do not operate on anonymous packets. A frame must first be recognized as part of a stream. That recognition is usually based on a rule over Ethernet, VLAN, IP, and transport-layer fields.

There is not always a new on-wire header for Qci or Qbv. These bridge functions often inspect existing packet headers and internal metadata. The protocol header details matter because the classifier must know exactly which bytes mean destination MAC, source MAC, VLAN identifier, Priority Code Point, EtherType, IP addresses, protocol, Differentiated Services Code Point, and transport ports.

4. Stream identification details

Stream identification maps a frame to a stream_handle or equivalent internal stream identity. That stream identity is then reused by FRER, Qci, and Quality of Service (QoS) functions.

Example stream identification rule IF ingress_port in {P1} AND destination_mac = 91:E0:F0:00:12:34 AND VLAN ID = 20 AND PCP = 5 AND EtherType = IPv4 AND UDP dst port = 30490 THEN stream_handle = "front_camera_critical_stream" internal_priority = 5 FRER enabled = yes Qci filter = 17

Why ingress port matters

The same MAC/VLAN tuple may appear on more than one port in a large network. Ingress port prevents false matches and can express topology assumptions.

Why VLAN/PCP matters

VLAN Identifier and Priority Code Point often map the frame into a traffic class and a TSN service domain.

Why IP/L4 fields matter

In a zonal architecture, many services may share the same VLAN. Extended stream identification can select the exact critical flow.

5. IEEE 802.1Qci PSFP internals

Per-Stream Filtering and Policing (PSFP) protects the bridge from frames that do not match the configured stream behavior. It can filter, gate, meter, mark, count, and discard frames before they damage the rest of the deterministic fabric.

Qci is not an on-wire protocol header. It is a bridge forwarding-process enhancement. It uses frame classification and managed tables.

6. Qci examples and packet walk-throughs

6.1 Correct frame passes Qci

1. Parser: frame has VLAN ID 20, PCP 5, IPv4/UDP destination port 30490.
2. Stream identification: rule matches front_camera_critical_stream.
3. Stream filter: frame length is 900 bytes, below max-sdu = 1200.
4. Stream gate: gate is open for the expected camera interval.
5. Flow meter: stream is within rate and burst contract.
6. Result: frame is accepted and can be passed to FRER sequencing/replication.

6.2 Oversize frame is blocked before FRER amplification

Input frame: stream = front_camera_critical_stream, size = 1600 bytes Configured max-sdu = 1200 bytes Stream filter check: 1600 > 1200 → fail Maximum SDU filter Result: frame is discarded before FRER replication oversize counter increments optional stream-blocked flag may be set
Why this matters: if the bridge replicated the oversize or over-rate frame first, a single bad sender could consume bandwidth on both redundant paths. Qci prevents invalid traffic from being redundantly amplified.

6.3 Gate-closed frame

Expected stream behavior: one frame every 1 ms, allowed arrival window = 100 µs Actual frame arrival: arrives 500 µs outside the allowed window Qci stream gate: state = CLOSED at this instant Result: frame is discarded or counted as gate violation FRER does not see this frame

7. FRER R-TAG and sequence fields

FRER must allow the receiver to recognize that two arriving frames are copies of the same original frame. The usual on-wire mechanism is the R-TAG, or Redundancy Tag.

For the 802.1CB R-TAG, the Tag Protocol Identifier uses EtherType 0xF1C1. The tag information contains a 16-bit Reserved field and a 16-bit Sequence Number field. The sequence number carried in the R-TAG is the low-order 16 bits of the stream sequence number.

Original stream frames at talker: Frame A → sequence = 100 Frame B → sequence = 101 Frame C → sequence = 102 After replication: Path 1 carries: A(seq100), B(seq101), C(seq102) Path 2 carries: A(seq100), B(seq101), C(seq102) At receiver: First seq100 copy accepted Second seq100 copy discarded as duplicate First seq101 copy accepted Second seq101 copy discarded as duplicate
Important: the sequence number has meaning only after stream identification. Sequence number 100 in Stream A is unrelated to sequence number 100 in Stream B.

8. FRER replication and elimination

8.1 Replication function

Replication takes one accepted stream frame and creates two or more member-stream copies. Those copies usually have the same payload and the same FRER sequence value, but they may be forwarded using different destination MAC addresses, VLANs, or explicit path-control state depending on the network design.

One original protected stream frame │ ▼ +--------------------------+ | Sequence generation | | seq = 0x038A | +--------------------------+ │ ▼ +--------------------------+ | R-TAG encode | | insert TPID F1C1, seq | +--------------------------+ │ ├──────────── member stream A ───────────► Path A │ └──────────── member stream B ───────────► Path B

8.2 Elimination / sequence recovery

At a merge point or final receiver, sequence recovery uses the stream identity and sequence number to determine whether a frame is new, duplicate, rogue, too old, or out of order.

Arrivals at receiver for one stream Time 10.000 ms: Path A, seq 200 → accepted, deliver upward Time 10.030 ms: Path B, seq 200 → duplicate, discard Time 11.000 ms: Path B, seq 201 → accepted, deliver upward Time 11.400 ms: Path A, seq 201 → duplicate, discard Time 12.000 ms: no seq 202 copy → lost, if recovery window/timer confirms absence Time 13.000 ms: Path A, seq 203 → accepted, lost counter may account for seq 202

8.3 Why elimination is not just byte comparison

Byte comparison is not enough because copies can arrive at different times, may have link-local metadata differences, and can be slightly reordered. FRER needs sequence-aware state:

per_stream_state = {
    last_accepted_sequence,
    recovery_window,
    history_vector or match_window,
    counters: passed, discarded, lost, rogue, out_of_order
}

The receiver should not accept a late duplicate just because it looks like a valid Ethernet frame. It must ask whether the sequence value is already inside the accepted history.

9. IEEE 802.1Qbv TAS fields and examples

Time-Aware Shaper (TAS) uses per-queue transmission gates. A closed gate makes a queue unavailable for transmission, even if frames are waiting. An open gate lets the queue participate in the normal transmission-selection process.

Qbv is not an on-wire packet header. It is a port schedule. The important protocol-facing fields are management/schedule fields such as gate state bitmasks, base time, cycle time, and Gate Control List entries.

9.1 Gate states as an 8-bit traffic-class mask

Example gate state byte for 8 traffic classes Bit: 7 6 5 4 3 2 1 0 TC: 7 6 5 4 3 2 1 0 State: 0 0 1 0 0 0 0 1 Meaning: Traffic Class 5 gate is OPEN Traffic Class 0 gate is OPEN All other traffic-class gates are CLOSED

Key point: FRER does not bypass Qbv. Each replicated copy is still subject to the gate state and scheduler on every egress port it traverses.

10. Frame preemption details

Frame preemption solves serialization blocking. Without preemption, once a large lower-priority frame starts transmission, a critical frame must wait until the whole frame is finished. With preemption, the lower-priority frame can be interrupted so an express frame can be sent sooner.

IEEE 802.1Qbu defines the bridge traffic-class selection aspect, while IEEE 802.3br defines the MAC Merge sublayer behavior. The MAC Merge sublayer separates traffic into an express MAC path and a preemptable MAC path.

10.1 MAC Merge model

Higher bridge queues │ ├──────── Express traffic classes ───────► eMAC ───┐ │ │ └──────── Preemptable traffic classes ───► pMAC ───┤ ▼ MAC Merge Sublayer │ ▼ Physical link

10.2 What changes on the wire?

The original Ethernet service interface remains Ethernet-like to higher layers, but at the MAC Merge layer preemptable frames may be transmitted as fragments. The Start mFrame Delimiter (SMD) identifies whether an mFrame is express, the start of a preemptable frame, or a continuation fragment of a preempted frame.

Without preemption Large best-effort frame ────────────────────────────────► critical frame waits With preemption Preemptable fragment 1 ──┐ ├─ Express critical frame ──► Preemptable continuation ┘

11.3 Bad-sender case: over-rate stream

Sensor ECU sends 3× configured rate │ ▼ Qci flow meter detects non-conforming traffic │ ├─ drop red frames before FRER │ └─ accepted conforming frames continue to FRER Result: FRER protects valid traffic network is not flooded with replicated invalid traffic

11.4 Timing case: Qbv window and preemption

At egress port, TC5 critical stream window opens at 0 µs Without preemption: a 1500-byte best-effort frame that started at -5 µs may block the TC5 frame. With preemption: best-effort frame is sent through pMAC and can be fragmented. TC5 express frame interrupts and enters the scheduled window with lower blocking.
Layered meaning: Qci decides whether the stream is valid, FRER protects against path failure, Qbv protects the time window, and preemption reduces blocking by lower-priority frames.

12. Failure observability and counters

FRER hides failures from the application by design. Therefore, management counters are essential. Otherwise a vehicle may continue working after one redundant path has failed, but it is no longer protected against the next fault.

CounterMeaningHealthy interpretationWarning interpretation
rx-passed-pktsFrames accepted by sequence recovery.Should increase with delivered stream frames.Stops increasing when no valid copy arrives.
rx-discarded-pktsDuplicates or invalid/rogue frames discarded.Duplicate discards are normal in FRER.Unexpected surge may indicate misrouting or sequence errors.
rx-lost-pktsSequence values not received on any ingress port.Should remain zero or within validated limits.Critical reliability problem.
rx-out-of-order-pktsFrames arriving outside expected order.Small amounts can occur with path skew.Growth indicates excessive asymmetry or path issue.
rx-rogue-pktsUnexpected sequence behavior or frame not acceptable to recovery.Should normally be zero.May indicate classification, routing, replay, or corruption issue.
encode-errored-pktsSequence encode/decode function could not process the metadata.Should be zero.Malformed R-TAG, parser problem, or interoperability issue.
generation-resetSequence generation reset event.Expected only during controlled startup/reconfiguration.Unexpected reset can disturb recovery windows.
Important: duplicate discards are not automatically bad. In healthy 1+1 FRER, duplicate elimination is expected. The more important question is whether one path silently stopped contributing, whether lost packets are increasing, or whether rogue/out-of-order counters are trending upward.

13. Automotive design examples

13.1 Critical control stream

Design goal

A chassis or motion-control stream must arrive within a tight cycle and must survive one link failure without application-level switchover delay.

Mechanism set

Use stream identification, Qci max-SDU and meter checks, Qbv scheduled egress windows, express traffic class with preemption support, and FRER across disjoint paths.

Central compute ── SW1 ── SW2A ── Actuator zone │ │ │ └── SW2B ───── Actuator zone │ └─ FRER copies use path A and path B Common-cause check: Are SW2A and SW2B physically independent? Do they share power? Do they share the same harness segment? Does the GCL reserve compatible windows on both paths?

13.2 Camera or sensor stream

Camera stream problem: high bandwidth, periodic bursts, strict latency requirement, possible path failure Recommended stack: classification: camera MAC/VLAN/IP/UDP tuple Qci: max SDU + meter to block malformed or over-rate bursts Qbv: scheduled windows on bottleneck ports preemption: protect windows from large background frames FRER: duplicate only if reliability requirement justifies bandwidth cost

13.3 Diagnostics and logging

Diagnostics and logging usually should not consume FRER resources unless there is a specific safety or availability requirement. They may still use Qci policing and lower-priority scheduled or weighted service to avoid disturbing critical traffic.

15. Common wrong explanations

Wrong: “FRER creates the redundant paths.”

Correct: FRER uses multiple paths. Path construction is done by topology design, explicit path control, or centralized configuration.

Wrong: “Qci is just a rate limiter.”

Correct: Qci includes stream filters, stream gates, maximum SDU filtering, flow meters, priority assignment, blocking behavior, and counters.

Wrong: “Qbv guarantees timing by itself.”

Correct: Qbv controls queue eligibility by time. The frame must still fit the window, avoid blocking, and pass the local scheduler.

Wrong: “Preemption is another priority queue.”

Correct: preemption is a MAC Merge mechanism that allows express frames to interrupt preemptable frame transmission.

Wrong: “Duplicate discards mean errors.”

Correct: duplicate discards are normal in FRER. Lost, rogue, encode-error, or changed per-path patterns are more concerning.

Wrong: “Enabling all standards automatically gives safety.”

Correct: the standards are composable, but the safety and timing argument depends on correct configuration, validation, and monitoring.

16. Summary

Combined TSN stream protection is a layered design. Stream identification decides which packets belong to the protected stream. Qci PSFP decides whether those packets are admissible according to size, gate, and meter rules. FRER adds sequence information, replicates frames over redundant paths, and eliminates duplicates. Qbv TAS controls when egress queues may transmit. Frame preemption reduces blocking by preemptable lower-priority frames. Management counters expose whether the protected stream is still redundant and healthy.

One-sentence implementation memory aid Identify the stream → police the stream → replicate the valid stream → schedule each copy → preempt blockers → recover one clean stream → monitor hidden degradation.
