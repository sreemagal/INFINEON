CAN-over-Ethernet tunnelling using IEEE 1722 ACF

A first-principles explanation of why CAN tunnelling is needed, how the AVTP Control Format carries CAN and CAN FD frames, and how a VLAN-aware automotive Ethernet switch forwards, filters, schedules, or terminates those frames.

CAN and CAN FD IEEE 1722 AVTP ACF_CAN / ACF_CAN_BRIEF 802.1Q VLAN and PCP Stream ID based filtering Zonal architecture

1. The mental model

Controller Area Network (CAN) is a local fieldbus. A CAN frame is identified mainly by a CAN identifier, not by a source and destination address. The identifier says what the message means and also participates in bus arbitration. Ethernet is a frame-switched network. An Ethernet frame is forwarded using a destination Media Access Control (MAC) address, a source MAC address, optionally a Virtual Local Area Network (VLAN) tag, and an EtherType that identifies the payload protocol.

CAN-over-Ethernet tunnelling does not turn CAN into Ethernet at the signal level. The CAN frame is captured, wrapped inside an IEEE 1722 Audio Video Transport Protocol (AVTP) frame as an AVTP Control Format (ACF) message, transported over Ethernet, unpacked at the far end, and then optionally retransmitted onto a local CAN bus.

CAN view

“A CAN frame with identifier 0x18DAF110 appeared on CAN bus 2.”

ACF view

“This CAN frame becomes one ACF_CAN or ACF_CAN_BRIEF message in an AVTP ACF payload.”

Ethernet switch view

“This is an Ethernet frame with VLAN, priority, destination MAC, EtherType 0x22F0, and an AVTP Stream ID.”

The most important practical distinction is this:

  • A plain Layer 2 VLAN switch normally forwards the IEEE 1722 ACF frame as an Ethernet frame. It does not need to understand the CAN identifier.
  • An ACF-aware zonal gateway or switch-integrated gateway terminates or originates the ACF stream. It parses the ACF message, maps can_bus_id and can_identifier to a physical CAN controller or routing rule, and then transmits or receives real CAN frames.

2. Why tunnel CAN over Ethernet?

Vehicle networks are moving from many small domain-specific buses toward zonal and centralized architectures. Legacy CAN, CAN FD, Local Interconnect Network (LIN), and other fieldbus islands still exist close to sensors and actuators, but high-bandwidth inter-zone communication increasingly uses Ethernet. Tunnelling solves the mismatch without replacing every end device at once.

Need 1: Preserve legacy CAN semantics

Existing Electronic Control Units (ECUs), diagnostics, and application software may already be built around CAN identifiers, CAN payloads, and CAN timing. Tunnelling keeps the CAN frame intact over the Ethernet backbone.

Need 2: Reduce gateway explosion

Instead of a chain of CAN gateways translating message by message, a zonal gateway can encapsulate collected CAN frames into an Ethernet stream and send them to another zone or to a central compute node.

Need 3: Use Ethernet quality-of-service

Once wrapped into Ethernet, the traffic can use VLAN membership, Priority Code Point (PCP), stream filters, policing, shaping, and scheduled or asynchronous traffic classes.

Need 4: Add stream timing

The time-synchronous ACF stream form can carry an AVTP presentation time. This lets a receiver defer delivery until the intended synchronized time where the implementation supports that behavior.

Old distributed style:

  CAN island A  ── gateway ── CAN/FlexRay/Ethernet ── gateway ── CAN island B

Zonal style with tunnelling:

  CAN devices ── zonal ECU ── Ethernet VLAN/TSN backbone ── zonal ECU ── CAN devices
                    │                                         │
                    └── pack CAN into IEEE 1722 ACF ──────────┘

3. What ACF is

AVTP Control Format (ACF) is the IEEE 1722 mechanism used to carry control or fieldbus-style messages inside an AVTP stream. In the automotive use case, an ACF stream can transport CAN, CAN FD, LIN, and other bus-frame messages as a sequence of ACF messages.

The architecture has two nested layers:

  1. The AVTP stream layer. This chooses whether the stream is non-time-synchronous or time-synchronous, identifies the stream, carries a sequence number, and gives the length of the ACF payload.
  2. The ACF message layer. The payload contains one or more ACF messages. Each ACF message begins with an ACF message type and an ACF message length, followed by message-type-specific fields and payload bytes.
Layer nesting
Ethernet frame
└─ Optional 802.1Q VLAN tag
   └─ EtherType = 0x22F0 for AVTP over Ethernet
      └─ AVTP ACF stream header
         ├─ NTSCF header, or
         └─ TSCF header
            └─ ACF payload
               ├─ ACF message #1: ACF_CAN or ACF_CAN_BRIEF
               ├─ ACF message #2: ACF_CAN or ACF_CAN_BRIEF
               └─ ACF message #n: possibly another supported ACF type
Important implementation point: ACF is not a separate Ethernet EtherType. It is a payload format inside IEEE 1722 AVTP. The Ethernet switch first sees an AVTP Ethernet frame. Only an ACF-aware function parses deeper into the AVTP subtype and ACF message fields.

5. NTSCF and TSCF ACF stream headers

An ACF payload is carried inside one of two AVTP stream subtypes:

  • Non-Time-Synchronous Control Format (NTSCF): used when the stream does not carry an AVTP presentation timestamp.
  • Time-Synchronous Control Format (TSCF): used when the stream carries an AVTP timestamp so receivers can schedule delivery relative to synchronized time.

5.1 NTSCF header view

NTSCF is the smaller form. It carries the AVTP subtype, header control flags, an ACF payload length, a sequence number, and the 64-bit Stream ID.

NTSCF ACF AVTPDU, simplified

+------------------------------ 32 bits ------------------------------+
| subtype=0x82 | common control bits | ntscf_data_length | seq_num     |
+---------------------------------------------------------------------+
| stream_id, high 32 bits                                             |
+---------------------------------------------------------------------+
| stream_id, low 32 bits                                              |
+---------------------------------------------------------------------+
| ACF payload: one or more ACF messages                               |
+---------------------------------------------------------------------+
Use NTSCF when: the receiver should forward the CAN frame as soon as it is processed, or when the deployment does not need an AVTP presentation time for the encapsulated bus frame.

5.2 TSCF header view

TSCF adds an AVTP timestamp and a stream-data length field. The timestamp-valid bit tells the receiver whether the AVTP timestamp is meaningful.

TSCF ACF AVTPDU, simplified

+------------------------------ 32 bits ------------------------------+
| subtype=0x05 | common control bits | sequence_num and status bits    |
+---------------------------------------------------------------------+
| stream_id, high 32 bits                                             |
+---------------------------------------------------------------------+
| stream_id, low 32 bits                                              |
+---------------------------------------------------------------------+
| avtp_timestamp                                                      |
+---------------------------------------------------------------------+
| format-specific fields and stream_data_length                       |
+---------------------------------------------------------------------+
| ACF payload: one or more ACF messages                               |
+---------------------------------------------------------------------+
Use TSCF when: the receiver must release the encapsulated CAN frame at a particular synchronized time, for example when a command must be aligned across several consumers.

6. ACF_CAN and ACF_CAN_BRIEF header formats

IEEE 1722 defines different ACF message types. For CAN tunnelling, the two important forms are ACF_CAN and ACF_CAN_BRIEF. Both carry the CAN identifier, CAN/CAN FD flags, a logical CAN bus identifier, and the CAN payload. The difference is that ACF_CAN includes a message timestamp field, while ACF_CAN_BRIEF omits it.

7. Packing multiple CAN frames into one ACF payload

The AVTP ACF payload can contain one or more ACF messages. This is important because a CAN bus produces small frames. Sending one Ethernet frame per tiny CAN frame is simple but inefficient. Collecting several CAN frames into one AVTPDU improves bandwidth efficiency but increases waiting time.

Low latency policy

Transmit immediately when a configured critical CAN identifier arrives. This minimizes delay but wastes more Ethernet overhead.

Batching policy

Collect until a byte threshold, frame-count threshold, or timeout fires. This improves bandwidth efficiency but adds bounded latency.

One AVTP ACF payload can look like this:

+------------------------+------------------------+------------------------+
| ACF_CAN_BRIEF msg #1   | ACF_CAN_BRIEF msg #2   | ACF_CAN msg #3         |
| CAN ID 0x120           | CAN ID 0x18DAF110      | CAN ID 0x301 + time    |
+------------------------+------------------------+------------------------+

Receiver parses message #1 using acf_msg_length,
then jumps to message #2,
then jumps to message #3,
then stops when the ACF payload length is consumed.
Trade-off: batching is a network-engineering decision. It must be chosen together with CAN signal deadlines, Ethernet queueing delay, VLAN priority, and stream policing limits.

9. How this is implemented in a VLAN switch

The phrase “implemented in a VLAN switch” can mean two different designs. Mixing them up causes confusion.

9.1 Case A: the switch is only an Ethernet VLAN bridge

In this case, the switch does not know that the payload contains CAN. It forwards the ACF traffic as ordinary IEEE 1722 Ethernet traffic.

Ingress Ethernet port
  ↓
Parser: Destination MAC, Source MAC, VLAN tag, EtherType 0x22F0
  ↓
VLAN ingress membership and filtering
  ↓
Optional stream identification: EtherType + VLAN + Destination MAC + IEEE 1722 Stream ID
  ↓
Optional PSFP: stream gate, flow meter, maximum frame size, destination-port action
  ↓
FDB / multicast / configured destination-port decision
  ↓
Priority regeneration and queue selection from PCP or internal priority
  ↓
Egress VLAN decision: tagged, untagged, or not a member
  ↓
Transmit unchanged AVTP/ACF payload
Key point: a pure bridge can enforce VLAN and stream-level behavior without parsing can_identifier. It may match EtherType 0x22F0 and IEEE 1722 Stream ID, but it usually leaves the ACF_CAN payload untouched.

9.2 Case B: the switch or zonal gateway is ACF-aware

In this case, the switch complex includes a CPU, Direct Memory Access (DMA) path, packet accelerator, or gateway engine that terminates the IEEE 1722 ACF stream. The Ethernet switch fabric still performs VLAN switching, but selected frames are redirected to an internal processing port.

Ethernet ingress port
  ↓
Normal VLAN bridge pipeline
  ↓
Stream filter matches: VLAN=100, EtherType=0x22F0, Stream ID=S_CAN_ZONE_2
  ↓
Destination-port action redirects or mirrors frame to internal ACF gateway port
  ↓
ACF gateway parses NTSCF/TSCF
  ↓
For each ACF message:
    if acf_msg_type == ACF_CAN or ACF_CAN_BRIEF:
        validate length, padding, fdf/brs/rtr/eff combination
        extract can_bus_id, can_identifier, payload
        apply whitelist/routing table
        enqueue to CAN controller TX FIFO
    else:
        drop or forward according to configured policy
  ↓
Physical CAN controller transmits on the selected CAN bus
Correct architectural statement: the VLAN switch forwards the IEEE 1722 ACF stream; the CAN tunnelling function is implemented either in the endpoints, in a switch-integrated gateway block, or in host software attached to an internal switch port. The pure 802.1Q bridge function is not itself a CAN controller.

9.3 Detailed VLAN switch pipeline for ACF traffic

1

Ingress parser

Extract MAC destination/source, optional 802.1Q tag, PCP, DEI, VLAN ID, EtherType.

2

Tag handling

If untagged, apply default VLAN and priority or drop. If tagged, validate TPID and VLAN policy.

3

VLAN membership

Drop if ingress port is not a member of the received or assigned VLAN.

4

Stream identification

Optionally match EtherType 0x22F0, VLAN, PCP, destination MAC, and IEEE 1722 Stream ID to assign a stream handle.

5

Filtering and policing

Apply PSFP: stream gate, flow meter, max SDU, destination-port modification, or VLAN modification.

6

Forwarding

Use FDB, multicast rules, or configured stream action to select egress ports.

7

Queueing

Map PCP/internal priority to traffic class. Apply shaping or scheduled gates where configured.

8

Egress

Transmit tagged, untagged, or not at all depending on egress VLAN membership and policy.

11. Worked byte-level example

Assume the zonal gateway receives a CAN FD frame from physical CAN channel 2:

Input CAN FD frame
  Physical channel: CAN2
  CAN identifier:   0x18DAF110
  Identifier type:  extended, 29-bit
  Payload length:   12 bytes
  Payload bytes:    11 22 33 44 55 66 77 88 99 AA BB CC
  CAN FD:           yes
  BRS:              yes
  ESI:              0

The gateway chooses ACF_CAN_BRIEF because it does not need a per-message timestamp. Since the payload is 12 bytes, it is already aligned to a 4-byte boundary, so pad = 0. The ACF message contains 8 bytes of ACF_CAN_BRIEF header plus 12 payload bytes, so the ACF message occupies 20 bytes, or 5 quadlets in the quadlet-oriented layout.

ACF_CAN_BRIEF construction
  acf_msg_type    = ACF_CAN_BRIEF
  acf_msg_length  = 5 quadlets
  pad             = 0
  mtv             = 0, because no message timestamp field is present
  rtr             = 0
  eff             = 1, extended CAN identifier
  brs             = 1
  fdf             = 1, CAN FD frame
  esi             = 0
  can_bus_id      = 2
  can_identifier  = 0x18DAF110
  can_payload     = 11 22 33 44 55 66 77 88 99 AA BB CC

The gateway then places this ACF message in an NTSCF AVTPDU and emits it as a VLAN-tagged Ethernet frame:

Ethernet / VLAN / AVTP wrapping
  Destination MAC  = configured stream destination, often multicast or central gateway unicast
  Source MAC       = zonal gateway MAC
  VLAN ID          = 100
  PCP              = 5
  EtherType        = 0x22F0
  AVTP subtype     = NTSCF, 0x82
  Sequence number  = previous + 1 modulo sequence range
  Stream ID        = S_CAN_ZONE2_BODY
  ACF payload      = the 20-byte ACF_CAN_BRIEF message above
Switch behavior in this example: a plain VLAN switch forwards this frame based on VLAN 100, destination MAC, PCP 5, and possibly the Stream ID. It does not need to know that the payload contains CAN ID 0x18DAF110. An ACF gateway at the destination parses the payload and retransmits a CAN FD frame on its configured CAN bus if policy allows it.

12. Engineering checks for a robust implementation

Length validation

Verify AVTP payload length, ACF message length, CAN payload length, and padding. Reject frames where the message length steps beyond the payload boundary.

Flag validation

Reject impossible combinations, such as a classical CAN remote frame combined with CAN FD semantics where unsupported by the endpoint.

Identifier whitelist

Do not transmit arbitrary received CAN identifiers to a safety-critical CAN bus. Match against configured ranges and directions.

Stream whitelist

Accept only configured Stream IDs, VLAN IDs, source ports, and destination MACs for each tunnel.

Rate limits

Use PSFP, flow metering, or gateway-level token buckets to prevent a faulty source from flooding the backbone or a CAN bus.

Timestamp sanity

For TSCF and timestamped ACF_CAN, check whether time synchronization is valid and decide what to do when timestamps are late or too far in the future.

Sequence monitoring

Track AVTP sequence numbers to detect loss, duplication, or reordering where this matters for the application.

VLAN hardening

Drop unexpected double-tagged frames, untrusted VLAN tags, and traffic arriving on ports not permitted for the tunnel VLAN.

13. Common misconceptions

MisconceptionCorrection
“The VLAN ID is the CAN bus number.”No. VLAN ID is an Ethernet switching and isolation field. The logical CAN bus number is can_bus_id inside the ACF_CAN or ACF_CAN_BRIEF message.
“Every switch must parse CAN identifiers.”No. Ordinary VLAN bridges can forward ACF streams using MAC, VLAN, PCP, EtherType, and Stream ID. CAN parsing is needed only where the tunnel is terminated, originated, or deeply inspected.
“CAN arbitration is preserved across Ethernet.”No. CAN arbitration happens only on the local CAN bus. Across Ethernet, priority comes from VLAN PCP, traffic class, shaping, gating, and stream configuration.
“Tunnelling and translation are the same.”No. Tunnelling preserves the original bus frame inside an Ethernet transport. Translation changes the communication model or signal representation.
“TSCF is always better than NTSCF.”No. TSCF is useful when presentation time matters. NTSCF is smaller and often sufficient for immediate forwarding.
