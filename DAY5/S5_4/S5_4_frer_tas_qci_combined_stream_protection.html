Time-Sensitive Networking training note

FRER + TAS + Qci Combined Stream Protection

A standards-oriented explanation of how Frame Replication and Elimination for Reliability, Time-Aware Shaping, and Per-Stream Filtering and Policing work together to protect automotive and industrial Ethernet streams. The important lesson is that these mechanisms do not protect the same failure mode. FRER protects delivery against path loss. Qci protects the switch against wrong, oversized, mistimed, or over-rate streams. TAS protects bounded egress timing.

IEEE 802.1CB IEEE 802.1Qci / PSFP IEEE 802.1Qbv / TAS IEEE 802.1AS time base

1. The combined protection idea

Think of a critical stream as a train that must reach the listener at a precise time and must not be allowed to damage the rest of the network if it behaves incorrectly.

FRER protects delivery

The talker or an intermediate bridge creates member-stream copies of the same frame. Each copy carries the same sequence value. The listener or merge point accepts one valid copy and discards later duplicates.

Failure addressed: one link, bridge, or path loses a frame.

Qci protects the ingress pipeline

A stream is identified and matched to a stream filter. Maximum size, stream gate, and flow meter decisions decide whether the frame can enter the forwarding path.

Failure addressed: rogue rate, wrong size, wrong priority, wrong stream, or burst flood.

TAS protects egress time

Egress queues have gates that open and close according to a time-aware gate control list. A frame can be transmitted only when its queue gate is open and the transmission can fit safely.

Failure addressed: loss of deterministic timing due to uncontrolled queue transmission.

The most important design rule: FRER does not replace Qci and Qci does not replace TAS. FRER can hide path loss, but it can also double offered load on member paths. Qci prevents each member path from becoming a policing blind spot. TAS then gives the admitted frames deterministic egress windows.

Why the mechanisms are usually deployed together

FRER alone is not enough

FRER duplicates frames. If each copy is not separately filtered and policed, a misconfigured talker or a duplicated burst can consume both redundant paths and the egress queues. FRER improves probability of delivery but does not by itself enforce the stream's traffic contract.

TAS alone is not enough

TAS gives deterministic egress service only to frames that have already been admitted into the correct queue. If oversized, over-rate, or spoofed frames are allowed into that queue, the scheduled window can still be polluted.

Qci alone is not enough

Qci can reject bad or excessive frames, but it cannot make a lost frame reappear on another path. It also does not schedule the egress port by itself.

Time sync is the silent dependency

TAS assumes that bridge egress gate schedules have a meaningful common time base. Without a correct time base, two redundant paths can deliver valid FRER copies at unexpected times.

What the R-TAG does

It carries the low-order sequence value used by FRER recovery. Copies of the same original frame carry the same sequence number, allowing a recovery point to accept the first valid copy and reject later duplicates.

What the R-TAG does not do

It does not carry the whole route, the TAS schedule, the Qci meter state, or the reason why a frame was dropped. Those are bridge configuration and runtime state, not data-frame fields.

4. Combined switch pipeline

A useful standards-oriented order is to separate ingress admission, FRER sequence handling, forwarding/replication, and egress time scheduling. Actual silicon pipelines differ, but the conceptual dependencies are stable.

1. ParseRead DA, SA, VLAN, EtherType, possible R-TAG, payload headers.
2. Stream IDCreate stream handle/member-stream handle from configured match rules.
3. Qci PSFPMax SDU, stream gate, flow meter, priority metadata. Drop bad traffic early.
4. FREREncode, split, decode, recover, or eliminate duplicates depending on node role.
5. ForwardFDB/VLAN/multicast destination selection and per-member-path egress choice.
6. QueuePriority/internal priority to traffic class to egress queue.
7. TASTransmit only when the queue gate is open and the frame can fit.
Important distinction: Qci stream gates and TAS queue gates are not the same gate. A Qci stream gate is an ingress admission control associated with a stream filter. A TAS gate is an egress queue transmission gate associated with a traffic class/queue and a time schedule.

4.2 Where mistakes usually happen

Policing original stream only

If only the compound stream is policed before replication, each member path can still be overrun after replication.

Same TAS slot for both paths

If both member streams share the same congested scheduled instant through common bottlenecks, redundancy can collapse into correlated delay.

Recover too early

If duplicates are eliminated before a true disjoint segment, protection may be lost for the remaining path.

5. FRER mechanics in detail

FRER is easiest to understand as five linked functions: identify, sequence, split, transmit member streams, and recover/eliminate. The sequence number is the anchor that connects copies of the same original frame.

5.1 Stream split and sequence encode

The talker or replication bridge takes one original stream frame, assigns a sequence value, and produces multiple member stream frames. Each member copy is forwarded on a configured path. The copies must be recognized as belonging to the same recovery context at the merge point. 

Original frame:
  stream = BrakeStatus
  seq    = 840

Replicated member frames:
  A: stream=BrakeStatus.memberA, seq=840, egress=Path-A
  B: stream=BrakeStatus.memberB, seq=840, egress=Path-B

5.2 Sequence recovery

The recovery function keeps recent sequence history. If a frame has a sequence number that is acceptable and not already seen, it is passed upward. If a second copy with the same sequence arrives later, it is discarded as a duplicate. 

Recovery state:
  accepted: 837, 838, 839
  arrival:  A seq=840  → pass
  arrival:  B seq=840  → duplicate discard
  arrival:  A seq=841  → pass

5.3 Sequence number wrap-around

With the R-TAG, the visible sequence field is 16 bits, so the field wraps after 65535 to 0. A recovery algorithm must not treat wrap-around as a permanent reset or as an attack by itself. It must compare sequence numbers within the configured recovery history/window.

ArrivalNaive interpretationCorrect FRER-style interpretation
65534newAccept and update history.
65535newAccept and update history.
0looks smaller than 65535Accept if it is the next sequence value modulo the sequence space.
65535 againoldDiscard as duplicate/old because it is already in recent history.

5.4 What FRER hides and what it exposes

Hidden from application

Single-path loss where the other copy arrives within the recovery expectation.

Visible in counters

Duplicate discards, lost member frames, sequence gaps, latent error alarms, path asymmetry.

Not solved by FRER

All paths failing, duplicated traffic overrunning queues, misconfigured TAS windows, spoofed stream identifiers.

6. Qci mechanics in detail

Qci is the ingress firewall for time-sensitive streams. Its job is to ensure that the frame entering the forwarding and scheduling path is the right stream, right size, right priority, and right rate.

6.1 Qci processing objects

Stream identificationMatch DA/SA, VLAN, IP tuple, transport port, AVTP/1722 Stream ID, ingress port, etc.
Stream filterBind stream handle + priority to max SDU, gate, meter, and actions.
Max SDUDrop frames larger than the configured stream-specific maximum.
Stream gatePass/close state and optional internal priority metadata.
Flow meterToken-bucket-like committed/excess rate and burst enforcement.

6.2 Qci is per stream, not just per port

A port policer answers, “Is this port too busy?” Qci answers, “Is this specific stream behaving according to its contract?” That difference is critical with FRER because each replicated member stream can take a different path and can fail differently.

In AUTOSAR Classic Ethernet Switch Driver terminology, the same teaching model appears as objects such as EthSwtStreamFilterEntry, EthSwtFilterMaxSduSizeRef, EthSwtStreamGateEntryRef, EthSwtFlowMeteringEntryRef, and EthSwtAssignedStreamHandle. Teaching caution: IEEE Qci stream gates can be time-dependent gates. Some AUTOSAR switch-driver profiles or hardware mappings may expose only a subset of that behavior. Always distinguish the IEEE bridge concept from the ECU software configuration interface.

7.2 What the gate really means

Open does not mean “must send”

It means the queue is eligible for selection. The scheduler still chooses among eligible queues according to configuration.

Closed means “cannot send”

A frame can wait in the queue, but the queue cannot transmit while the corresponding gate is closed.

Guard band matters

A frame should not start if it cannot finish before a protected window, unless preemption/hold-release behavior is configured appropriately.

7.3 TAS does not police the stream

TAS can stop a queue from transmitting at the wrong time, but it does not decide whether a particular stream is valid. If bad frames are queued in the same traffic class, TAS will faithfully schedule the wrong workload. That is why Qci must happen before queueing.

9. Failure observability: where to look when the stream fails

A combined deployment is only useful if you can tell which layer failed. The same application symptom can come from very different root causes.

FRER counters and symptoms

  • Accepted packets per compound stream
  • Discarded duplicates
  • Out-of-window or rogue sequence numbers
  • Sequence gaps and latent error indications
  • Member path imbalance: A copy usually first, B usually late

Qci counters and symptoms

  • Stream identification misses
  • Max SDU drops
  • Stream gate closed drops
  • Flow meter red/yellow decisions
  • Unexpected internal priority or traffic-class assignment

TAS counters and symptoms

  • Frames waiting for gate open
  • Frames not fitting because of guard band
  • Queue occupancy spikes before scheduled windows
  • Missed cycle due to time-sync or base-time configuration
  • Wrong queue selected due to priority mapping error

10. Configuration checklist for combined stream protection

10.1 Stream identity

  • Define the compound stream and each member stream.
  • Bind identification rules to ingress ports wherever possible.
  • Use sufficiently specific primitives: VLAN, DA/SA, IP tuple, UDP port, AVTP/1722 Stream ID as applicable.
  • Make sure the recovery point can map all members to the same FRER recovery context.

10.2 FRER

  • Choose where sequence encoding happens: talker or replication bridge.
  • Choose where replication happens and where recovery/elimination happens.
  • Ensure disjointness. Two copies through the same failure domain are not true protection.
  • Configure sequence history/window/reset logic to tolerate expected path skew but reject stale duplicates.

10.3 Qci

  • Configure max SDU for the actual stream payload plus headers expected at that point.
  • Configure per-member stream meters, not just one aggregate meter, unless the design explicitly intends aggregate policing.
  • Map stream gate/internal priority metadata consistently to traffic classes.
  • Verify that Qci drops are observable and not silently counted only as generic ingress drops.

10.4 TAS

  • Calculate each egress window from frame size, link speed, interference, guard band, and clock uncertainty.
  • Schedule redundant paths to avoid correlated congestion.
  • Ensure queueMaxSDU and guard-band assumptions match actual maximum frames.
  • Validate base-time and cycle-time alignment across bridges.

10.5 A compact design recipe

For each critical stream S:
  1. Identify S at the talker/ingress and assign stream handle.
  2. Add FRER sequence encoding before path split.
  3. Create member streams S.A and S.B with disjoint next hops.
  4. On every ingress bridge port:
       apply Qci max SDU + stream gate + flow meter for each member stream.
  5. Map admitted frames to the intended internal priority / traffic class.
  6. Queue on the egress port.
  7. TAS opens the queue only in the planned cycle interval.
  8. At merge/listener:
       use FRER recovery to accept first copy and discard duplicates.
  9. Monitor:
       FRER duplicate/gap counters, Qci drop reasons, TAS queue/gate timing.

11. Misconceptions to correct during training

“FRER gives deterministic latency.”

No. FRER improves availability by redundant delivery. TAS and traffic-class scheduling are responsible for deterministic egress timing.

“Qci and TAS are both gates, so they are the same.”

No. Qci stream gates are ingress stream admission gates. TAS gates are egress queue transmission gates.

“If the R-TAG sequence is valid, the frame is trustworthy.”

No. Sequence values are not authentication. Stream identification and security controls are still required.

“Duplicate discards mean something is wrong.”

Not necessarily. In normal FRER, the second good copy is expected to be discarded by the recovery function.

“TAS will fix a talker that sends too much.”

No. TAS will delay or block queue transmission according to time, but Qci policing is needed to enforce stream rate and burst contracts.

“Policing before replication is always enough.”

Not always. After replication, each member stream consumes resources on a different path. Member-stream Qci is usually needed to protect every path.
