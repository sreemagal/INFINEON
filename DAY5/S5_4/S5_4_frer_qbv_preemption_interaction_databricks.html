How scheduled traffic, frame preemption, and FRER really interact

This note explains the interaction in switch-pipeline language. It keeps the layers separate, shows the PDU formats that matter, and uses concrete failure and timing scenarios to explain what you should expect to observe.

Time-Aware Shaper gates queues Preemption fragments only preemptable MAC traffic FRER duplicates and eliminates whole frames

Core answer in one view

802.1Qbv decides when a traffic-class queue is eligible for selection by opening or closing a time-controlled gate.

Frame preemption is the 802.1Q / 802.3 MAC-level mechanism that lets an express frame interrupt a preemptable frame on a single physical link.

FRER is a stream-level reliability mechanism. It identifies a stream, attaches or uses a sequence number, replicates frames across member streams, and eliminates duplicate complete frames at recovery points.

Important: preemption does not replicate frames, FRER does not fragment frames, and Qbv does not by itself make traffic reliable.

2. First-principles mental model

Think of a TSN bridge as a sequence of transformations. FRER works on streams and complete frames. Qbv works on queues. Preemption works at the MAC Merge Sublayer on a single outgoing link.

Switch-pipeline position of the three mechanisms

1. Stream identification

Classify a frame as belonging to a compound stream or member stream. Match may use destination MAC, VLAN, priority, IP tuple, or configured stream identification.

2. FRER encode / split

Attach or derive sequence number. Replicate one logical stream into two or more member streams.

3. VLAN / priority / queue

PCP and bridge configuration map each copy to a traffic class and egress queue.

4. Qbv gate

The traffic class gate is open or closed according to the Gate Control List.

5. Express or preemptable

Each traffic class has a frame preemption status. Express goes to eMAC. Preemptable goes to pMAC.

6. MAC Merge

Express mPackets may interrupt preemptable mPackets. Fragments are local to this link.

7. FRER recovery

At a recovery point, complete frames are checked using sequence numbers. First valid copy passes. Later duplicates are discarded.

What preemption can improve

It can reduce the blocking delay caused by a lower-priority or preemptable frame already being transmitted when a scheduled express frame becomes eligible.

What preemption cannot improve

It cannot rescue a frame lost on both FRER member paths. It also cannot fragment a frame across multiple bridges as an end-to-end mechanism.

3. PDU and header formats that matter

There are two distinct packet views. FRER sees the Ethernet frame and its sequence information. Preemption sees MAC Merge mPackets on one link. Do not mix them.

3.1 VLAN-tagged Ethernet frame with FRER R-TAG

IEEE 802.1CB defines a 6-octet Redundancy Tag, or R-TAG. It carries EtherType or TPID 0xF1C1, a 16-bit reserved field, and a 16-bit sequence number. A typical 802.1Q C-VLAN example places the C-TAG before the R-TAG and the original payload length/EtherType after the R-TAG.

4. Rules for 802.1Qbv, preemption, and FRER interaction

Rule 1: Qbv schedules queues, not FRER streams directly

A FRER stream is eventually mapped to one or more egress queues through stream identification, VLAN, priority, and forwarding configuration. Qbv gates then decide whether each queue can be selected at a given time.

Rule 2: preemption is link-local

A preemptable frame can be cut into fragments on one physical link. The next system reassembles it before normal MAC-client processing continues. The next link may preempt the same frame again, but that is a new link-local event.

Rule 3: FRER operates on complete frames

The sequence generation and recovery functions work on packet or frame service primitives. If a preemptable frame is fragmented on a link, the recovery function downstream should receive either a complete reassembled frame or no frame if reassembly fails.

Rule 4: preemption changes latency skew

FRER often delivers the first valid copy and discards later duplicates. Preemption and Qbv can make one member path consistently faster or less jittery than another, which changes which copy usually wins.

4.2 Timing side effect that matters for FRER

FRER recovery windows are not scheduling windows. They are sequence acceptance windows. A Qbv schedule or preemption delay can cause one copy to arrive much later than its sibling copy. That late copy should be discarded as a duplicate if the earlier copy already passed. However, if paths are badly skewed, the recovery function parameters must still tolerate the expected duplicate arrival pattern and possible sequence wrap behavior.

Good design expectation: the fastest path may change during congestion. FRER should not assume that member stream A always wins. It should accept the first valid copy regardless of which member stream carried it, then eliminate duplicates using the sequence algorithm.

5 Sequence wraparound during scheduled bursts

Setup. A high-rate sensor stream uses the 16-bit R-TAG sequence number. It eventually wraps from 65535 to 0. Because Qbv sends traffic in bursts, duplicates from the older sequence range may coexist briefly with newer wrapped values if path skew is extreme.

Meaning. Wraparound is normal. Recovery logic must use the standard sequence recovery algorithm and configured history/window behavior rather than naive numerical comparison such as “0 is always lower than 65535”.

6 Preemption not enabled on one hop

Setup. All bridges support Qbv and FRER, but one physical link has not successfully enabled frame preemption at both ends. Express scheduled traffic still exists, but long lower-priority frames can now cause a larger guard-band or blocking effect on that hop.

Failure signature: FRER may still show successful delivery because at least one copy arrives, but latency histograms show a fat tail or member path B becomes the frequent winner. The problem is not FRER sequence recovery. It is an egress link configuration or capability problem.

Validation. Check LLDP or management state for preemption capability and operational status on both link partners. Also check per-priority express/preemptable assignment. The link must understand MAC Merge formats to reassemble fragments safely.

6. Observability: what to measure and where

When debugging this interaction, do not rely on a single packet capture point. You need counters and captures at the queueing layer, the MAC Merge layer, and the FRER recovery layer.

Check queue gate state.
Verify base time, cycle time, Gate Control List entries, and whether the traffic-class gate is open during the expected window.
Check traffic-class to express/preemptable mapping.
The VLAN PCP may map to a traffic class, and that traffic class has a frame preemption status. A wrong PCP value can silently move a FRER copy from eMAC to pMAC.
Check MAC Merge operational state.
Preemption needs hardware support and compatible link partners. If one side cannot process fragments, preemption must not be treated as operational.
Check FRER sequence generation.
Confirm that all replicated copies of one logical frame carry the same sequence number and are not accidentally re-sequenced independently after splitting.
Check recovery behavior.
Expected counters include passed frames, discarded duplicates, discarded old frames, rogue or reset events, and possibly missing member-stream observations.
Correlate clocks and captures.
Qbv schedules depend on synchronized time. FRER debugging depends on knowing whether a late copy is late because of path delay, gate closure, preemption blocking, or actual loss.

7. Common design pitfalls

Pitfall A: Treating FRER duplicates as packet loss

At the recovery point, duplicate discard is success. The application should see one delivered frame per sequence value, not both copies.

Pitfall B: Forgetting preemption is not end-to-end fragmentation

A packet capture between bridges may show MAC Merge behavior only if the capture point sees the physical link encoding or a device exposes fragment information. Ordinary host captures usually see complete frames.

Pitfall C: Replicating before sequence generation

If each copy is assigned a different sequence number after splitting, the recovery function cannot recognize them as duplicates of one logical frame.

Pitfall D: Making only one member path deterministic

If the backup path is preemptable and unscheduled, failover may preserve connectivity but violate the deadline. Test the degraded case, not just the normal case.

Pitfall E: Wrong R-TAG parsing relative to VLAN tags

802.1CB allows placement flexibility depending on function position, but the bridge configuration must parse and transform tags consistently. The typical C-VLAN example is C-TAG followed by R-TAG.

Pitfall F: Using a naive sequence comparison

The 16-bit sequence number wraps. Recovery must use a wrap-safe sequence acceptance rule and configured history/window behavior.
