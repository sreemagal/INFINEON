How FRER classifies a flow before it can replicate, recover, and eliminate duplicates

Frame Replication and Elimination for Reliability (FRER) does not start with duplication. It starts with a precise answer to a deceptively simple question: which Stream does this frame belong to?

The classification result is a local stream_handle. That handle is then carried internally to sequence generation, R-TAG encode/decode, stream splitting, individual recovery, sequence recovery, and counters. The R-TAG carries the sequence number, not the Stream identity.

Learning outcome

  • Separate flow classification from normal bridge forwarding.
  • Read the FRER-relevant PDU fields in Ethernet, VLAN, IP, transport, and R-TAG headers.
  • Understand why stream_handle is local metadata, not a header field.
  • Predict pass, duplicate discard, rogue discard, tagless handling, and reset behavior.
  • Recognize common configuration traps in automotive Ethernet switches.

1. The mental model

FRER is a small chain of functions. Flow classification is the entry point into that chain.

Stream identification

Examines selected packet attributes and assigns a local stream_handle. In the receive direction, unknown packets pass as non-Stream traffic unless a local policy later drops them.

This is classification. It is not yet duplicate elimination.

Sequence handling

Sequence generation creates the monotonically increasing sequence value for a Compound Stream. Sequence encode/decode inserts or extracts that value. R-TAG is one standard sequence encapsulation.

The R-TAG carries sequence_number, not stream_handle.

Recovery and elimination

Sequence recovery uses the stream handle plus sequence number history to pass the first valid copy and discard duplicates, rogue packets, or tagless packets depending on algorithm and policy.

The recovery decision is per configured Stream or Member Stream context.

2. Where flow classification sits in the switch pipeline

The same physical frame passes through ordinary bridge functions and FRER-specific functions. Keep the internal metadata separate from the transmitted PDU.

Pipeline rule: stream identification creates local metadata. R-TAG encode/decode creates or extracts sequence metadata. Forwarding and queuing can then act on the resulting internal state.

What “passive” means

A passive identification method only examines the received frame and assigns local metadata. The wire frame is not changed by the identification function itself.

match { dmac=91:E0:F0:00:01:01, vid=10 } → stream_handle = 0x101 → continue with same DMAC, same VLAN

What “active” means

An active identification method can encode a Stream identity by changing header parameters on transmit and can decode by recognizing those changed values on receive.

upper stream_handle = 0x401 → transmit as dmac=91:E0:F0:AA:00:01, vid=100, pcp=5 → receiver/proxy restores original addressing

5. From flow classification to sequence recovery classification

Once the frame has a stream_handle and a sequence number, FRER can classify the arrival as new, duplicate, old-but-unseen, rogue, tagless, or reset-triggered.

VectorRecoveryAlgorithm

Designed for streams where a history window is useful. It maintains RecovSeqNum and a bit-vector SequenceHistory of length frerSeqRcvyHistoryLength.

  • First packet after reset is accepted.
  • Packet within the window and unseen can pass.
  • Repeated sequence number already marked seen is discarded as duplicate.
  • Packet too far ahead or too far behind is discarded as rogue.
  • Missing sequence numbers can later be counted as lost when the window shifts.

MatchRecoveryAlgorithm

Simpler algorithm. It only compares the incoming sequence number against the last accepted sequence number.

  • First packet after reset is accepted.
  • If the sequence number equals the last accepted value, discard.
  • If it differs, accept and update RecovSeqNum.
  • It is useful for individual recovery on single member streams but does not provide the same sliding-window duplicate protection as Vector.

7. Configuration patterns and anti-patterns

Use these as switch configuration review checks before integrating FRER in an automotive Ethernet topology.

Pattern A: FRER-aware Talker and Listener

  1. Talker classifies outgoing application packets with Null, IP, or active DMAC/VLAN Stream identification.
  2. Sequence generation assigns a sequence number per Compound Stream.
  3. Sequence encode inserts R-TAG.
  4. Stream splitting creates two or more Member Streams, often with path-specific L2 values.
  5. Listener or merge bridge classifies both members back to the same Compound Stream and performs sequence recovery.
Best when both end systems understand FRER and the network is engineered for redundant disjoint paths.

Pattern B: Proxy FRER in bridges

  1. Legacy Talker emits ordinary IP/SOME-IP/AVTP frames.
  2. Ingress bridge uses IP or mask-and-match classification to assign stream_handle.
  3. Bridge sequence-generates and splits the stream into redundant network paths.
  4. Near Listener, a merge bridge recovers, eliminates duplicates, and restores the ordinary frame.
  5. Legacy Listener sees one normal packet, not duplicate FRER member frames.
Best when ECUs cannot be changed but the Ethernet switching fabric supports FRER.

Anti-pattern: “VLAN equals flow”

Using only VLAN ID for a whole zonal ECU may merge control, diagnostics, camera, and update traffic into one Stream. Recovery then treats unrelated packets as if they were one sequence.

Bad: VID 40 → stream_handle 0x40 Better: {VID 40, UDP dst 30501} → stream_handle 0x401 {VID 40, UDP dst 30502} → stream_handle 0x402

Anti-pattern: “R-TAG tells me the path”

R-TAG carries the sequence number. It does not identify the redundant leg. If two legs use the same VLAN and same R-TAG format, the bridge may need input port, active DMAC/VLAN, or configured member stream context to know which instance it is processing.

R-TAG: F1 C1 00 00 00 47 Meaning: sequence = 71 Not meaning: path A, path B, stream 7, or port 3

8. Validation checklist

Use this to validate captures, switch tables, and FRER behavior.

Capture checks

  • Do you see R-TAG EtherType 0xF1C1 where the switch expects it?
  • Do replicated copies carry the same sequence number?
  • Does the sequence number wrap from 65535 to 0 without being treated as fatal?
  • Does the listener receive one delivered packet for two network copies?

Classification checks

  • Is the identification method specific enough for the application flow?
  • Are entries ordered so the specific rule matches before a broad wildcard?
  • Does the rule include ingress port when needed to separate member paths?
  • Is active DMAC/VLAN restoration configured at the far end if the destination header was changed?

Recovery checks

  • Is Vector history length large enough for expected path skew and reordering?
  • Is reset timeout appropriate for talker reboot or silence?
  • Are tagless frames intentionally passed or intentionally dropped?
  • Are rogue, duplicate, out-of-order, lost, tagless, and passed counters moving as expected?
Rule of thumb: classify by the most stable unique tuple available. Use L2 only when L2 is truly unique. Use IP/L4 or mask-and-match when multiple logical flows share L2 identity.
Debug shortcut: when recovery looks wrong, first prove that both copies map to the same intended compound stream_handle and that no unrelated traffic maps there.
