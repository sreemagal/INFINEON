FRER failure observability from the packet, header, counter, and recovery-state viewpoint

Frame Replication and Elimination for Reliability makes a network resilient by sending replicated copies over redundant paths and eliminating duplicates at a merge point. The important diagnostic challenge is that the best case for FRER is also the most dangerous case for observability: the application may see no packet loss while one redundancy path has silently failed.

1. The central idea

FRER is not a failure-alarm protocol. It is a frame-processing mechanism. It identifies a stream, generates or decodes a sequence number, sends copies over multiple member streams, and removes duplicates at one or more recovery points. Observability comes from comparing what the recovery function accepted, discarded, lost, marked tagless, reset, or found malformed.

Useful teaching rule: a healthy two-path FRER system should normally show one accepted packet and roughly one duplicate-discarded packet for each original packet. If accepted packets remain normal but duplicate discards disappear, the service is still working but redundancy is already degraded.

What FRER hides

Single-path loss after replication. The listener still receives one copy, so application-level packet loss can remain zero.

What FRER exposes

Sequence gaps, duplicate arrivals, old arrivals, rogue sequence values, tagless packets, decode errors, resets, and latent-error events.

What FRER cannot know alone

Whether the root cause is cable damage, wrong VLAN, PSFP drop, shaping delay, congestion, security filtering, stream misclassification, or switch table misconfiguration.

2. FRER function stack and where failures become visible

FRER observability follows the order of the functional blocks. A failure before stream identification looks different from a failure after stream splitting. A packet dropped before replication produces a sequence gap on every path. A packet dropped on one replicated path does not produce application loss, but it changes the duplicate-discard pattern.

3. PDU and header format: what is actually on the wire?

FRER does not define a separate failure-report PDU that travels between switches. The data packet itself carries sequence information. Management and status are exposed through managed objects such as YANG, SNMP, vendor APIs, or switch-driver counters.

Important distinction: the stream_handle is internal metadata used by the bridge or end system. It is not a transmitted field in the Ethernet frame. The sequence number can be transmitted, commonly using an R-TAG.

3.3 Concrete byte example

Assume VLAN ID 10, Priority Code Point 5, IPv4 payload, and sequence number 333 decimal, which is 0x014D.

DA SA VLAN R-TAG IPv4 01 00 5E 00 00 01 02 00 00 00 00 01 81 00 A0 0A F1 C1 00 00 01 4D 08 00 45 00 ... PCP=5 VID=10 TPID RSV SEQ=333 payload EtherType
Capture trap: if the recovery point strips the R-TAG before forwarding to the listener, a listener-side capture will not show the sequence number. Capture before stripping or inspect the recovery counters.
Offset trap: if VLAN tags are inserted, removed, or reordered, the R-TAG can appear at a different byte position. Monitoring tools should search the tag stack rather than assume “R-TAG always starts at byte 12.”

3.4 Other sequence encodings

IEEE 802.1CB is architected around sequence encode/decode functions. R-TAG is the common TSN teaching example, but sequence information can also be associated with other redundancy technologies such as High-availability Seamless Redundancy sequence tags or Parallel Redundancy Protocol trailers. The observability principle is the same: find the stream, find or derive the sequence number, then watch recovery decisions.

4. Sequence recovery logic: how a packet becomes passed, duplicate, lost, old, rogue, or tagless

The sequence recovery function holds state for a stream. The exact implementation can vary by algorithm and hardware, but the diagnostic mental model is stable.

RecovSeqNum

Reference sequence number used by the recovery algorithm. It tells the function where the stream appears to be now.

SequenceHistory

A bitmap-style memory used by vector recovery to remember recent sequence numbers and identify duplicates or old arrivals.

Reset timeout

Allows the receiver to recover after a talker reset, sequence generator restart, or long idle interval.

TakeNoSequence

Policy knob for frames that belong to the stream but arrive without a sequence number.

  1. No stream match: the frame is outside this FRER function. FRER counters for this stream will not move.
  2. Stream match but no sequence number: TaglessPackets increases. The configured no-sequence policy determines whether it is accepted or discarded.
  3. New expected sequence: the frame is passed and recovery state moves forward.
  4. Second copy of a sequence already accepted: the frame is discarded as a duplicate. This is normal in a healthy replicated network.
  5. Sequence jumps forward: missing sequence numbers are inferred as lost. The receiver may still pass the new packet.
  6. Very old or impossible sequence: the packet is classified as rogue or outside the acceptable window.
  7. No valid accepted packets for the reset interval: the recovery function resets to avoid permanent lockout.

4.2 Wrap-around example

A 16-bit sequence number wraps from 65535 to 0. A correct FRER-aware monitor treats this as normal modulo arithmetic, not as a talker reset.

Healthy wrap sequence: ... 65533, 65534, 65535, 0, 1, 2 ... Suspicious reset-like jump: ... 45102, 45103, 45104, 0, 1, 2 ... Diagnostic difference: The first transition is expected at sequence-space wrap. The second transition may indicate sequence generator restart, wrong recovery state, capture from a new talker, or a recovery reset event.

5. Counters and what each counter really tells you

For failure observability, counters are more useful when interpreted as ratios and time derivatives, not as absolute totals. A counter value of zero can mean “no problem,” “feature not configured,” “wrong stream matched,” or “counter was cleared.” Always correlate with port counters, stream counters, packet capture, and the configured FRER tables.

Rule of thumb for two paths: if the stream sends 10,000 frames in a diagnostic window, a healthy merge point often passes about 10,000 and discards about 10,000 duplicates. If it passes 10,000 and discards 0, the listener looks happy but FRER redundancy is gone.

6. Latent error detection: the observability mechanism for “hidden” FRER failures

A latent error is a condition where the replicated service may still deliver packets, but the redundancy assumptions are no longer being met. The classic example is one path down in a two-path FRER system. The listener sees the stream, but the recovery function no longer sees duplicate copies.

Healthy two-path arithmetic

During a 1 s window: PassedPackets delta = 1000 DiscardedPackets delta = 1000 Expected duplicates = Passed × (paths - 1) = 1000 × (2 - 1) = 1000 Result = healthy redundancy pattern

Path B failed but service survives

During a 1 s window: PassedPackets delta = 1000 DiscardedPackets delta = 0 Expected duplicates = 1000 Difference = 1000 missing duplicate copies Result = latent-error candidate

6.1 Key latent-error parameters

ParameterPurposeHow bad tuning misleads you
pathsNumber of redundant paths expected by this recovery instance.If configured too low, a missing path will not be detected. If too high, false latent errors appear.
differenceAllowed mismatch between expected duplicate behavior and observed duplicate behavior.Too tight creates false alarms during transient reordering. Too loose hides path degradation.
periodTime between latent-error tests.Too short can react to normal burstiness. Too long delays detection.
reset periodTime between latent-error reset actions.Too frequent can erase evidence before diagnostics collect it.
Do not use application success as your only FRER health metric. FRER is designed to make application success continue during a single path failure. Latent-error detection and duplicate-discard trends are what tell you the redundancy budget has been consumed.

7. Where to instrument: six observation points

A good FRER failure investigation places instruments before and after the points where the packet is transformed. A single capture at the listener is often insufficient.

P0 Source
Original unreplicated stream. No R-TAG if source is FRER-unaware.
P1 Sequence generation
First point where sequence numbers should appear.
P2 Split egress A/B
Both copies should share sequence number and leave on intended ports.
P3 Mid-path
VLAN, shaping, policing, MACsec, or FDB issues can drop only one copy.
P4 Recovery merge
Main point for passed, duplicate, lost, rogue, tagless, reset counters.
P5 Listener

10. Packet capture and observability guide

Use packet capture to validate the header and sequence behavior, but do not rely on packet capture alone. FRER-aware counters tell you what the recovery algorithm did with the frames after classification.

Wireshark/tcpdump checks

  • Filter for the redundancy tag EtherType: eth.type == 0xf1c1 when the R-TAG is visible as an EtherType.
  • For command-line hex inspection: tcpdump -i ethX -e -xx ether proto 0xf1c1.
  • If VLAN tags precede the R-TAG, a simple BPF expression may miss it. Capture full frames and inspect the tag stack.
  • Check that both member paths carry the same sequence number for the same original packet.
  • Check whether R-TAG stripping is enabled before the capture point.

Switch counter checks

  • Read recovery counters at the merge point, not only at the listener.
  • Read member-stream counters per input or output path if the platform exposes them.
  • Read normal port counters: link down, CRC errors, dropped frames, oversized/undersized frames, ingress-filter drops.
  • Read VLAN/FDB/stream-classification hits to prove the stream is entering the expected FRER instance.
  • Timestamp counter samples. Counter deltas per second are more useful than absolute values.

11. Common traps in FRER failure observability

Trap 1: “No application loss means FRER is healthy.”
Wrong. It may mean FRER consumed the redundancy budget and is now one failure away from service outage.
Trap 2: “No R-TAG in listener capture means no FRER.”
Wrong. The recovery node may strip the tag before forwarding.
Trap 3: “DiscardedPackets are bad.”
Wrong in a replicated system. Duplicate discards are expected. A sudden reduction can be a worse sign than an increase.
Trap 4: “FRER creates redundant paths.”
Wrong. FRER uses paths provided by bridge configuration, routing, VLAN planning, or a controller. Path disjointness is a network-design issue.
Trap 5: “TaglessPackets can be safely accepted.”
Dangerous. TakeNoSequence=true may be useful during migration, but it weakens duplicate elimination and replay detection.
Trap 6: “Sequence number jump always means failure.”
Not always. Natural 16-bit wrap is normal. A jump to zero outside wrap is suspicious.

12. FRER failure-debug checklist

1
Confirm stream identity. Verify ingress port, destination/source MAC, VLAN ID, Priority Code Point, IP tuple, or mask-and-match rule used by the stream identification function.
2
Confirm sequence generation. Capture after the generation point. Verify the R-TAG or other sequence encoding appears and increments modulo the sequence space.
3
Confirm split. Both redundant egress paths should carry copies with the same sequence number and expected VLAN/priority treatment.
4
Confirm path delivery. Sample port counters and captures on each path. Look for one-sided drops caused by VLAN membership, FDB, PSFP, policing, shaping, link errors, or security filtering.
5
Confirm recovery counters. For two paths, PassedPackets should track the original stream rate and duplicate discards should be roughly one per passed packet when both paths are healthy.
6
Check latent-error configuration. Verify paths, difference, period, and reset period. A wrong path count makes latent-error detection either blind or noisy.
7
Check reset behavior. If the talker restarts or sequence jumps, confirm the reset timeout is not so long that it creates a blackhole and not so short that it masks real errors.
8
Preserve evidence. Do not clear counters before recording them. Collect counter deltas, packet samples, port status, configuration dumps, and time stamps together.
