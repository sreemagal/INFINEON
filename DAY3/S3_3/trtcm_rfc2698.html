RFC 2698 · Heinanen & Guerin · September 1999

trTCMTwo rate three color marker.

Policing · PIR · CIR · PBS · CBS

Where srTCM meters one rate with two burst budgets, trTCM meters two rates with their own burst budgets. The peak rate sets the absolute ceiling; the committed rate sets the guaranteed floor. Everything between is sold as "excess" — delivered if capacity exists, dropped first if it doesn't. This is the marker for contracts that promise "up to X Mbps with Y Mbps guaranteed".

01The parameters

PIR
Peak Information Rate — bytes per second. The maximum rate the flow may ever sustain. Defines the absolute ceiling above which packets are always red.
PBS
Peak Burst Size — bytes. Capacity of the peak bucket Tp. Tolerates short bursts above PIR up to this many bytes before turning them red.
CIR
Committed Information Rate — bytes per second. The guaranteed rate — traffic at or below this is marked green regardless of network load.
CBS
Committed Burst Size — bytes. Capacity of the committed bucket Tc. Tolerates short bursts above CIR up to this many bytes before downgrading to yellow.
Tp, Tc
The two bucket fill levels, in tokens. Initialised full: Tp(0) = PBS, Tc(0) = CBS. The RFC requires PIR ≥ CIR — otherwise the scheme is degenerate.

02Two independent buckets

The critical structural difference from srTCM: the buckets are parallel, not cascaded. Tokens arrive on two separate pipes — PIR into Tp, CIR into Tc. There is no overflow path between them. Each bucket fills independently to its own capacity and spilled tokens are simply discarded.

On a packet of size B bytes, the test order is peak first, then committed. This is counterintuitive at first — you might expect "test the guaranteed rate first" — but the logic is that peak is the gate, and committed is only consulted for packets the gate admits.

01
If Tp < B → mark red. No token is consumed from either bucket. The packet exceeded the peak ceiling and is dropped.
02
Else if Tc < B → mark yellow, decrement Tp by B. The packet is within peak but exceeded the committed rate — forwarded as excess.
03
Else → mark green, decrement both Tp and Tc by B. The packet is inside the committed envelope and consumes from both pools.

Note the dual decrement on green — a green packet "costs" tokens from both buckets because it consumed peak and committed capacity simultaneously. This is the second subtlety people often miss. Yellow only debits Tp because Tc was already too low to charge.

04Three regimes worth watching

Low load
Offered rate < CIR. Both buckets stay near full; every packet is green. Neither constraint binds.
Excess
Offered rate between CIR and PIR. Tc empties first (it fills slower); once drained, new packets become yellow — they still debit Tp, which has enough capacity. Steady-state yellow fraction ≈ (load − CIR) / load.
Overload
Offered rate > PIR. Tp drains too; once empty, packets are red. Long-run: green fraction = CIR/load, yellow = (PIR − CIR)/load, red = (load − PIR)/load.

05srTCM vs trTCM at a glance

srTCM · RFC 2697

  • One rate: CIR
  • Two buckets: Tc, Te — cascaded
  • Overflow from Tc feeds Te
  • Yellow means "used idle-accumulated budget"
  • Contract: "rate X with burst tolerance"
  • Test order: committed → excess

trTCM · RFC 2698

  • Two rates: PIR ≥ CIR
  • Two buckets: Tp, Tc — independent
  • No overflow between buckets
  • Yellow means "above committed but below peak"
  • Contract: "up to PIR with CIR guaranteed"
  • Test order: peak → committed

A concrete way to feel the difference: in srTCM, yellow is a reward the flow earns by being quiet — unused committed tokens spill over and pay for future excess. In trTCM, yellow is a permission the contract grants — the flow is always allowed to burst up to PIR, regardless of what it did earlier. Same three colours, very different economics.

06Where trTCM shows up

The canonical use is carrier Ethernet tiered services — a 100 Mbps access link sold as "50 Mbps committed, 100 Mbps peak, 10 ms burst". CIR = 50 Mbps gets the SLA, PIR = 100 Mbps is the physical link rate, yellow traffic uses whatever headroom the carrier has. MEF 10.3 Bandwidth Profiles are defined in trTCM terms. In DiffServ domains, trTCM at the edge maps green/yellow/red to AF class drop precedence (e.g. AF11/AF12/AF13), so downstream congestion points drop yellow before green and red before either.
