Egress scheduling · transmission selection algorithms

QueuingFIFO · Strict priority · WRR · WFQ.

Latency, jitter, fairness · pick two

Every output port on a bridge runs a transmission selection algorithm that decides which queued packet leaves next. The choice determines the port's latency, jitter, and fairness behaviour — and the failure modes it's prone to. Four disciplines are worth knowing well: the degenerate baseline (FIFO), the strict-priority scheduler (SP), and two fair-share variants (WRR, WFQ). They sit on a spectrum from simplest-but-worst-guarantees to most-complex-but-best-fairness.

01FIFO · the degenerate case

A FIFO port has one queue. Packets leave in the order they arrived. There is no prioritisation, no fairness, no shaping. A single congested talker fills the queue and delays every packet behind it — including latency-critical frames that would have been trivially ahead in any priority scheme.

FIFO is acceptable only when exactly one traffic class exists on the port. The moment a port carries mixed latency classes — which is the default in any TSN-aware design — FIFO becomes a liability. Every scheduler below exists because FIFO is not enough.

03How to read the output tape

The tape along the bottom shows each served packet laid out left-to-right in the order the scheduler dequeued them, with width proportional to byte size. This is the single most useful view for building intuition. Compare the same queued packets under different schedulers — the pattern on the tape tells you what the scheduler is actually doing.

SP
The tape starts with a run of red packets. Only once all red packets have been served does any amber appear. Slate only shows after red and amber are both exhausted. This is strict priority — lower classes starve behind higher classes.
WRR
The tape cycles red × 4 → amber × 2 → slate × 1, then repeats. Count packets, not bytes: each round always serves the same number of packets per queue regardless of packet size. Notice how few slate bytes are served compared to red, even though slate packets are ~4× larger.
WFQ
The tape mixes the three colours in a more intricate pattern — the scheduler interleaves packets so that the total bytes served from each queue track the 4:2:1 weight ratio. Small high-priority packets come in clusters between the larger low-priority ones.
FIFO
In FIFO mode, the three queues are merged into a single arrival-ordered queue. The tape shows packets in enqueue order regardless of class — no smoothing, no prioritisation.

04The byte-vs-packet fairness question

WRR and WFQ exist for the same job — proportional sharing according to weights — but they measure different things. Watch the Bytes served panel after running each scheduler for several seconds with the defaults:

Under WRR, the packet counts ratio will match 4:2:1 but the byte counts will be skewed: low gets disproportionately many bytes because each of its packets carries 1500 bytes, while high gets fewer bytes because its packets are only 400 bytes. If your contract promises bandwidth in bytes per second — which is what every SLA actually says — WRR will under-deliver to queues with small packets and over-deliver to queues with large ones.

Under WFQ, the byte ratio settles at 4:2:1 regardless of packet size. This is what WFQ exists to fix. The cost is per-packet timestamp computation: each enqueued packet gets a virtual finish time of finish = max(virtualClock, prevFinish) + size / weight, and the scheduler always serves the packet with the smallest finish time next. It's a priority queue keyed on a computed quantity rather than a round-robin pointer.

In practice, most hardware implements Deficit Round Robin (DRR) rather than pure WFQ. DRR approximates byte-fairness using a per-queue deficit counter instead of virtual finish times — much cheaper to implement, and the fairness error is bounded by one MTU per queue per round. For automotive and industrial TSN switches, DRR is the default "fair" scheduler you'll actually find.

05Schedulers at a glance

FIFO

  • One queue
  • Arrival order = departure order
  • No prioritisation
  • HOL blocking
  • Single-class links only

SP

  • N queues, one priority each
  • Always drains highest non-empty
  • Minimum latency for top class
  • Starves lower classes
  • Top 1–2 queues only (PTP, net-ctrl)

WRR

  • N queues, per-queue weight
  • Round-robin by packet count
  • Starvation-free
  • Byte-unfair with mixed sizes
  • Cheap to implement

WFQ

  • N queues, per-queue weight
  • Selects by virtual finish time
  • Exact byte-fairness
  • Priority-queue in hardware
  • DRR used as cheap proxy

06Composing schedulers

Real TSN egress ports never run one algorithm across all queues. A typical 8-queue port runs a hierarchy: SP for the top two queues (PTP event messages, gPTP/network control), CBS-shaped transmission for the two below (AVB class A and B, or ADAS streams), WRR or DRR across the remaining best-effort queues. The selector at each tick asks: "is there a non-empty, credit-eligible SP queue? If yes, serve from it. Otherwise, is there a credit-eligible CBS queue? If yes, serve from it. Otherwise, run WRR across the best-effort queues." This composition is how one port simultaneously delivers bounded latency to control traffic, reserved bandwidth to ADAS, and fair sharing to bulk traffic — all with a few dozen registers of hardware state.
